The context for skipping Ratatui's internal widget render involves avoiding the massive CPU overhead caused by Ratatui's default immediate-mode paradigm. Normally, calling `Terminal::draw` and utilizing `Widget::render` forces the application to globally rebuild the entire widget tree and evaluate the full terminal grid on every single frame to compute the output diff. For a terminal window manager processing high-throughput background PTY streams, this global redraw cycle results in severe CPU bottlenecking, frame drops, and degraded rendering latency.

To integrate an optimized "damage tracking" system, the architecture must bypass these default declarative behaviors. Here is how the direct memory access pipeline works:

*   **Direct Buffer Access:** Instead of using `Terminal::draw`, the architecture calls `Terminal::current_buffer_mut()`, which returns a mutable reference directly to the underlying `Buffer` that the terminal intends to draw. 
*   **Memory Blitting:** Once the engine calculates the absolute minimum visible damaged areas using occlusion culling (via 2D rectangle subtraction), it performs a direct memory copy (blitting) cell-by-cell from the background PTY's virtual grid into Ratatui's buffer. This completely bypasses the `Widget::render` cycle.
*   **Hardware Flush:** After the target buffer is manually populated with only the visible spans, the custom event loop invokes `terminal.apply_buffer()` or `terminal.apply_buffer_with_cursor()`. This low-level function takes the manually mutated buffer, computes the final differential against the previous physical hardware state, and flushes the optimal ANSI payload directly to standard output.

By bypassing the declarative widget layout phase entirely for static and PTY content, CPU utilization is drastically minimized. However, the research notes that when performing this direct memory blitting, you must be extremely careful with **Unicode correctness and cell widths**. Because you are manipulating memory directly, you must ensure complex Unicode glyphs (like halfwidth Katakana characters) are assigned the correct grid spacing utilizing Ratatui's `CellDiffOption::ForcedWidth`, otherwise the subsequent hardware diffing engine will suffer alignment panics.
