# Fix: Tiling Layout Corruption from Drag-and-Drop Snap Operations

## Problem

Dragging windows around the tiling grid produces corrupted states:
- Empty spaces with drag handles next to phantom windows
- Window snap previews thrown to the opposite side of the screen

## Root Cause

**State machine flaw in the `Moved` event handler** (`mod.rs:1054-1058`).

When the cursor leaves the terminal during a drag with a pending snap target:
1. `Moved` fires → calls `apply_snap(key)` (removes window from tiling tree, re-inserts at snap position)
2. Returns `(true, false)` → **drops `mouse_capture`**
3. `snap_preview` is **never cleared** by `apply_snap` → stale ghost preview persists
4. `snap_projection_cache` is **never invalidated** after the tree mutation → stale preview rects
5. If a subsequent `Release` event fires, there's no capture to process → the window may be left in an inconsistent dual state (floating rect set + position in tiling tree)

Secondary issue: `apply_snap` doesn't clear `snap_projection_cache` after mutating the layout tree.

## Plan

### Change 1: Add `snap_applied` field to `DraggingWindow`

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs`

Add `snap_applied: bool` after `detach_coordinate` in the `DraggingWindow` variant (line 44). This tracks whether `apply_snap` has already committed the window to the tiling tree during a `Moved` event.

### Change 2: Initialize `snap_applied: false` in all construction sites

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs`

Add `snap_applied: false` to:
- Production constructor at line 1332 (`HeaderAction::Drag` handler)
- 5 test constructors at lines 2984, 3466, 3497, 3520, 3601

### Change 3: Fix the `Moved` handler

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (lines 1054-1058)

Replace the current handler with:
```rust
MouseEventKind::Moved if self.drag_snap.is_some() => {
    self.cancel_drag_snap_timer();
    self.drag_last_event = None;
    self.apply_snap(*key);
    *snap_applied = true;
    self.snap_preview = None;
    self.snap_projection_cache = None;
    (true, true)  // keep capture alive instead of dropping it
}
```

Key changes:
- Set `*snap_applied = true` to mark the snap as committed
- Clear `snap_preview` (prevents stale ghost preview)
- Clear `snap_projection_cache` (invalidates stale projection data)
- Return `(true, true)` instead of `(true, false)` — keep the capture alive so `Release` can clean up properly

### Change 4: Guard `detach_from_tiling_layout` in the `Release` handler

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (lines 1045-1048)

Change the final `else` branch from:
```rust
} else {
    self.detach_from_tiling_layout(*key);
}
```
To:
```rust
} else if !*snap_applied {
    self.detach_from_tiling_layout(*key);
}
```

When `snap_applied` is true (set by `Moved` handler), the window is already correctly positioned in the tiling tree. Skipping `detach_from_tiling_layout` prevents the double-detach that creates empty spaces.

### Change 5: Clear `snap_projection_cache` in `apply_snap`

**File:** `crates/term-wm-core/src/window/window_manager/drag.rs`

Add `self.snap_projection_cache = None;` after each tree mutation path in `apply_snap`:
- After the Void insert path (before `return` at line 294)
- After the Edge/Corner insert path (before the `pending_snap` loop at line 374)

### Change 6: Add regression test

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (after line 3022)

Add `moved_snap_then_release_does_not_corrupt_layout` test that verifies:
1. Two-window tiling layout exists
2. Drag starts on one window
3. `Moved` event commits snap (drag_snap consumed)
4. Capture survives (not dropped by Moved)
5. `Release` event fires without corrupting the layout
6. Both windows remain in tiling tree
7. Regions are non-overlapping (no phantom windows)
