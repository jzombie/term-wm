# Plan: Advanced Window Physics for term-wm

## Overview
Implement 5 phases of window physics on top of the existing layout engine, which already has: edge snap detection, `EdgeResistance` (X-axis only), quadrant-based tiling insertion, maximize/unmaximize with `prev_floating_rect`, drag-to-detach from tiling, and floating resize.

Layout engine (`term-wm-layout-engine`) is `no_std` and must remain so — pure math only. All wall-clock time tracking lives in the core crate.

## Phase 1: Edge Snapping and State Restoration

### 1a. Sacred Top Edge (Preview-then-Commit) — Corner-First Priority
**Files:** `crates/term-wm-core/src/window/window_manager/mod.rs`, `drag.rs`; `crates/term-wm-console/src/draw_plan_renderer.rs`; `src/lib.rs`

**Current:** Top-edge panels clamp y to `bounds_y`; no maximize-on-top-edge behavior.  
**Change (Preview-then-Commit, corner-first spatial priority):**
1. **Spatial priority order** (smallest region first): `corner_snap → top_edge_maximize → edge_snap → tiled_insert`. The hit-test matrix in `update_snap_preview` must evaluate in this order so corners (which occupy `row == 0` + an X edge) are not masked by the generic `row == 0` check.
2. In `dispatch_mouse` Phase-1 `DraggingWindow` handler (`mod.rs:871-892`):
   - First: call `detect_corner_snap(...)` — if it matches, set `SnapPreviewState::Corner(pos)`.
   - Second (only if corner is `None`): check `row == 0` (absolute terminal top) — if matched, set `SnapPreviewState::Maximize`.
   - Third (only if both are `None`): proceed with existing `detect_edge_snap` and `detect_quadrant` for edge/tiled snap.
   - **Do NOT call `toggle_maximize` immediately.** Ghost previews render at the appropriate rect during drag.
   - On `Release`: commit the snap based on the current `SnapPreviewState` (Maximize → `toggle_maximize`, Corner/Edge/TiledInsert → existing `apply_snap`).
   - On drag leaving the trigger zone by >3 cells: clear `SnapPreviewState`.
3. Add `SnapPreviewState` enum to mod.rs:
   ```rust
   pub(crate) enum SnapPreviewState {
       Edge(InsertPosition),
       Corner(InsertPosition),  // TopLeft/TopRight/BottomLeft/BottomRight
       TiledInsert(WindowKey, InsertPosition),
       Maximize,                // full-screen outline
   }
   ```

### 1b. Corner Quarter-Screen Snap
**Files:** `crates/term-wm-layout-engine/src/snap.rs`

**Current:** `InsertPosition` has only `Left/Right/Top/Bottom`; `edge_preview_rect` does 50% splits only. No corner detection.  
**Changes:**
- Add `InsertPosition::TopLeft | TopRight | BottomLeft | BottomRight` variants.
- Implement `to_orientation()`: TopLeft/BottomLeft → Horizontal, TopRight/BottomRight → Horizontal (all corners create vertical BSP splits). Actually each corner maps: TopLeft = Left+Top combined. For preview rect it's a quarter of the managed area.
- Add `detect_corner_snap(col, row, managed_area, sensitivity) -> Option<InsertPosition>`:
  - Use independent orthogonal axis checks with saturating arithmetic exclusively (no raw `+` or `-`):
    - **TopLeft**:      `col <= sensitivity.saturating_add(margin_left) && row <= sensitivity.saturating_add(margin_top)`
    - **TopRight**:     `col >= margin_right.saturating_sub(sensitivity) && row <= sensitivity.saturating_add(margin_top)`
    - **BottomLeft**:   `col <= sensitivity.saturating_add(margin_left) && row >= margin_bottom.saturating_sub(sensitivity)`
    - **BottomRight**:  `col >= margin_right.saturating_sub(sensitivity) && row >= margin_bottom.saturating_sub(sensitivity)`
    - where margins are derived with saturating arithmetic: `margin_left = x`, `margin_right = x.saturating_add(width).saturating_sub(1)`, `margin_top = y`, `margin_bottom = y.saturating_add(height).saturating_sub(1)`.
  - Priority order (tiebreak): TopLeft > TopRight > BottomLeft > BottomRight when the cursor is in a corner overlap zone.
- Add `corner_preview_rect(managed_area, pos) -> LayoutRect`:
  - TopLeft: x=area.x, y=area.y, w=area.w/2, h=area.h/2
  - TopRight: x=area.x+area.w/2, y=area.y, w=area.w/2, h=area.h/2
  - BottomLeft: x=area.x, y=area.y+area.h/2, w=area.w/2, h=area.h/2
  - BottomRight: x=area.x+area.w/2, y=area.y+area.h/2, w=area.w/2, h=area.h/2
- Update `SnapTarget` with a `Maximize` variant (for top-edge full-screen).
- Update `edge_preview_rect` and `tiled_preview_rect` to handle all 8 InsertPositions.

### 2c. Velocity Threshold (Fast-Move-to-Break)
**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (+ drag.rs)

**Current:** `DraggingWindow` has no velocity.  
**Changes:**
1. Add `prev_col: u16, prev_row: u16, prev_time_ns: u64` fields to `MouseCaptureState::DraggingWindow`.
2. In `dispatch_mouse` Phase-1: on each Drag event, compute dx, dy, dt from current vs previous. Use cross-multiplication (pure integer, no truncation):
   ```rust
   let dist_sq = u32::from(dx.abs_diff(0)).pow(2) + u32::from(dy.abs_diff(0)).pow(2);
   let dt_ms_sq = u64::from(dt_ms).pow(2);
   // threshold: e.g., 10 cells per 100ms
   // Cross-multiplication: dist_sq / dt_ms_sq > threshold_cells_sq / threshold_time_sq
   // → dist_sq * threshold_time_sq > threshold_cells_sq * dt_ms_sq
   let threshold_cells_sq = 10u64.pow(2);   // cells²
   let threshold_time_sq = 100u64.pow(2);   // ms²
   if u64::from(dist_sq).saturating_mul(threshold_time_sq)
       > threshold_cells_sq.saturating_mul(dt_ms_sq)
   {
       // velocity exceeded: force snap off for this frame
   }
   ```
3. Configurable via `WmConfig` with `velocity_threshold_cells` and `velocity_threshold_ms` fields. Pass `velocity_exceeded: bool` into `EdgeResistance::apply_x/y` → when true, skip magnetic snapping for that frame.

## Phase 3: Angular Quadrant Detection

**Files:** `crates/term-wm-layout-engine/src/hit_test.rs`, `crates/term-wm-core/src/window/window_manager/drag.rs`

**Current:** `detect_quadrant` uses axis-aligned bisection (`|dx| > |dy|`). This is incorrect per spec.  
**Changes:**

Replace `detect_quadrant` with a dimension-scaled cross-product to handle arbitrary target aspect ratios. The raw `|dx| > |dy|` comparison only works for perfect squares — for a 100×20 pane it creates massive horizontal drop zones and microscopic vertical ones.

```rust
/// Divide target rect into four diagonal quadrants by calculating the
/// angle of the cursor relative to the geometric center, scaled by the
/// target's aspect ratio.
///
/// Uses dimension-scaled cross-product (no floating-point, no sqrt):
///   scaled_dx = |dx| * target.height
///   scaled_dy = |dy| * target.width
///
/// This guarantees the diagonal dividing lines always pass through the
/// center at the correct slope for the rectangle's aspect ratio,
/// producing equal-area triangular quadrants regardless of shape.
pub fn detect_quadrant(cursor_col: u16, cursor_row: u16, target: &LayoutRect) -> Quadrant {
    let (cx, cy) = target.center();
    let dx = i32::from(cursor_col).saturating_sub(cx);
    let dy = i32::from(cursor_row).saturating_sub(cy);

    if dx == 0 && dy == 0 {
        return Quadrant::East;
    }

    let adx = u32::from(dx.unsigned_abs());
    let ady = u32::from(dy.unsigned_abs());
    let w = u32::from(target.width);
    let h = u32::from(target.height);

    // Scale by the opposite dimension: the true diagonal of a rectangle
    // has slope ±(height/width), so |dx|*h > |dy|*w means the cursor is
    // the East/West diagonal quadrant.
    let scaled_dx = adx.saturating_mul(h);
    let scaled_dy = ady.saturating_mul(w);

    if scaled_dx > scaled_dy || (scaled_dx == scaled_dy && dx >= 0) {
        // East/West diagonal quadrant (cursor is within the diagonal band)
        if dx >= 0 { Quadrant::East } else { Quadrant::West }
    } else {
        // North/South diagonal quadrant
        if dy < 0 { Quadrant::North } else { Quadrant::South }
    }
}
```

**Why this works:** Multiplying `|dx|` by `target.height` and `|dy|` by `target.width` transforms the coordinate system into a unit-square space where the diagonals are at exactly 45°. In the original space, the comparison `|dx| * h > |dy| * w` is equivalent to checking `|dx| / w > |dy| / h` — i.e., whether the slope from center to cursor is shallower than the true geometric diagonal `(±h/±w)`. This produces drop zones that are exact diagonal quarter-circles (in linear space) matching the actual aspect ratio.

**Update existing tests** to verify behavior on non-square targets (e.g., 100×20 pane).

## Phase 3b: BSP Dry-Run Projection for Accurate Previews

**Files:** `crates/term-wm-core/src/layout/tiling.rs`, `crates/term-wm-core/src/window/window_manager/mod.rs`, `crates/term-wm-core/src/window/window_manager/drag.rs`

**Root Cause:** All snap preview rects (`edge_preview_rect`, `corner_preview_rect`, `tiled_preview_rect`) are pure geometric approximations (simple width/height halving). But `apply_snap` inserts the window into the `LayoutNode` BSP tree, which produces a **different** final geometry because:
- BSP rects come from recursive splitting, not simple division
- Gaps are subtracted before splitting via `split_rects_with_gaps`
- The tree structure (weights, ratios) determines exact boundaries

The preview shows one rect, the window lands in another — a SEV-1 desynchronization.

**Solution:** Replace geometric preview rects with a **dry-run projection** that clones the tree, performs the insertion on the clone, computes the full layout, and returns the exact rect.

During drag, the dragged window is always floating and **not** in the managed layout (`detach_to_floating` is called on press at `mod.rs:1229`), so the clone needs no removal step.

### Step 1: Add `TilingLayout::project_insert`

**File:** `crates/term-wm-core/src/layout/tiling.rs` (after line ~446)

Add a method that clones `self.root` (which derives `Clone`), performs the same insertion logic as `apply_snap` on the clone, computes `root.layout(area)`, and returns the rect for the inserted ID.

**Critical constraint:** When `target` is `None` and `position` is a corner variant (`TopLeft`/`TopRight`/`BottomLeft`/`BottomRight`), the BSP tree **cannot** produce a 25% quarter-screen snap — its fallback `split_root` would bisect the entire area into a vertical 50/50 split, destroying the corner semantics. `project_insert` must return `None` for these cases, forcing the caller to use the geometric `corner_preview_rect`:

```rust
/// Dry-run insert into a cloned layout. Returns the exact `Rect` the
/// inserted leaf would occupy after `apply_snap`.
/// Returns `None` when a corner snap targets the root (no target window)
/// — the BSP tree cannot represent a 25% corner without void nodes.
pub fn project_insert(
    &self,
    target: Option<Id>,
    insert: Id,
    position: InsertPosition,
    area: Rect,
) -> Option<Rect> {
    // BSP cannot natively produce a 25% corner snap against the root.
    // Fall through to geometric corner_preview_rect.
    if target.is_none() {
        match position {
            InsertPosition::TopLeft | InsertPosition::TopRight |
            InsertPosition::BottomLeft | InsertPosition::BottomRight => return None,
            _ => {}
        }
    }
    let mut root = self.root.clone();
    let success = match target {
        Some(t) => root.insert_leaf(t, insert, position),
        None => false,
    };
    if !success {
        let (dir, children) = match position {
            InsertPosition::Left =>
                (Direction::Horizontal, vec![LayoutNode::leaf(insert), root]),
            InsertPosition::Right =>
                (Direction::Horizontal, vec![root, LayoutNode::leaf(insert)]),
            InsertPosition::Top =>
                (Direction::Vertical, vec![LayoutNode::leaf(insert), root]),
            InsertPosition::Bottom =>
                (Direction::Vertical, vec![root, LayoutNode::leaf(insert)]),
            // Corners with a target are handled by the for-loop iteration
            // in LayoutNode::insert_leaf — this arm is dead code here but
            // required for exhaustiveness.
            InsertPosition::TopLeft | InsertPosition::TopRight =>
                (Direction::Vertical, vec![LayoutNode::leaf(insert), root]),
            InsertPosition::BottomLeft | InsertPosition::BottomRight =>
                (Direction::Vertical, vec![root, LayoutNode::leaf(insert)]),
        };
        root = LayoutNode::Split {
            direction: dir, children,
            weights: vec![1.0, 1.0],
            constraints: Vec::new(),
            resizable: true,
        };
    }
    root.layout(area).into_iter()
        .find(|(id, _)| *id == insert)
        .map(|(_, r)| r)
}
```

### Step 2: Add projection cache to `WindowManager`

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (field declaration + reset logic)

Deep cloning the `LayoutNode` tree on every mouse-tick (60+ Hz) would thrash the allocator. The `(target, position, area)` triple that determines the projection result only changes when the user moves between distinct snap zones — not on every (x, y) coordinate tick.

Add a cache field to `WindowManager`:

```rust
/// Cache the last BSP projection result to avoid deep-cloning the
/// layout tree on every drag frame. Keyed by (target, position, area).
snap_projection_cache: Option<(Option<WindowKey>, InsertPosition, Rect, Option<Rect>)>,
```

The tuple is: `(target_key_opt, position, managed_area, projected_rect_opt)`.

**Reset points:** Clear cache at the top of `update_snap_preview` (alongside `self.drag_snap = None`), and in the Release handler.

### Step 3: Use cache + projection in `update_snap_preview`

**File:** `crates/term-wm-core/src/window/window_manager/drag.rs` (lines 150-234)

Replace the three `*_preview_rect()` geometric calls with a cache-check-then-project pattern:

```rust
fn get_projected_preview(
    &mut self,
    target_key_opt: Option<WindowKey>,
    dragging_key: WindowKey,
    pos: InsertPosition,
    area: Rect,
) -> Option<Rect> {
    // Check cache first (avoids clone on every frame)
    if let Some((t, p, a, r)) = &self.snap_projection_cache {
        if *t == target_key_opt && *p == pos && *a == area {
            return *r;
        }
    }
    // Compute fresh
    let rect = self.managed_layout.as_ref()
        .and_then(|layout| layout.project_insert(target_key_opt, dragging_key, pos, area));
    self.snap_projection_cache = Some((target_key_opt, pos, area, rect));
    rect
}
```

Then in each snap case:

```rust
// Corner snap: target = None, fallback = corner_preview_rect
let preview = self.get_projected_preview(None, dragging_key, corner_pos, area)
    .unwrap_or_else(|| {
        let ep = corner_preview_rect(managed_layout_rect, corner_pos);
        Rect { x: ep.x, y: ep.y, width: ep.width, height: ep.height }
    });

// Tiled insert: target = Some(target_key), fallback = tiled_preview_rect
let preview = self.get_projected_preview(Some(target_key), dragging_key, pos, area)
    .unwrap_or_else(|| {
        let ep = tiled_preview_rect(target_layout, pos);
        Rect { x: ep.x, y: ep.y, width: ep.width, height: ep.height }
    });

// Edge snap: target = None, fallback = edge_preview_rect
let preview = self.get_projected_preview(None, dragging_key, pos, area)
    .unwrap_or_else(|| {
        let ep = edge_preview_rect(managed_layout_rect, pos);
        Rect { x: ep.x, y: ep.y, width: ep.width, height: ep.height }
    });
```

**Why corner snaps always use geometric fallback:** `project_insert` returns `None` for `target=None` + corner position (Step 1), so `get_projected_preview` returns `None`, and the `unwrap_or_else` falls through to `corner_preview_rect`. This is correct because:
- No other windows exist → `apply_snap` keeps the window floating at the geometric rect (line 247-262)
- Other windows exist → `apply_snap` calls `split_root(key, corner_pos)` which does a 50% vertical split. This is a pre-existing limitation in `apply_snap`'s corner handling (it doesn't truly quarter-screen tile), and fixing it is out of scope for this preview-accuracy change.

**Cache miss rate:** The cache is keyed by `(target, position, area)`. During a single drag, these only change when the user moves between snap zones (e.g., from corner to edge to tiled insert). The typical miss rate is ≤1 per snap-zone transition. Continuous drag within the same zone hits the cache every frame — zero allocations.

## Bug Fixes: Critical Regressions (from Code Review)

### Fix 1: Duplicate BSP Leaf in `project_insert` (SEV-1)

**File:** `crates/term-wm-core/src/layout/tiling.rs` (method `project_insert`)

**Root Cause:** `detach_to_floating` does NOT call `remove_leaf` — it only sets a floating rect. The window's leaf stays intact in the BSP tree. When `project_insert` clones the tree and inserts the same key again, a duplicate leaf exists → corrupted layout rects.

**Fix:** Add `root.remove_leaf(insert)` before the insertion logic:

```rust
let mut root = self.root.clone();
root.remove_leaf(insert);    // <-- cull stale leaf before re-insertion
let success = match target {
    Some(t) => root.insert_leaf(t, insert, position),
    None => false,
};
```

`LayoutNode::remove_leaf` returns `false` if the leaf isn't found (e.g., the window was never in the layout), which is fine — noop.

### Fix 2: Focus-Induced Geometry Collapse (SEV-1)

**File:** `crates/term-wm-core/src/window/window_manager/focus.rs`

**Root Cause:** Both `focus_app_window` and `focus_window_key` call `self.unmaximize_window(prev)` on the previously focused window. A maximized window unmaximizes when another window is focused — violating the mandate that focus shifts must not affect geometry.

**Fix:** Remove the `unmaximize_window` call from both functions:

```rust
pub fn focus_app_window(&mut self, key: WindowKey) {
    let prev = *self.focus.current();
    if prev != key {
        // REMOVED: self.unmaximize_window(prev);
    }
    self.focus.set_current(key);
    self.bring_to_front_key(key);
    self.managed_draw_order = self.z_order.clone();
    self.mark_layout_dirty();
}
```

Delete the identical block from `focus_window_key` as well. Maximized floating windows are at the top of the z-order and naturally occlude lower windows — the user must manually toggle_maximize to restore.

### Fix 3: Invisible Top-Edge Maximize Preview (SEV-2)

**File:** `crates/term-wm-core/src/window/window_manager/drag.rs` (method `update_snap_preview`)

**Root Cause:** The `mouse_y == 0` branch sets `self.drag_snap = None`, but the renderer reads the preview rect exclusively from `self.drag_snap_rect_data()`. No rect → no ghost preview renders.

**Fix:** Set `drag_snap` to the full managed area so the renderer can draw the ghost outline:

```rust
// Priority 2: Sacred top edge — full-screen maximize (deferred to release)
if mouse_y == 0 {
    self.drag_snap = Some((None, InsertPosition::Top, area));
    self.snap_preview = Some(SnapPreviewState::Maximize);
    return;
}
```

The `area` is the full `managed_area` — exactly what the maximize will occupy. The preview renders, and on release the existing `toggle_maximize` commit fires.
