# Plan: Advanced Window Physics for term-wm

## Overview
Implement 5 phases of window physics on top of the existing layout engine, which already has: edge snap detection, `EdgeResistance` (X-axis only), quadrant-based tiling insertion, maximize/unmaximize with `prev_floating_rect`, drag-to-detach from tiling, and floating resize.

Layout engine (`term-wm-layout-engine`) is `no_std` and must remain so — pure math only. All wall-clock time tracking lives in the core crate.

## Phase 1: Edge Snapping and State Restoration

### 1a. Sacred Top Edge (Preview-then-Commit) — Corner-First Priority
**Files:** `crates/term-wm-core/src/window/window_manager/mod.rs`, `drag.rs`; `crates/term-wm-console/src/draw_plan_renderer.rs`; `src/lib.rs`

**Current:** Top-edge panels clamp y to `bounds_y`; no maximize-on-top-edge behavior.  
**Change (Preview-then-Commit, corner-first spatial priority):**
1. **Spatial priority order** (smallest region first): `corner_snap → top_edge_maximize → edge_snap → tiled_insert`. The hit-test matrix in `update_snap_preview` must evaluate in this order so corners (which occupy `row == 0` + an X edge) are not masked by the generic `row == 0` check.
2. In `dispatch_mouse` Phase-1 `DraggingWindow` handler (`mod.rs:871-892`):
   - First: call `detect_corner_snap(...)` — if it matches, set `SnapPreviewState::Corner(pos)`.
   - Second (only if corner is `None`): check `row == 0` (absolute terminal top) — if matched, set `SnapPreviewState::Maximize`.
   - Third (only if both are `None`): proceed with existing `detect_edge_snap` and `detect_quadrant` for edge/tiled snap.
   - **Do NOT call `toggle_maximize` immediately.** Ghost previews render at the appropriate rect during drag.
   - On `Release`: commit the snap based on the current `SnapPreviewState` (Maximize → `toggle_maximize`, Corner/Edge/TiledInsert → existing `apply_snap`).
   - On drag leaving the trigger zone by >3 cells: clear `SnapPreviewState`.
3. Add `SnapPreviewState` enum to mod.rs:
   ```rust
   pub(crate) enum SnapPreviewState {
       Edge(InsertPosition),
       Corner(InsertPosition),  // TopLeft/TopRight/BottomLeft/BottomRight
       TiledInsert(WindowKey, InsertPosition),
       Maximize,                // full-screen outline
   }
   ```

### 1b. Corner Quarter-Screen Snap
**Files:** `crates/term-wm-layout-engine/src/snap.rs`

**Current:** `InsertPosition` has only `Left/Right/Top/Bottom`; `edge_preview_rect` does 50% splits only. No corner detection.  
**Changes:**
- Add `InsertPosition::TopLeft | TopRight | BottomLeft | BottomRight` variants.
- Implement `to_orientation()`: TopLeft/BottomLeft → Horizontal, TopRight/BottomRight → Horizontal (all corners create vertical BSP splits). Actually each corner maps: TopLeft = Left+Top combined. For preview rect it's a quarter of the managed area.
- Add `detect_corner_snap(col, row, managed_area, sensitivity) -> Option<InsertPosition>`:
  - Use independent orthogonal axis checks with saturating arithmetic (no underflow panics):
    - **TopLeft**:      `col <= sensitivity + margin_left && row <= sensitivity + margin_top`
    - **TopRight**:     `col >= margin_right.saturating_sub(sensitivity) && row <= sensitivity + margin_top`
    - **BottomLeft**:   `col <= sensitivity + margin_left && row >= margin_bottom.saturating_sub(sensitivity)`
    - **BottomRight**:  `col >= margin_right.saturating_sub(sensitivity) && row >= margin_bottom.saturating_sub(sensitivity)`
    - where `margin_left = x`, `margin_right = x + width - 1`, `margin_top = y`, `margin_bottom = y + height - 1`.
  - Priority order (tiebreak): TopLeft > TopRight > BottomLeft > BottomRight when the cursor is in a corner overlap zone.
- Add `corner_preview_rect(managed_area, pos) -> LayoutRect`:
  - TopLeft: x=area.x, y=area.y, w=area.w/2, h=area.h/2
  - TopRight: x=area.x+area.w/2, y=area.y, w=area.w/2, h=area.h/2
  - BottomLeft: x=area.x, y=area.y+area.h/2, w=area.w/2, h=area.h/2
  - BottomRight: x=area.x+area.w/2, y=area.y+area.h/2, w=area.w/2, h=area.h/2
- Update `SnapTarget` with a `Maximize` variant (for top-edge full-screen).
- Update `edge_preview_rect` and `tiled_preview_rect` to handle all 8 InsertPositions.

### 1c. Kinetic Restoration from Maximized (Flag-Based)
**Files:** `crates/term-wm-core/src/window/entry.rs`, `window/window_manager/drag.rs`, `mod.rs`, `chrome.rs`

**Current:** No `is_maximized` flag — relies on brittle rect comparison. Dragging a maximized window moves it at full width/height.  
**Changes:**

1. **Add `is_maximized` boolean flag to `Window`:**
   ```rust
   // crates/term-wm-core/src/window/entry.rs
   pub is_maximized: bool,
   ```
   Initialize to `false` in constructor.

2. **Set on maximize, clear on unmaximize:**
   - `toggle_maximize` (`chrome.rs`): when entering maximized state, set `is_maximized = true`; when restoring, set `is_maximized = false`.
   - `unmaximize_window` (`focus.rs`): set `is_maximized = false`.

3. **In `dispatch_mouse` Phase-4 `HeaderAction::Drag` handler (`mod.rs:1151-1186`):**
   - Read `window.is_maximized` (not rect comparison).
   - If maximized: call `toggle_maximize(key)` to restore `prev_floating_rect`, then compute new `initial_x`, `initial_y` from the restored rect.
   - **Proportional grab offset** (no teleportation): compute the cursor's offset within the maximized window proportional to the restored size. Use multiplication-first to avoid integer truncation:
     ```rust
     let max_w = u32::from(max_width).max(1);  // guard against div-by-zero
     let max_h = u32::from(max_height).max(1);
     let x_offset = (u32::from(col.saturating_sub(max_rect.x)) * u32::from(restored_width)) / max_w;
     let initial_x = col.saturating_sub(x_offset as u16);
     let y_offset = (u32::from(row.saturating_sub(max_rect.y)) * u32::from(restored_height)) / max_h;
     let initial_y = row.saturating_sub(y_offset as u16);
     ```
     Clamp final values to terminal bounds. This preserves the exact grab point: if you grabbed at 90% of the maximized title bar (x_offset = 0.9 * max_width), the cursor stays at 90% of the restored title bar (x_offset = 0.9 * restored_width).
   - Set `MouseCaptureState::DraggingWindow` as before; the restored (smaller) floating rect persists, so `move_floating` operates on the correct pre-maximized size.

## Phase 2: Magnetic Edge Resistance

**Current:** `move_floating` calls `resistance.apply_x()` but NOT `resistance.apply_y()`.

### 2b. Temporal Threshold — Pure Integer Timestamp (no std::time in engine)
**File:** `crates/term-wm-layout-engine/src/snap.rs` + `crates/term-wm-core/src/window/window_manager/drag.rs` + `mod.rs`

**Current:** `EdgeResistanceConfig._temporal_threshold_ns` exists (always `None`), but `std::time::Instant` would violate `no_std`.  
**Changes:**

1. **Layout engine (`snap.rs`):** Keep `no_std`. Change `_temporal_threshold_ns: Option<u64>` → `temporal_threshold_ns: Option<u64>` (default `Some(500_000_000)` for 500ms).
   - Add `entered_magnetic_at_ns: Option<u64>` field to `EdgeResistance`.
   - Modify `snap_axis` to accept `now_ns: Option<u64>` — a raw nanosecond timestamp from the event loop.
   - When `already_snapped` AND `now_ns` sufficiently exceeds `entered_magnetic_at_ns`, allow the value through (break magnetic lock).
   - The layout engine treats the timestamp as an opaque integer; it performs only saturating subtraction (`now - entered`) and comparison. No `std::time` types.

2. **Core crate (`mod.rs`, `drag.rs`):** Capture `Instant::now()` → `duration_since_epoch().as_nanos() as u64` at the start of each drag event dispatch. Pass this `u64` through `move_floating` into `EdgeResistance::apply_x(_, _, now_ns)` and `apply_y(_, _, now_ns)`.

### 2c. Velocity Threshold (Fast-Move-to-Break)
**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (+ drag.rs)

**Current:** `DraggingWindow` has no velocity.  
**Changes:**
1. Add `prev_col: u16, prev_row: u16, prev_time_ns: u64` fields to `MouseCaptureState::DraggingWindow`.
2. In `dispatch_mouse` Phase-1: on each Drag event, compute dx, dy, dt from current vs previous. Use squared-distance comparison (no sqrt, pure integer):
   ```rust
   let dist_sq = u32::from(dx.abs_diff(0)).pow(2) + u32::from(dy.abs_diff(0)).pow(2);
   // threshold = 10 cells per 100ms → 100 cells² per (100ms)²
   let threshold_dist_sq = (speed_threshold_cells_per_ms * dt_ms).pow(2) as u32;
   if dist_sq > threshold_dist_sq {
       // velocity exceeded: force snap off for this frame
   }
   ```
3. Default threshold: ~10 cells/100ms (configurable via `WmConfig`). Pass `velocity_exceeded: bool` into `EdgeResistance::apply_x/y` → when true, skip magnetic snapping for that frame.

## Phase 3: Angular Quadrant Detection

**Files:** `crates/term-wm-layout-engine/src/hit_test.rs`, `crates/term-wm-core/src/window/window_manager/drag.rs`

**Current:** `detect_quadrant` uses axis-aligned bisection (`|dx| > |dy|`). This is incorrect per spec.  
**Changes:**

Replace `detect_quadrant` with a dimension-scaled cross-product to handle arbitrary target aspect ratios. The raw `|dx| > |dy|` comparison only works for perfect squares — for a 100×20 pane it creates massive horizontal drop zones and microscopic vertical ones.

```rust
/// Divide target rect into four diagonal quadrants by calculating the
/// angle of the cursor relative to the geometric center, scaled by the
/// target's aspect ratio.
///
/// Uses dimension-scaled cross-product (no floating-point, no sqrt):
///   scaled_dx = |dx| * target.height
///   scaled_dy = |dy| * target.width
///
/// This guarantees the diagonal dividing lines always pass through the
/// center at the correct slope for the rectangle's aspect ratio,
/// producing equal-area triangular quadrants regardless of shape.
pub fn detect_quadrant(cursor_col: u16, cursor_row: u16, target: &LayoutRect) -> Quadrant {
    let (cx, cy) = target.center();
    let dx = i32::from(cursor_col).saturating_sub(cx);
    let dy = i32::from(cursor_row).saturating_sub(cy);

    if dx == 0 && dy == 0 {
        return Quadrant::East;
    }

    let adx = u32::from(dx.unsigned_abs());
    let ady = u32::from(dy.unsigned_abs());
    let w = u32::from(target.width);
    let h = u32::from(target.height);

    // Scale by the opposite dimension: the true diagonal of a rectangle
    // has slope ±(height/width), so |dx|*h > |dy|*w means the cursor is
    // the East/West diagonal quadrant.
    let scaled_dx = adx.saturating_mul(h);
    let scaled_dy = ady.saturating_mul(w);

    if scaled_dx > scaled_dy || (scaled_dx == scaled_dy && dx >= 0) {
        // East/West diagonal quadrant (cursor is within the diagonal band)
        if dx >= 0 { Quadrant::East } else { Quadrant::West }
    } else {
        // North/South diagonal quadrant
        if dy < 0 { Quadrant::North } else { Quadrant::South }
    }
}
```

**Why this works:** Multiplying `|dx|` by `target.height` and `|dy|` by `target.width` transforms the coordinate system into a unit-square space where the diagonals are at exactly 45°. In the original space, the comparison `|dx| * h > |dy| * w` is equivalent to checking `|dx| / w > |dy| / h` — i.e., whether the slope from center to cursor is shallower than the true geometric diagonal `(±h/±w)`. This produces drop zones that are exact diagonal quarter-circles (in linear space) matching the actual aspect ratio.

**Update existing tests** to verify behavior on non-square targets (e.g., 100×20 pane).

**File:** `crates/term-wm-layout-engine/src/floating.rs`

**Current:** `apply_resize_drag_signed` clamps min size (already done correctly — lines 250-267 clamp `w = w.max(min_w)`, `h = h.max(min_h)`).  
**Change:** None needed for floating — keep the existing clamping behavior.
