# Plan: Advanced Window Physics for term-wm

## Overview
Implement 5 phases of window physics on top of the existing layout engine, which already has: edge snap detection, `EdgeResistance` (X-axis only), quadrant-based tiling insertion, maximize/unmaximize with `prev_floating_rect`, drag-to-detach from tiling, and floating resize.

Layout engine (`term-wm-layout-engine`) is `no_std` and must remain so â€” pure math only. All wall-clock time tracking lives in the core crate.

### 1b. Corner Quarter-Screen Snap
**Files:** `crates/term-wm-layout-engine/src/snap.rs`

**Current:** `InsertPosition` has only `Left/Right/Top/Bottom`; `edge_preview_rect` does 50% splits only. No corner detection.  
**Changes:**
- Add `InsertPosition::TopLeft | TopRight | BottomLeft | BottomRight` variants.
- Implement `to_orientation()`: TopLeft/BottomLeft â†’ Horizontal, TopRight/BottomRight â†’ Horizontal (all corners create vertical BSP splits). Actually each corner maps: TopLeft = Left+Top combined. For preview rect it's a quarter of the managed area.
- Add `detect_corner_snap(col, row, managed_area, sensitivity) -> Option<InsertPosition>`:
  - Use independent orthogonal axis checks with saturating arithmetic exclusively (no raw `+` or `-`):
    - **TopLeft**:      `col <= sensitivity.saturating_add(margin_left) && row <= sensitivity.saturating_add(margin_top)`
    - **TopRight**:     `col >= margin_right.saturating_sub(sensitivity) && row <= sensitivity.saturating_add(margin_top)`
    - **BottomLeft**:   `col <= sensitivity.saturating_add(margin_left) && row >= margin_bottom.saturating_sub(sensitivity)`

  - Priority order (tiebreak): TopLeft > TopRight > BottomLeft > BottomRight when the cursor is in a corner overlap zone.
- Add `corner_preview_rect(managed_area, pos) -> LayoutRect`:
  - TopLeft: x=area.x, y=area.y, w=area.w/2, h=area.h/2
  - TopRight: x=area.x+area.w/2, y=area.y, w=area.w/2, h=area.h/2
  - BottomLeft: x=area.x, y=area.y+area.h/2, w=area.w/2, h=area.h/2

- Update `SnapTarget` with a `Maximize` variant (for top-edge full-screen).
- Update `edge_preview_rect` and `tiled_preview_rect` to handle all 8 InsertPositions.

## Phase 3: Angular Quadrant Detection

**Files:** `crates/term-wm-layout-engine/src/hit_test.rs`, `crates/term-wm-core/src/window/window_manager/drag.rs`

**Current:** `detect_quadrant` uses axis-aligned bisection (`|dx| > |dy|`). This is incorrect per spec.  
**Changes:**

Replace `detect_quadrant` with a dimension-scaled cross-product to handle arbitrary target aspect ratios. The raw `|dx| > |dy|` comparison only works for perfect squares â€” for a 100Ă—20 pane it creates massive horizontal drop zones and microscopic vertical ones.

```rust
/// Divide target rect into four diagonal quadrants by calculating the
/// angle of the cursor relative to the geometric center, scaled by the
/// target's aspect ratio.
///
/// Uses dimension-scaled cross-product (no floating-point, no sqrt):
///   scaled_dx = |dx| * target.height
///   scaled_dy = |dy| * target.width
///
/// This guarantees the diagonal dividing lines always pass through the
/// center at the correct slope for the rectangle's aspect ratio,
/// producing equal-area triangular quadrants regardless of shape.
pub fn detect_quadrant(cursor_col: u16, cursor_row: u16, target: &LayoutRect) -> Quadrant {
    let (cx, cy) = target.center();
    let dx = i32::from(cursor_col).saturating_sub(cx);
    let dy = i32::from(cursor_row).saturating_sub(cy);

    if dx == 0 && dy == 0 {
        return Quadrant::East;
    }

    let adx = u32::from(dx.unsigned_abs());
    let ady = u32::from(dy.unsigned_abs());
    let w = u32::from(target.width);
    let h = u32::from(target.height);

    // Scale by the opposite dimension: the true diagonal of a rectangle
    // has slope Â±(height/width), so |dx|*h > |dy|*w means the cursor is
    // the East/West diagonal quadrant.
    let scaled_dx = adx.saturating_mul(h);
    let scaled_dy = ady.saturating_mul(w);

    if scaled_dx > scaled_dy || (scaled_dx == scaled_dy && dx >= 0) {
        // East/West diagonal quadrant (cursor is within the diagonal band)
        if dx >= 0 { Quadrant::East } else { Quadrant::West }
    } else {
        // North/South diagonal quadrant
        if dy < 0 { Quadrant::North } else { Quadrant::South }
    }
}
```

**Why this works:** Multiplying `|dx|` by `target.height` and `|dy|` by `target.width` transforms the coordinate system into a unit-square space where the diagonals are at exactly 45Â°. In the original space, the comparison `|dx| * h > |dy| * w` is equivalent to checking `|dx| / w > |dy| / h` â€” i.e., whether the slope from center to cursor is shallower than the true geometric diagonal `(Â±h/Â±w)`. This produces drop zones that are exact diagonal quarter-circles (in linear space) matching the actual aspect ratio.

**Update existing tests** to verify behavior on non-square targets (e.g., 100Ă—20 pane).

## Phase 3b: BSP Dry-Run Projection for Accurate Previews

**Files:** `crates/term-wm-core/src/layout/tiling.rs`, `crates/term-wm-core/src/window/window_manager/mod.rs`, `crates/term-wm-core/src/window/window_manager/drag.rs`

**Root Cause:** All snap preview rects (`edge_preview_rect`, `corner_preview_rect`, `tiled_preview_rect`) are pure geometric approximations (simple width/height halving). But `apply_snap` inserts the window into the `LayoutNode` BSP tree, which produces a **different** final geometry because:
- BSP rects come from recursive splitting, not simple division
- Gaps are subtracted before splitting via `split_rects_with_gaps`
- The tree structure (weights, ratios) determines exact boundaries

The preview shows one rect, the window lands in another â€” a SEV-1 desynchronization.

**Solution:** Replace geometric preview rects with a **dry-run projection** that clones the tree, performs the insertion on the clone, computes the full layout, and returns the exact rect.

During drag, the dragged window is always floating and **not** in the managed layout (`detach_to_floating` is called on press at `mod.rs:1229`), so the clone needs no removal step.

### Step 1: Add `TilingLayout::project_insert`

**File:** `crates/term-wm-core/src/layout/tiling.rs` (after line ~446)

Add a method that clones `self.root` (which derives `Clone`), performs the same insertion logic as `apply_snap` on the clone, computes `root.layout(area)`, and returns the rect for the inserted ID.

**Critical constraint:** When `target` is `None` and `position` is a corner variant (`TopLeft`/`TopRight`/`BottomLeft`/`BottomRight`), the BSP tree **cannot** produce a 25% quarter-screen snap â€” its fallback `split_root` would bisect the entire area into a vertical 50/50 split, destroying the corner semantics. `project_insert` must return `None` for these cases, forcing the caller to use the geometric `corner_preview_rect`:

```rust
/// Dry-run insert into a cloned layout. Returns the exact `Rect` the
/// inserted leaf would occupy after `apply_snap`.
/// Returns `None` when a corner snap targets the root (no target window)
/// â€” the BSP tree cannot represent a 25% corner without void nodes.
pub fn project_insert(
    &self,
    target: Option<Id>,
    insert: Id,
    position: InsertPosition,
    area: Rect,
) -> Option<Rect> {
    // BSP cannot natively produce a 25% corner snap against the root.
    // Fall through to geometric corner_preview_rect.
    if target.is_none() {
        match position {
            InsertPosition::TopLeft | InsertPosition::TopRight |
            InsertPosition::BottomLeft | InsertPosition::BottomRight => return None,
            _ => {}
        }
    }
    let mut root = self.root.clone();
    let success = match target {
        Some(t) => root.insert_leaf(t, insert, position),
        None => false,
    };
    if !success {
        let (dir, children) = match position {
            InsertPosition::Left =>
                (Direction::Horizontal, vec![LayoutNode::leaf(insert), root]),
            InsertPosition::Right =>
                (Direction::Horizontal, vec![root, LayoutNode::leaf(insert)]),
            InsertPosition::Top =>
                (Direction::Vertical, vec![LayoutNode::leaf(insert), root]),
            InsertPosition::Bottom =>
                (Direction::Vertical, vec![root, LayoutNode::leaf(insert)]),
            // Corners with a target are handled by the for-loop iteration
            // in LayoutNode::insert_leaf â€” this arm is dead code here but
            // required for exhaustiveness.
            InsertPosition::TopLeft | InsertPosition::TopRight =>
                (Direction::Vertical, vec![LayoutNode::leaf(insert), root]),
            InsertPosition::BottomLeft | InsertPosition::BottomRight =>
                (Direction::Vertical, vec![root, LayoutNode::leaf(insert)]),
        };
        root = LayoutNode::Split {
            direction: dir, children,
            weights: vec![1.0, 1.0],
            constraints: Vec::new(),
            resizable: true,
        };
    }
    root.layout(area).into_iter()
        .find(|(id, _)| *id == insert)
        .map(|(_, r)| r)
}
```

### Step 2: Add projection cache to `WindowManager`

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (field declaration + reset logic)

Deep cloning the `LayoutNode` tree on every mouse-tick (60+ Hz) would thrash the allocator. The `(target, position, area)` triple that determines the projection result only changes when the user moves between distinct snap zones â€” not on every (x, y) coordinate tick.

Add a cache field to `WindowManager`:

```rust

/// layout tree on every drag frame. Keyed by SnapPreviewState + area.
/// SnapPreviewState captures all snap variants including VoidInsert(Rect).
snap_projection_cache: Option<(SnapPreviewState, Rect, Option<Rect>)>,
```

The tuple is: `(snap_preview_state, managed_area, projected_rect_opt)`.

**Reset points:** Clear cache at the top of `update_snap_preview` (alongside `self.drag_snap = None`), and in the Release handler.

### Step 3: Use cache + projection in `update_snap_preview`

**File:** `crates/term-wm-core/src/window/window_manager/drag.rs` (lines 150-234)

Replace the three `*_preview_rect()` geometric calls with a cache-check-then-project pattern. The cache key is `(SnapPreviewState, area)` â€” every snap variant (including `VoidInsert(Rect)`) is captured by `SnapPreviewState`:

```rust
fn get_projected_preview(
    &mut self,
    dragging_key: WindowKey,
    state: SnapPreviewState,
    area: Rect,
) -> Option<Rect> {
    // Check cache first (avoids clone on every frame)
    if let Some((s, a, r)) = &self.snap_projection_cache {
        if *s == state && *a == area {
            return *r;
        }
    }
    // Compute fresh based on state
    let rect = match state {
        SnapPreviewState::Corner(pos) | SnapPreviewState::Edge(pos) => {
            self.managed_layout.as_ref()
                .and_then(|layout| layout.project_insert(None, dragging_key, pos, area))
        }
        SnapPreviewState::TiledInsert(target_key, pos) => {
            self.managed_layout.as_ref()
                .and_then(|layout| layout.project_insert(Some(target_key), dragging_key, pos, area))
        }
        SnapPreviewState::VoidInsert(void_rect) => {
            self.managed_layout.as_ref()
                .and_then(|layout| layout.project_insert_void(dragging_key, void_rect, area))
        }
        SnapPreviewState::Maximize => None,
    };
    self.snap_projection_cache = Some((state, area, rect));
    rect
}
```

Then in each snap case:

```rust
// Corner snap: fallback = corner_preview_rect
let preview = self
    .get_projected_preview(dragging_key, SnapPreviewState::Corner(corner_pos), area)
    .unwrap_or_else(|| {
        let ep = corner_preview_rect(managed_layout_rect, corner_pos);
        Rect { x: ep.x, y: ep.y, width: ep.width, height: ep.height }
    });

// Tiled insert: fallback = tiled_preview_rect
let preview = self
    .get_projected_preview(dragging_key, SnapPreviewState::TiledInsert(target_key, pos), area)
    .unwrap_or_else(|| {
        let ep = tiled_preview_rect(target_layout, pos);
        Rect { x: ep.x, y: ep.y, width: ep.width, height: ep.height }
    });

// Edge snap: fallback = edge_preview_rect
let preview = self
    .get_projected_preview(dragging_key, SnapPreviewState::Edge(pos), area)
    .unwrap_or_else(|| {
        let ep = edge_preview_rect(managed_layout_rect, pos);
        Rect { x: ep.x, y: ep.y, width: ep.width, height: ep.height }
    });
```

**Why corner snaps always use geometric fallback:** `project_insert` returns `None` for `target=None` + corner position (Step 1), so `get_projected_preview` returns `None`, and the `unwrap_or_else` falls through to `corner_preview_rect`. This is correct because:
- No other windows exist â†’ `apply_snap` keeps the window floating at the geometric rect (line 247-262)
- Other windows exist â†’ `apply_snap` calls `split_root(key, corner_pos)` which does a 50% vertical split. This is a pre-existing limitation in `apply_snap`'s corner handling (it doesn't truly quarter-screen tile), and fixing it is out of scope for this preview-accuracy change.

**Cache miss rate:** The cache is keyed by `(target, position, area)`. During a single drag, these only change when the user moves between snap zones (e.g., from corner to edge to tiled insert). The typical miss rate is â‰¤1 per snap-zone transition. Continuous drag within the same zone hits the cache every frame â€” zero allocations.

## Bug Fixes: Critical Regressions (from Code Review)

### Fix 1: Duplicate BSP Leaf in `project_insert` (SEV-1)

**File:** `crates/term-wm-core/src/layout/tiling.rs` (method `project_insert`)

**Root Cause:** `detach_to_floating` does NOT call `remove_leaf` â€” it only sets a floating rect. The window's leaf stays intact in the BSP tree. When `project_insert` clones the tree and inserts the same key again, a duplicate leaf exists â†’ corrupted layout rects.

**Fix:** Add `root.remove_leaf(insert)` before the insertion logic:

```rust
let mut root = self.root.clone();
root.remove_leaf(insert);    // <-- cull stale leaf before re-insertion
let success = match target {
    Some(t) => root.insert_leaf(t, insert, position),
    None => false,
};
```

    /// `target_rect` within `area`, and replace it with `new_leaf`.
    /// Returns true if a void was found and replaced.
    pub fn replace_void_at_rect(
        &mut self,

pub fn replace_void_by_id(&mut self, void_id: usize, new_leaf: LayoutNode<Id>) -> bool {
    match self {

fn void_regions_recursive(&self, area: Rect, out: &mut Vec<(usize, Rect)>) {
    match self {
        LayoutNode::Void(id) => out.push((*id, area)),
        LayoutNode::Split { direction, children, weights, constraints, resizable } => {

            }
        }
        _ => {}
    }
}

## Fix 11: Timer-Expired Maximize â€” Stuck Preview (SEV-2)

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (method `apply_drag_snap_if_pending`)

**Root Cause:** When the user drags to the top edge (y=0) and their mouse leaves the terminal, the drag-snap timer expires and `apply_drag_snap_if_pending()` fires. It unconditionally calls `apply_snap(key)`, which uses `drag_snap` to insert the window into the layout tree â€” it never checks `SnapPreviewState::Maximize`. The snap is never committed, `drag_snap` stays set, and the ghost preview renders forever.

**Fix:** Have `apply_drag_snap_if_pending` check `SnapPreviewState` and handle `Maximize` the same way the Release handler does â€” by calling `toggle_maximize(key)`:

```rust
pub fn apply_drag_snap_if_pending(&mut self) {
    if let Some(capture) = self.mouse_capture.take()
        && let MouseCaptureState::DraggingWindow { key, .. } = capture
    {
        self.drag_last_event = None;
        self.drag_timer_id = None;
        if self.snap_preview == Some(SnapPreviewState::Maximize) {
            self.toggle_maximize(key);
        } else if self.drag_snap.is_some() {
            self.apply_snap(key);
        }
    }
    // Unconditional flush â€” prevents stale ghost previews and cache bleed
    self.drag_snap = None;
    self.snap_preview = None;
    self.snap_projection_cache = None;
}
```

This mirrors the exact logic from the Release handler (line 984-990), ensuring the timer path and the interactive path handle `Maximize` identically.
