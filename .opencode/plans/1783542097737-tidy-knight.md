# Plan: Advanced Window Physics for term-wm

## Overview
Implement 5 phases of window physics on top of the existing layout engine, which already has: edge snap detection, `EdgeResistance` (X-axis only), quadrant-based tiling insertion, maximize/unmaximize with `prev_floating_rect`, drag-to-detach from tiling, and floating resize.

---

## Phase 1: Edge Snapping and State Restoration

### 1a. Sacred Top Edge
**Files:** `crates/term-wm-core/src/window/window_manager/drag.rs`, `mod.rs`

**Current:** Top-edge panels clamp y to `bounds_y`; no maximize-on-top-edge behavior.
**Change:** In `dispatch_mouse` Phase-1 `DraggingWindow` handler (`mod.rs:871-892`), before `move_floating()`, check if `row == 0` (absolute terminal top). If so, call `toggle_maximize(key)` and release capture instead of dragging. This must NOT trigger for snap-to-tiling â€” the top edge is sacred.

### 1b. Corner Quarter-Screen Snap
**Files:** `crates/term-wm-layout-engine/src/snap.rs`, `rect.rs`

**Current:** `InsertPosition` has only `Left/Right/Top/Bottom`; `edge_preview_rect` does 50% splits only. No corner detection.
**Changes:**
- Add `InsertPosition::TopLeft | TopRight | BottomLeft | BottomRight` variants.
- Add `detect_corner_snap(col, row, managed_area, sensitivity) -> Option<InsertPosition>` â€” returns corner variant when cursor is within `sensitivity` of BOTH an X edge AND a Y edge simultaneously.
- Add `corner_preview_rect(managed_area, pos) -> LayoutRect` â€” returns 50%Ă—50% rect in the appropriate corner.
- Add `Corner` variant to `SnapTarget` (or reuse `Edge` with new InsertPositions).
- Update `tiled_preview_rect` to handle all 8 InsertPositions (the 4 tiled corners).

### 1c. Kinetic Restoration from Maximized
**Files:** `crates/term-wm-core/src/window/window_manager/drag.rs`, `mod.rs`, `chrome.rs`

**Current:** Dragging a maximized window (floating_rect == full managed_area) moves it at full width/height. No unmaximize-on-drag.
**Changes:**
1. Add `is_maximized(key) -> bool` helper to `chrome.rs` â€” checks if floating rect equals full managed area.
2. In `dispatch_mouse` Phase-4 `HeaderAction::Drag` handler (`mod.rs:1151-1186`), after bringing to front but before setting `MouseCaptureState::DraggingWindow`:
   - If `is_maximized(key)`, call `toggle_maximize(key)` first (which restores `prev_floating_rect`).
   - Then compute `initial_x`, `initial_y` from the *restored* floating rect.
   - Re-center under cursor: `initial_x = col - (restored_width / 2)`, `initial_y = row - (restored_height / 2)`, clamped to bounds.
3. Store the restored frame in `MouseCaptureState::DraggingWindow` (or recompute from current floating rect) so `move_floating` uses the restored size, not full-screen size.

## Phase 2: Magnetic Edge Resistance

### 2a. Y-Axis Edge Resistance
**File:** `crates/term-wm-core/src/window/window_manager/drag.rs`

**Current:** `move_floating` (line 91) calls `resistance.apply_x()` but NOT `resistance.apply_y()` â€” the `apply_y` method exists on `EdgeResistance` but is dead code.
**Change:** After `apply_x`, add: `let y = resistance.apply_y(y, bounds_layout);` â€” this wires the existing hysteresis-based Y-axis magnetic zone. Also keep the panel-y clamp (panels take priority over resistance when active).

### 2b. Temporal Threshold (Hold-to-Break)
**File:** `crates/term-wm-layout-engine/src/snap.rs`

**Current:** `EdgeResistanceConfig._temporal_threshold_ns: Option<u64>` exists as a placeholder, always `None`. No time-based escape from magnetic zone.
**Changes:**
1. Rename `_temporal_threshold_ns` â†’ `temporal_threshold_ns` and set default to `500_000_000` (500ms).
2. Add `entered_zone_at: Option<Instant>` field to `EdgeResistance` (needs `std::time`).
3. Add `entered_magnetic: bool` field â€” set to `true` when `snap_axis` returns a snapped value (low or high) for the first time.
4. Modify `snap_axis` to accept an `Option<Instant>` â€” if `already_snapped` AND enough time has passed (> `temporal_threshold_ns`), allow the value through (break magnetic lock).
5. Wrap `EdgeResistance::apply_x` and `apply_y` to track time-in-zone.

**Note:** This requires `Instant` from `std::time`. The layout-engine crate is currently `no_std` â€” either gate behind `std` feature or add the timestamp as an `Option<u64>` (nanos passed in externally).

### 2c. Velocity Threshold (Fast-Move-to-Break)
**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (and `drag.rs`)

**Current:** `DraggingWindow` only stores initial+start coords, no velocity.
**Changes:**
1. Add `prev_col: u16, prev_row: u16, prev_time: Option<Instant>` fields to `MouseCaptureState::DraggingWindow`.
2. In `dispatch_mouse` Phase-1 (line 882), on each Drag event update these fields.
3. Pass computed velocity (cells/ms) into `EdgeResistance` â€” if velocity exceeds a threshold (e.g., 10 cells/100ms), skip magnetic snapping entirely for that frame.
4. Add velocity threshold config to `WmConfig` or `EdgeResistanceConfig`.

## Phase 3: Quadrant Detection for Tiling Integration

**Files:** `crates/term-wm-layout-engine/src/hit_test.rs`, `crates/term-wm-core/src/window/window_manager/drag.rs`

**Current:** `detect_quadrant` uses axis-aligned bisection (`|dx| > |dy|` â†’ East/West, else North/South). This maps correctly to `InsertPosition::Left/Right/Top/Bottom`. The behavior is functionally correct for the user's spec (East â†’ vertical BSP split, right child).

**Changes required: Minimal / none.** The spec asks for "diagonal quadrants" but the axis-aligned approach produces the correct insertion behavior. However, to match the spec exactly:
1. Add a `diagonal_quadrant(col, row, target) -> Quadrant` function that uses atan2-style angle comparison.
2. The existing axis-aligned version sees "almost exactly East" when slightly above-right; the diagonal version would see "North" in that case. Leave both available; keep using the current axis-aligned version by default (it's simpler and matches user expectations better).
3. **No functional change needed** â€” the current mapping produces the same tiling outcomes.

## Phase 5: Strict Size Constraints (Request Rejection)

**File:** `crates/term-wm-layout-engine/src/floating.rs`

**Current:** `apply_resize_drag_signed` clamps width/height to `FLOATING_MIN_WIDTH`/`FLOATING_MIN_HEIGHT` (lines 250-267), allowing the drag to continue at minimum size.
**Change:** Implement strict rejection:
1. After computing new `w`, `h` (before the min-w check), if `w < min_w || h < min_h`, return the **original** `LayoutRect { start_x, start_y, start_width, start_height }` unchanged â€” the drag request is fully rejected.
2. Move the min-size check before the bounds-clamping block (lines 273-285). If the size is below minimum, reject immediately without bounds clamping.
3. Update the function signature docs to describe this as "Strict Request Rejection."
