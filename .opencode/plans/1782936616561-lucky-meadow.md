# Plan: Dynamic Configurability (Item 1 from wants list)

## Goal
Unify the initialization path so standalone vs embedded is just a difference of defaults on the same structs. Introduce a `WmBuilder` that replaces divergent constructors. Embed `Theme` and `KeyBindings` directly into `WmConfig` so there's one config struct that owns everything. No serialization, no config file loading ŌĆö just the Rust structs designed cleanly so that eventual serde support and arc_swap hot-reload can be added without architectural changes.

---

## Danger Zone #1: Components Must Not Cache Config Values

**The Trap**: If components store `&Theme` references or clone colors/keybindings during init, those become stale caches. Hot-reload later will silently fail ŌĆö UI shows old colors while config says new ones.

**The Fix**: Every `render()` and `handle_event()` tick reads config values fresh from a `&WmConfig` reference on `ComponentContext`. No component struct retains any config-derived value as a field. This is contract-enforced during code review.

## Danger Zone #2: Config Must Be Immutable After Construction

**The Trap**: `WindowManager::with_config` currently duplicates many `WmConfig` fields as separate struct fields (e.g., `keyboard_focus_enabled`, `clipboard_enabled`, `decorator`, `keybindings`). If code reads the duplicate instead of `self.config.*`, swapping the config pointer later won't take effect.

**The Fix**: Only runtime-toggleable features get a separate field (initialized from config but independently mutable). Non-toggleable fields are removed ŌĆö all readers go through `self.config.*`. This makes adding `ArcSwap<WmConfig>` later a one-line change.

| Field in `WindowManager` | Currently duplicated? | Action |
|---|---|---|
| `mouse_capture_enabled` | Yes | **Keep** (runtime-toggleable), init from config |
| `keyboard_focus_enabled` | Yes | **Remove**, read from `self.config` |
| `mouse_focus_click_enabled` | Yes | **Remove**, read from `self.config` |
| `floating_resize_offscreen` | Yes | **Remove**, read from `self.config` |
| `clipboard_enabled` | Yes | **Keep** (runtime-toggleable), init from config |
| `window_selection_enabled` | Yes | **Keep** (runtime-toggleable), init from config |
| `decorator` | Yes | **Remove**, read from `self.config.decorator()` |
| `keybindings` | Cloned | **Remove**, read from `self.config.keybindings` |
| `hint_visibility` | Yes | **Remove**, read from `self.config` |

## Implementation Steps

### Step 1: Embed `Theme` into `WmConfig`
- Add `pub theme: Theme` field to `WmConfig`
- Update `standalone()` / `embedded()` presets to set `theme: NOIR`
- Remove the `OnceLock` + `current()` global from `theme.rs`
- Keep `NOIR` as a public `const` for tests / convenience
- The convenience free fns (`accent()`, `panel_bg()`, etc.) become methods on `Theme` or are accessed via `config.theme.fg(FgColor::Accent)`
- Deprecate the free fns, or just update all call sites in one pass

### Step 2: Add `&WmConfig` to `ComponentContext`
```rust
pub struct ComponentContext<'a> {
    pub focused: bool,
    pub app_context: Option<Arc<AppContext>>,
    pub config: &'a WmConfig,   // fresh every frame ŌĆö don't cache!
    // ... existing fields (viewport, hover_pos, etc.)
}
```
- `ComponentContext::new()` and builder methods require a `&WmConfig`
- Every `render()` and `handle_event()` now has access: `ctx.config.theme.bg(BgColor::PanelBg)`
- This is how all components read config ŌĆö no cached copies

### Step 3: Clean up `WindowManager` field duplication
In `with_config()`:
- Remove `keyboard_focus_enabled`, `mouse_focus_click_enabled`, `floating_resize_offscreen` fields ŌĆö readers go through `self.config.*`
- Remove `decorator` field ŌĆö callers use `self.config.decorator()`
- Remove `keybindings` field ŌĆö callers use `self.config.keybindings.*`
- Remove `hint_visibility` field ŌĆö callers use `self.config.hint_visibility`
- Keep `mouse_capture_enabled`, `clipboard_enabled`, `window_selection_enabled` as runtime fields initialized from config

Update all internal `WindowManager` methods that referenced the removed fields.

### Step 4: Create `WmBuilder` in `crates/term-wm-core/src/config.rs`
```rust
pub struct WmBuilder<Id> {
    app_ctx: Option<Arc<AppContext>>,
    config: Option<WmConfig>,
    top_panel: Option<Box<dyn FnOnce() -> Box<dyn TopPanel<WindowId<Id>>>>>,
    bottom_panel: Option<Box<dyn FnOnce() -> Box<dyn BottomPanel>>>,
    menu: Option<Box<dyn FnOnce() -> Box<dyn MenuOverlay<WmMenuAction>>>>,
}
```

Methods:
- `WmBuilder::new()` ŌĆö empty, will use defaults on build
- `.standalone()` ŌĆö set internal config to `WmConfig::standalone()`
- `.embedded()` ŌĆö set internal config to `WmConfig::embedded()`
- `.config(cfg: WmConfig)` ŌĆö set config directly (embedders)
- `.app_context(ctx)` ŌĆö set `Arc<AppContext>`
- `.top_panel(p)` / `.bottom_panel(p)` / `.menu(m)` ŌĆö inject custom components
- `.build(current: Id) -> WindowManager<Id>` ŌĆö resolve and construct

`build()` logic:
1. Start with `WmConfig::standalone()` as base
2. If `.standalone()` or `.embedded()` was called, that preset replaces base
3. If `.config()` was called, that replaces everything (embedder knows best)
4. Choose panels/menu based on config flags: if `panel_enabled && wm_overlay_enabled`, use provided factories or default standalone panels+menu; otherwise `NoopMenu` + no panels
5. Call `WindowManager::with_config(current, config, app_ctx, top_panel, bottom_panel, menu)`

### Step 5: Simplify `src/main.rs`
```rust
#[derive(Parser)]
struct Cli {
    #[arg(short = 'n', long = "count")]
    count: Option<usize>,
    #[arg(value_name = "CMD", num_args = 0..)]
    cmds: Vec<String>,
    #[arg(long = "embedded")]
    embedded: bool,
}

fn main() -> io::Result<()> {
    let cli = Cli::parse();
    let app_ctx = Arc::new(
        AppContext::new(env!("CARGO_PKG_NAME"), env!("CARGO_PKG_VERSION"))
            .with_hostname(...)
    );
    let mut builder = WmBuilder::new().app_context(Arc::clone(&app_ctx));
    if cli.embedded {
        builder = builder.embedded();
    }
    // panels and menu created once, passed to builder
    // ... rest of App::new_with: spawn terminals, open help overlay
}
```

`App::new_with` still handles terminal spawning and overlay opening ŌĆö that's application orchestration, not builder concern.

### Step 7: Update `src/lib.rs` `WindowManagerExt`
```rust
impl<Id: ...> WindowManagerExt<Id> for WindowManager<Id> {
    fn new_standalone(current: Id, app_ctx: AppContext) -> Self {
        WmBuilder::new().standalone().app_context(Arc::new(app_ctx)).build(current)
    }
    fn new_embedded(current: Id, app_ctx: AppContext) -> Self {

```

## Border Cases & Edge Handling

- **`ComponentContext` without config** ŌĆö shouldn't happen; config is mandatory. Constructor requires `&WmConfig`.
- **Existing tests construct `WmConfig::standalone()` directly** ŌĆö no change needed, presets still exist
- **`WindowManager::with_config` is a public API** ŌĆö keep it for backward compat, but mark doc comment directing users to `WmBuilder`
- **Component accidentally caches config value** ŌĆö enforce via code review; no compiler-level enforcement possible without complex lifetimes. Clear docs on `ComponentContext.config` field.
- **Runtime toggles need a new initial value from config** ŌĆö the runtime fields keep their current state; config swap only affects non-toggleable values. Toggleable values can be reset explicitly if desired.

## No-Go's (explicitly out of scope)

- No serde derives or `serde` dependency
- No TOML parsing or config file loading
- No `directories` crate or XDG path resolution
- No config file watch or hot-reload
- No `arc-swap` or lock-free atomic swapping
- No component registry / factory pattern

These are clearly separated follow-up phases that this refactor enables.
