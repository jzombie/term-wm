# Plan: Dynamic Configurability (Item 1 from wants list)

## Goal
Unify the initialization path so standalone vs embedded is just a difference of defaults on the same structs. Introduce a `WmBuilder` that replaces divergent constructors. Embed `Theme` and `KeyBindings` directly into `WmConfig` so there's one config struct that owns everything. No serialization, no config file loading — just the Rust structs designed cleanly so that eventual serde support would be trivial to add.

## Current Problems
1. `WmConfig` lives in `term-wm-core` but `Theme` is a separate global (`OnceLock` in `theme.rs`) and `KeyBindings` is constructed separately — three separate configuration concepts not owned by one struct
2. `main.rs` hardcodes `WmConfig::standalone()` — the `WindowManagerExt` trait in `src/lib.rs` provides `new_standalone`/`new_embedded` but `main.rs` doesn't use them
3. No builder pattern — host apps must construct panels, menus, config, and app_ctx separately then call `WindowManager::with_config(...)` with all pieces
4. The initialization in `main.rs` (`App::new_with`) duplicates logic that should live in the builder

## Design (from `configuration/dynamic-declarative-config-systems.md`)
- **Builder pattern**: Single `WmBuilder` is the one way to construct a `WindowManager`. `.standalone()` and `.embedded()` are just convenience methods that set groups of bool flags.
- **One config struct**: `WmConfig` owns `theme: Theme` and `keybindings: KeyBindings`. No separate globals.
- **Structs serialization-ready**: Fields are flat, logically grouped, and use standard types so that adding `#[derive(Serialize, Deserialize)]` later is a one-line change per struct. No serde added in this phase.

## Key Files
- `crates/term-wm-core/src/wm_config.rs` — embed `Theme`/`KeyBindings`; add builder-suitable API
- `crates/term-wm-core/src/theme.rs` — remove `OnceLock` global; `Theme` is just a data struct owned by `WmConfig`
- `crates/term-wm-core/src/window/window_manager/mod.rs` — `with_config` stays as-is; builder calls it
- `crates/term-wm-core/src/lib.rs` — export `WmBuilder` from new module
- `crates/term-wm-core/src/config.rs` — **new file**: `WmBuilder` struct (replaces need for a `config/` dir)
- `src/main.rs` — use `WmBuilder`, add `--embedded` CLI flag
- `src/lib.rs` — reimplement `WindowManagerExt` via `WmBuilder`, or keep as thin wrappers

## Implementation Steps

### Step 1: Embed `Theme` into `WmConfig`
- Add `pub theme: Theme` field to `WmConfig`
- Add `pub keybindings: KeyBindings` field to `WmConfig` (already partially there via `config.keybindings`)
- Update `standalone()` / `embedded()` presets to set `theme: NOIR` and appropriate keybindings
- Remove the `OnceLock` + `current()` global pattern from `theme.rs` — consumers access theme via `config.theme`
- Keep `NOIR` as a public `const` for tests / convenience
- Update all theme consumers to receive `&Theme` through component context or config reference

### Step 2: Own all keybindings through WmConfig
The `keybindings` field already exists on `WmConfig`. Ensure both presets set it fully:
- `standalone()` → `KeyBindings::standalone()`
- `embedded()` → `KeyBindings::embedded()`

### Step 3: Create `WmBuilder` in `crates/term-wm-core/src/config.rs`
```rust
pub struct WmBuilder<Id> {
    app_ctx: Option<Arc<AppContext>>,
    config: Option<WmConfig>,
    top_panel: Option<Box<dyn FnOnce() -> Box<dyn TopPanel<WindowId<Id>>>>>,
    bottom_panel: Option<Box<dyn FnOnce() -> Box<dyn BottomPanel>>>,
    menu: Option<Box<dyn FnOnce() -> Box<dyn MenuOverlay<WmMenuAction>>>>,
}
```

Methods:
- `WmBuilder::new()` — empty, will use defaults on build
- `.standalone()` — set internal config to `WmConfig::standalone()`
- `.embedded()` — set internal config to `WmConfig::embedded()`
- `.config(cfg)` — set config directly (for programmatic embedders)
- `.app_context(ctx)` — set `Arc<AppContext>`
- `.top_panel(p)` / `.bottom_panel(p)` / `.menu(m)` — inject custom components
- `.build(current: Id) -> WindowManager<Id>` — resolve and construct

`build()` logic:
1. Start with `WmConfig::standalone()` as base
2. If `.standalone()` or `.embedded()` was called, use that preset instead
3. If `.config()` was called, use that (highest priority)
4. Choose panels/menu based on config flags: if `panel_enabled && wm_overlay_enabled`, use provided factories or default standalone panels; otherwise `NoopMenu` + no panels
5. Call `WindowManager::with_config(current, config, app_ctx, top_panel, bottom_panel, menu)`

### Step 4: Update `src/lib.rs` `WindowManagerExt`
Reimplement using `WmBuilder`:
```rust
impl<Id: ...> WindowManagerExt<Id> for WindowManager<Id> {
    fn new_standalone(current: Id, app_ctx: AppContext) -> Self {
        WmBuilder::new()
            .standalone()
            .app_context(Arc::new(app_ctx))
            .build(current)
    }
    fn new_embedded(current: Id, app_ctx: AppContext) -> Self {
        WmBuilder::new()
            .embedded()
            .app_context(Arc::new(app_ctx))
            .build(current)
    }
}
```

### Step 5: Simplify `src/main.rs`
Replace the manual construction in `App::new_with` with `WmBuilder`:

```rust
struct Cli {
    #[arg(short = 'n', long = "count")]
    count: Option<usize>,
    #[arg(value_name = "CMD", num_args = 0..)]
    cmds: Vec<String>,
    #[arg(long = "embedded")]
    embedded: bool,
}

fn main() -> io::Result<()> {
    let cli = Cli::parse();
    let app_ctx = Arc::new(
        AppContext::new(env!("CARGO_PKG_NAME"), env!("CARGO_PKG_VERSION"))
            .with_hostname(...)
    );
    let mut builder = WmBuilder::new().app_context(Arc::clone(&app_ctx));
    if cli.embedded {
        builder = builder.embedded();
    }
    // ... create panels, menu, build window manager, spawn windows, run
}
```

The `App::new_with` method still handles spawning terminals and opening overlays — that's application logic, not builder logic.

### Step 6: Update all Theme consumers
- `theme.rs` convenience fns (`accent()`, `panel_bg()`, etc.) currently read from `OnceLock`. These need to either be removed or changed to methods on `&Theme`.
- Components currently call `theme::panel_bg()` etc. — they need `&Theme` passed through `ComponentContext` or via config.
- Add `pub fn theme(&self) -> &Theme` to `WmConfig`
- Thread config reference through render path where needed

## No-Go's (explicitly out of scope)
- No serde derives or `serde` dependency
- No TOML parsing or config file loading
- No `directories` crate or XDG path resolution
- No config file watch or hot-reload
- No `arc-swap` or lock-free atomic swapping
- No component registry / factory pattern

These are clearly separated follow-up phases.
