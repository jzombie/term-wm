# Plan: Profile & Fix High CPU Usage on Idle

## Root Cause Analysis

The event loop (`event_loop.rs:55-80`) unconditionally calls a full `output.draw()` on every iteration — even when there are zero events (idle):

```
loop {
    handler(None)   // ← ALWAYS triggers full redraw
    poll(interval)  // ← blocks for 16ms (active) or 200ms (idle)
}
```

This gives **5 renders/second on idle** and **~60 renders/second for 500ms after any event**, even if nothing changed.

Key files:
- `crates/term-wm-core/src/event_loop.rs` — the event loop (lines 55-80)
- `crates/term-wm-core/src/runner.rs` — the handler closure, idle path (lines 487-516) triggers full `output.draw()`
- `crates/term-wm-core/src/io/console.rs` — poll_interval: active=16ms, idle=200ms (lines 121-148)
- `crates/term-wm-ui-components/src/terminal.rs` — `render_screen` (lines 317-503), the expensive per-window render
- `crates/term-wm-pty-engine/src/pty.rs` — `read_loop` (lines 388-427, blocking I/O, not the issue)

## Investigation Steps

1. **Add tracing instrumentation** to measure render duration and frequency:
   - Add `tracing::info!(...)` spans in `event_loop.rs::run()` at the idle-render call and poll call
   - Add timing spans in `runner.rs` around the `output.draw()` call
   - Add spans in `terminal.rs::render_screen()` to measure per-terminal cost

2. **Build and run with debug logging** to confirm the render-every-cycle pattern

3. **Add a dirty-flag optimization** to skip idle renders when nothing changed:
   - Track a `dirty` flag on `TerminalComponent` (set on new PTY data or event)
   - Track a global dirty flag on `FakeApp` / `WindowManager`
   - In the idle (`None`) handler path, skip `output.draw()` if no terminal has new data and no events are pending

4. **Verify** the fix with:
   - `cargo clippy --workspace --all-targets --all-features -- -D warnings`
   - `cargo test`
   - Manual testing: run the app, observe CPU idle usage drops
