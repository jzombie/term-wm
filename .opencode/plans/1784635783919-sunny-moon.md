# Plan: Enum-Backed Components for WindowManager

## Goal

Replace `Box<dyn Component<TermWmAction>>` in `Window::component` with a concrete enum stored in a generational arena, eliminating heap allocation per-window, vtable dispatch, and the `component_downcast_mut` downcast pattern. No backward compat.

---

## Design

### Two-Level SlotMap (Window non-generic, WindowManager<C> generic)

Components live in a separate `SlotMap` inside `WindowManager<C>`. Windows hold only a `ComponentKey`. This keeps `Window` non-generic while containing the generic to `WindowManager<C>` alone.

```rust
pub struct WindowManager<C: Component<TermWmAction>> {
    windows: SlotMap<WindowKey, Window>,
    components: SlotMap<ComponentKey, C>,
    // ... existing fields
}

pub struct Window {
    pub component_key: ComponentKey,  // replaces `component: Box<dyn Component>`
    // ... other fields unchanged
}
```

### Compositional Extensibility (Fixes SEV-1 Extensibility)

Two-layer enum pattern enabling third-party injection without modifying source:

1. **Built-in enum** defined in `term-wm-ui-components` (or similar high-level crate):
```rust
// crates/term-wm-ui-components/src/core_component.rs
pub enum CoreWmComponent {
    Terminal(ScrollViewComponent<TerminalComponent>),
    DebugLog(WmDebugLogComponent),
    SystemPanel(WmSystemPanelComponent),
    SessionManager(WmSessionManagerComponent),
    Noop(term_wm_core::NoopComponent),
}

impl Component<TermWmAction> for CoreWmComponent { ... }
```

2. **Third-party/user enum** wraps the core enum:
```rust
// src/components.rs (binary) or downstream user crate
pub enum AppRootComponent {
    Core(CoreWmComponent),
    CustomAuthWidget(MyAuthWidget),  // third-party extension
}

impl Component<TermWmAction> for AppRootComponent {
    fn render(&mut self, ...) {
        match self {
            Self::Core(c) => c.render(...),       // delegates to CoreWmComponent
            Self::CustomAuthWidget(c) => c.render(...),
        }
    }
}
```

3. **Binary instantiates** via the outer enum:
```rust
type AppWM = WindowManager<AppRootComponent>;
```

### No Vtable Coercion (Fixes SEV-1 Dispatch Leak)

`component_for_key_mut` returns `Option<&mut C>` — never casts to `&dyn Component`. All callers work through `<C: Component<TermWmAction>>` bounds using static dispatch:

```rust
impl<C: Component<TermWmAction>> WindowManager<C> {
    pub fn component_for_key_mut(&mut self, key: WindowKey) -> Option<&mut C> {
        let w = self.windows.get_mut(key)?;
        self.components.get_mut(w.component_key)
    }
}
```

### Generic Propagation to Application Root (Fixes SEV-2)

`App`, `TermWmApp`, `AppBuilder::build()`, `render_app()`, and `DrawPlanRenderer::render()` all become generic over `C: Component<TermWmAction>`. Monomorphization at the binary entry point generates one copy.

# Post-Implementation Amendments

Two SEV-2 issues identified in the code review after the initial implementation was completed.

## Amendment A: Extract CoreWmComponent into a Dedicated Crate

**Problem:** `CoreWmComponent` was placed in `term-wm-sys-ui-components` because it needs types from both `term-wm-ui-components` (TerminalComponent, ScrollViewComponent) and `term-wm-sys-ui-components` (WmDebugLogComponent, WmSystemPanelComponent, WmSessionManagerComponent). This forces the system-chrome crate to serve as the global component registry, violating workspace modularity.

**Fix:** Create a new `term-wm-ui-facade` crate that sits above both UI crates and contains only `CoreWmComponent`.

### Steps

1. **Create `crates/term-wm-ui-facade/Cargo.toml`**:
   ```toml
   [package]
   name = "term-wm-ui-facade"
   version.workspace = true
   edition.workspace = true
   
   [dependencies]
   term-wm-core = { workspace = true }
   term-wm-ui-components = { workspace = true }
   term-wm-sys-ui-components = { workspace = true }
   ```
   Only `term-wm-core` (for `Component` trait, `NoopComponent`, `TermWmAction`), `term-wm-ui-components` (for `TerminalComponent`, `ScrollViewComponent`), and `term-wm-sys-ui-components` (for `WmDebugLogComponent`, `WmSystemPanelComponent`, `WmSessionManagerComponent`) are needed. No layout or render deps — the registry only delegates trait methods, never calls them directly.

2. **Create `crates/term-wm-ui-facade/src/core_component.rs`** — copy `CoreWmComponent` enum + 21-method `Component<TermWmAction>` impl from `term-wm-sys-ui-components/src/core_component.rs`

3. **Create `crates/term-wm-ui-facade/src/lib.rs`**:
   ```rust
   pub mod core_component;
   pub use core_component::CoreWmComponent;
   ```

4. **Add to workspace members** in root `Cargo.toml`:
   ```toml
   members = [
       ...
       "crates/term-wm-ui-facade",
   ]
   ```

5. **Remove from `term-wm-sys-ui-components`**: delete `core_component.rs`, remove `pub mod core_component;` from `lib.rs`

6. **Update binary `Cargo.toml`**: add `term-wm-ui-facade` dependency

7. **Update imports**:
   - `src/components.rs`: `use term_wm_ui_facade::core_component::CoreWmComponent;`
   - `src/main.rs`: `use term_wm_ui_facade::core_component::CoreWmComponent;`
   - `examples/dual_image.rs`: no change (uses `AppRootComponent`, not `CoreWmComponent` directly)

### Files to Modify

## Amendment B: Render Pipeline Validation with RenderTracker

**Problem:** All render pipeline tests use `NoopComponent` which emits zero draw primitives. `make_wm()` in `draw_plan_renderer.rs` creates `WindowManager<NoopComponent>`. There is no test verifying that the monomorphized `C::render()` output is projected to the correct screen coordinates.

**Fix:** Add a `RenderTracker` test component that records render calls and writes a visible marker, then add an assertion test.

### Steps

1. **Add `RenderTracker` to `crates/term-wm-core/src/window/test_component.rs`**:
   ```rust
   #[derive(Default, Clone)]
   pub struct RenderTracker {
       pub last_area: Option<LayoutRect>,
       pub render_count: usize,
   }
   
   impl Component<TermWmAction> for RenderTracker {
       fn render(&mut self, backend: &mut dyn RenderBackend, area: LayoutRect, _ctx: &ComponentContext, _registry: &mut HitboxRegistry) {
           self.last_area = Some(area);
           self.render_count += 1;
           // Write a visible marker at the top-left of the render area
           if let Some(ratatui) = backend.as_any_mut().downcast_mut::<RatatuiBackend>() {
               let col = area.x.max(0) as u16;
               let row = area.y.max(0) as u16;
               if col < ratatui.area.width && row < ratatui.area.height {
                   ratatui.buffer.get_mut(col, row).set_symbol("X");
               }
           }
       }
       fn init(&mut self) {}
       fn on_mount(&mut self, _key: WindowKey, _app: &AppContext) {}
       fn hitbox_id(&self) -> Option<HitboxId> { None }
       fn handle_events(&mut self, _event: &Event, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
       fn on_key(&mut self, _event: &Event, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
       fn update(&mut self, _action: TermWmAction, _ctx: &ComponentContext, _actions: &mut VecDeque<(WindowKey, TermWmAction)>) {}
       fn destroy(&mut self) {}
       fn clear_selection(&mut self) {}
       fn selection_status(&self) -> SelectionStatus { SelectionStatus::Inactive }
       fn selection_text(&self) -> Option<String> { None }
       fn desired_height(&self, _width: u16) -> u16 { 0 }
       fn take_pending_title(&mut self) -> Option<String> { None }
       fn take_teardown_parts(&mut self) -> Option<(Box<dyn Any + Send + Sync>, std::thread::JoinHandle<()>)> { None }
       fn set_selection_enabled(&mut self, _enabled: bool) {}
       fn paste(&mut self, _text: &str) -> bool { false }
       fn on_mouse_press(&mut self, _col: u16, _row: u16, _button: crate::events::MouseButton, _modifiers: crate::events::KeyModifiers, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
       fn on_mouse_release(&mut self, _col: u16, _row: u16, _button: crate::events::MouseButton, _modifiers: crate::events::KeyModifiers, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
       fn on_mouse_drag(&mut self, _col: u16, _row: u16, _button: crate::events::MouseButton, _modifiers: crate::events::KeyModifiers, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
       fn on_mouse_scroll(&mut self, _col: u16, _row: u16, _kind: crate::events::MouseEventKind, _modifiers: crate::events::KeyModifiers, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
       fn on_mouse_move(&mut self, _col: u16, _row: u16, _modifiers: crate::events::KeyModifiers, _ctx: &ComponentContext) -> EventResult<TermWmAction> { EventResult::Ignored }
   }
   ```

2. **Add `RenderTracker` variant to `TestComponent`**:
   ```rust
   pub enum TestComponent {
       Noop(NoopComponent),
       ActionRecorder(ActionRecorder),
       KeyRecorder(KeyRecorder),

   }
   ```
   Add the match arm to all 21 `Component` trait methods on `TestComponent`.

3. **Add render-verify test** in `crates/term-wm-core/src/window/window_manager/mod.rs` or a new test file:
   - Create `WindowManager<TestComponent>` with a `RenderTracker` window
   - Register a layout, call `render_app` or equivalent
   - Assert `render_count > 0` and `last_area` matches expected content bounds
   - When using `RatatuiBackend` buffer, assert `'X'` appears at the expected coordinate

## Amendment C: Remove Box<T> Blanket Impl + Fix Leaking runner.rs Tests

**Problem:** A blanket `impl<Msg, T: Component<Msg>> Component<Msg> for Box<T>` in `components/mod.rs` (added as a compatibility shim) allows `Box<NoopComponent>` to be used as a valid `C` in `WindowManager<C>`. Runner.rs test functions exploit this by calling `wm.create_window(Box::new(NoopComponent))`, which infers `C = Box<NoopComponent>` — creating `WindowManager<Box<NoopComponent>>` instead of `WindowManager<TestComponent>`. This subverts the entire enum architecture and keeps boxed components in the test pipeline.

### Steps

1. **Remove blanket impl from `crates/term-wm-core/src/components/mod.rs`**: delete lines 338–454 (the `impl<Msg, T> Component<Msg> for Box<T>` block with all 21 delegation methods).

2. **Update runner.rs tests** to use `TestComponent::Noop(NoopComponent)` instead of `Box::new(NoopComponent)`. Locations in `crates/term-wm-core/src/runner.rs`:
   - Line 671: `wm.create_window(Box::new(NoopComponent))` → `wm.create_window(TestComponent::Noop(NoopComponent))`
   - Line 919: same
   - Lines 975–976: two windows, same pattern
   - Lines 997–999: three windows, same pattern
   - Line 1018: same pattern
   - All must also use `WindowManager::<TestComponent>::with_config(...)` instead of bare `WindowManager::with_config(...)`.

3. **Verify no remaining `Box::new` in window creation**: run `grep -rn "Box::new.*Component" crates/term-wm-core/src/` — must return zero matches for `runner.rs` (the only hits should be `Box::new(Pane)` in `terminal.rs` which is unrelated — `Pane` is not a `Component`).
