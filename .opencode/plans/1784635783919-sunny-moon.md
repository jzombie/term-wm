# Plan: Enum-Backed Components for WindowManager

## Goal

Replace `Box<dyn Component<TermWmAction>>` in `Window::component` with a concrete enum `WindowRootComponent` stored in a generational arena, eliminating heap allocation per-window, vtable dispatch, and the `component_downcast_mut` downcast pattern.

No backward compat â€” all existing callers (binary, tests, examples) are updated to use the enum.

---

## Design

### Core Insight: Two-Level SlotMap

Rather than making `Window` generic (which infects every type that touches it), components live in a **separate SlotMap** inside `WindowManager<C>`. Windows hold only a `ComponentKey`. This keeps `Window` non-generic while containing the generic parameter to `WindowManager<C>` alone.

### Type Parameter (No Default)

```rust
pub struct WindowManager<C: Component<TermWmAction>> {
    windows: SlotMap<WindowKey, Window>,
    components: SlotMap<ComponentKey, C>,   // new arena
    // ... all existing fields unchanged
}

pub struct Window {
    pub component_key: ComponentKey,  // replaces `component: Box<dyn Component>`
    // ... all other fields unchanged
}
```

No default type parameter. All tests and the binary must supply a concrete `C`.

### The Enum (Binary Crate)

Defined in `src/components.rs` â€” all concrete window root types are known here:

```rust
pub enum WindowRootComponent {
    Terminal(ScrollViewComponent<TerminalComponent>),
    DebugLog(WmDebugLogComponent),
    SystemPanel(WmSystemPanelComponent),
    SessionManager(WmSessionManagerComponent),
    Noop(NoopComponent),
}

impl Component<TermWmAction> for WindowRootComponent {
    fn render(&mut self, ...) {
        match self {
            Self::Terminal(c) => c.render(...),  // static dispatch
            Self::DebugLog(c) => c.render(...),
            // ...
        }
    }
    // Same pattern for all ~20 Component trait methods
}
```

### Test Enum (Core Crate)

Defined inside `crates/term-wm-core/src/window/window_manager/mod.rs` under `#[cfg(test)]`:

```rust
#[cfg(test)]
pub(crate) enum TestComponent {
    Noop(NoopComponent),
    ActionRecorder(ActionRecorder),
    KeyRecorder(KeyRecorder),
    SelComponent(SelComponent),
}

impl Component<TermWmAction> for TestComponent { ... }
```

## Implementation Steps

### Step 1: Add `ComponentKey` type

**File:** `crates/term-wm-core/src/window/mod.rs`

```rust
slotmap::new_key_type! {
    pub struct ComponentKey;
}
```

### Step 2: Change `Window` struct

**File:** `crates/term-wm-core/src/window/entry.rs`

- Remove `pub component: Box<dyn Component<TermWmAction>>`
- Add `pub component_key: ComponentKey`
- `Window::new()` takes `ComponentKey` instead of `Box<dyn Component>`

### Step 3: Make `WindowManager<C>` generic

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs`

- Add `C: Component<TermWmAction>` generic parameter to struct
- Add field `components: SlotMap<ComponentKey, C>`
- Add field `next_component_seq: usize`
- Update constructor `with_config(...)` to initialize new fields

### Step 4: Update methods that touch `w.component`

| Method | Current | New |
|--------|---------|-----|
| `create_window` | `Window::new(order, Box::new(component))` | Allocate in `components`, pass key to `Window::new` |
| `component_for_key` | `w.component.as_ref()` | `self.components.get(w.component_key).map(|c| c as &dyn Component)` |
| `component_for_key_mut` | `w.component.as_mut()` | `self.components.get_mut(w.component_key).map(|c| c as &mut dyn Component)` |
| `set_direct_mode` | `w.component.clear_selection()` | `self.components.get_mut(w.component_key).map(|c| c.clear_selection())` |
| `close_window` (chrome.rs:98) | `w.component.destroy()` | `self.components.get_mut(w.component_key).map(|c| c.destroy())` + `self.components.remove(...)` |
| `window_pane_title` | `c.take_pending_title()` | Same pattern via arena |

### Step 5: Remove/rewrite public API methods

- `create_window` takes `C` directly (no Box wrapper)
- `spawn` takes `C` directly and calls `on_mount`
- Remove `spawn_boxed` (no longer needed)
- `set_system_window` takes `C` directly
- `register_boxed` in `term_wm_app.rs` â†’ remove or change to take `WindowRootComponent`

### Step 6: Create test enum + update tests

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs` (tests section)

Add `TestComponent` enum that wraps all test stub types, implement `Component<TermWmAction>` for it.

Update all test functions (~55+ in mod.rs, runner.rs, integration tests) to use `WindowManager::<TestComponent>::with_config(...)` and create windows with `TestComponent::Noop(NoopComponent)`, etc.

### Step 7: Create `WindowRootComponent` enum

**New file:** `src/components.rs`

- Enum with all concrete window root variants
- Full `Component<TermWmAction>` impl delegating all methods via match
- Export from `src/lib.rs`

### Step 8: Update binary creation sites

**Files:** `src/main.rs`, `src/term_wm_app.rs`, `src/lib.rs`

Replace `wm.create_window(Box::new(...))` / `wm.set_system_window(Box::new(...))` with typed enum variants:
```rust
wm.create_window(WindowRootComponent::Terminal(ScrollViewComponent::new(pane)))
wm.create_window(WindowRootComponent::DebugLog(WmDebugLogComponent::new(...)))
```

### Step 9: Update examples

**File:** `examples/dual_image.rs` â€” update to use `WindowRootComponent` enum or create its own test enum.

## Left for Future

The `WmComponent` and `Overlay<TermWmAction>` trait-object patterns (`Box<dyn WmComponent>` in `LayerManager`, `Box<dyn Overlay>` in `WindowManager`) are NOT changed in this plan. The component-key pattern can be extended to them in a follow-up.
