# Plan: Configurable Keybindings with Bottom Panel Hints and Overlay

## Goal
Allow users to pass custom keybindings to the WM, render a hint strip of the most common bindings in the bottom panel bar, and provide a full keybinding reference overlay.

## Design

### 1. Configurable KeyBindings in WmConfig

**Problem**: `KeyBindings` is currently `Default`-only. All call sites use `KeyBindings::default()`. No way to inject custom bindings.

**Solution**:
- Add `keybindings: KeyBindings` field to `WmConfig` (defaults to `KeyBindings::default()`)
- Store the configured `KeyBindings` in `WindowManager` (copied from `WmConfig` on construction)
- Provide `WindowManager::keybindings() -> &KeyBindings` getter
- Replace all `KeyBindings::default()` call sites with `wm.keybindings()`:
  - `runner.rs` lines 159, 188, 292
  - `overlays.rs` line 110, 154
  - `help_overlay.rs` lines 83, 200 (these use `KeyBindings::default()` directly ŌĆö thread via `ComponentContext` or a shared reference)

**Files to modify**:
- `crates/term-wm-core/src/wm_config.rs` ŌĆö add `keybindings: KeyBindings` field
- `crates/term-wm-core/src/window/window_manager/mod.rs` ŌĆö add `keybindings` field, populate from config, expose getter
- `crates/term-wm-core/src/runner.rs` ŌĆö replace `KeyBindings::default()` with `app.windows().keybindings()`
- `crates/term-wm-core/src/window/window_manager/overlays.rs` ŌĆö replace `KeyBindings::default()` with `self.keybindings()`
- `crates/term-wm-ui-components/src/sys/help_overlay.rs` ŌĆö accept `&KeyBindings` reference or pass via context; or have `WindowManager` pass it through to the overlay during event dispatch

### 2. Keybinding Hint Strip in the Bottom Panel Bar

**Problem**: The bottom bar currently shows only `pkg ┬Ę platform ┬Ę hostname`. We need space for keybinding hints.

**Solution**:
- Add a `keybinding_hints: Vec<(Action, Vec<String>)>` field to `Panel` ŌĆö the hints to render
- Add `set_keybinding_hints(&mut self, hints: Vec<(Action, Vec<String>)>)` method to `Panel`
- In `render_bottom()`, render hints on the left side, right-aligned info on the right (or truncate if too long)
- Use a config field `WmConfig.show_bottom_keybindings: bool` to control visibility
- **Self-describing priority**: Add `fn bottom_hint_priority(&self) -> Option<u8>` to `Action` ŌĆö returns `Some(n)` for actions that should appear in the bottom bar (higher = shown first), `None` to exclude. This makes the priority list a property of the `Action` enum itself, not a magic hardcoded list elsewhere.
- `KeyBindings::bottom_hints(max: usize)` returns top N actions by priority that have bindings, using the `Action` metadata
- Populate hints during rendering via `WindowManager::keybindings().bottom_hints(6)`

**Files to modify**:
- `crates/term-wm-core/src/actions.rs` ŌĆö add `bottom_hint_priority()` method to `Action`
- `crates/term-wm-core/src/keybindings.rs` ŌĆö add `bottom_hints()` method
- `crates/term-wm-core/src/panel.rs` ŌĆö add hints field, render in `render_bottom()`
- `crates/term-wm-core/src/wm_config.rs` ŌĆö add `show_bottom_keybindings: bool`
- `crates/term-wm-core/src/window/window_manager/mod.rs` ŌĆö compute and set hints in begin_frame or render_overlays

### 3. Keybinding Reference Overlay

**Problem**: No overlay shows all keybindings at a glance. The help overlay has some inline in markdown but it's not a structured keybinding reference.

**Solution**:
- Create `KeybindingOverlayComponent` in `crates/term-wm-ui-components/src/sys/` (alongside `help_overlay.rs`)
- Uses `DialogOverlayComponent` for the modal backdrop + dialog frame
- Uses `ScrollViewComponent<KeybindingListComponent>` (or just render a custom list with paragraph widgets)
- Render each `Action` + its key combo strings, grouped by category (Navigation, Scrolling, Windows, Dialogs)
- Register in `sys/mod.rs` and `lib.rs` exports
- Add `OpenKeybindings` Action variant and a default keybinding (e.g., `Ctrl+?` or `F1`)
- Wire through `runner.rs` event handling (similar to `OpenHelp`)
- Add `open_keybindings_overlay()` to `WindowManagerHost` trait
- The overlay accepts `&KeyBindings` at construction time and renders all entries

**Files to modify**:
- `crates/term-wm-core/src/actions.rs` ŌĆö add `OpenKeybindings` action variant
- `crates/term-wm-core/src/keybindings.rs` ŌĆö add default binding for `OpenKeybindings`
- `crates/term-wm-ui-components/src/sys/keybinding_overlay.rs` ŌĆö new file (the overlay component)
- `crates/term-wm-ui-components/src/sys/mod.rs` ŌĆö add `pub mod keybinding_overlay;`
- `crates/term-wm-ui-components/src/lib.rs` ŌĆö export `KeybindingOverlayComponent`
- `crates/term-wm-core/src/runner.rs` ŌĆö handle `OpenKeybindings` action in event loop
- `src/main.rs` ŌĆö override `open_help_overlay` or add keybinding overlay integration

## Key Design Decisions

1. **Threading keybindings**: Rather than pushing `KeyBindings` through `ComponentContext` (which would be a large refactor), we store it in `WindowManager` and pass to overlay constructors as needed. The `HelpOverlayComponent` already receives a reference in its `handle_help_event_in_area` ŌĆö we'll pass it from `WindowManager` which owns the config.

2. **Bottom hints priority**: Self-describing via `Action::bottom_hint_priority()`. Each `Action` variant declares its own priority level (`Some(u8)` for shown, `None` for hidden). `KeyBindings::bottom_hints(max)` reads this metadata to pick the top N. No separate hardcoded list ŌĆö the `Action` enum is the single source of truth. Users who add custom actions or remove defaults automatically get correct hints.

3. **Overlay vs new screen**: Keep it as a dialog overlay (like help) so it works in both standalone and embedded modes.
