# Plan: Configurable Keybindings with Bottom Panel Hints and Overlay

## Goal
Allow users to pass custom keybindings to the WM, render a hint strip of the most common bindings in the bottom panel bar, and provide a full keybinding reference overlay.

## Design

### 1. Configurable KeyBindings in WmConfig

**Problem**: `KeyBindings` is currently `Default`-only. All call sites use `KeyBindings::default()`. No way to inject custom bindings.

**Solution**:
- Add `keybindings: KeyBindings` field to `WmConfig` (defaults to `KeyBindings::default()`)
- Store the configured `KeyBindings` in `WindowManager` (copied from `WmConfig` on construction)
- Provide `WindowManager::keybindings() -> &KeyBindings` getter
- Replace all `KeyBindings::default()` call sites with `wm.keybindings()`:
  - `runner.rs` lines 159, 188, 292
  - `overlays.rs` line 110, 154
  - `help_overlay.rs` lines 83, 200 (these use `KeyBindings::default()` directly — thread via `ComponentContext` or a shared reference)

**Files to modify**:
- `crates/term-wm-core/src/wm_config.rs` — add `keybindings: KeyBindings` field
- `crates/term-wm-core/src/window/window_manager/mod.rs` — add `keybindings` field, populate from config, expose getter
- `crates/term-wm-core/src/runner.rs` — replace `KeyBindings::default()` with `app.windows().keybindings()`
- `crates/term-wm-core/src/window/window_manager/overlays.rs` — replace `KeyBindings::default()` with `self.keybindings()`
- `crates/term-wm-ui-components/src/sys/help_overlay.rs` — accept `&KeyBindings` reference or pass via context; or have `WindowManager` pass it through to the overlay during event dispatch

### 2. Keybinding Hint Strip in the Bottom Panel Bar

**Problem**: The bottom bar currently shows only `pkg · platform · hostname`. We need space for keybinding hints.

**Solution**:
- Add a `keybinding_hints: Vec<(Action, Vec<String>)>` field to `Panel` — the hints to render
- Add `set_keybinding_hints(&mut self, hints: Vec<(Action, Vec<String>)>)` method to `Panel`
- In `render_bottom()`, render hints on the left side, right-aligned info on the right (or truncate if too long)
- Use a config field `WmConfig.show_bottom_keybindings: bool` to control visibility
- **Self-describing priority**: Add `fn bottom_hint_priority(&self) -> Option<u8>` to `Action` — returns `Some(n)` for actions that should appear in the bottom bar (higher = shown first), `None` to exclude. This makes the priority list a property of the `Action` enum itself, not a magic hardcoded list elsewhere.
- `KeyBindings::bottom_hints(max: usize)` returns top N actions by priority that have bindings, using the `Action` metadata
- Populate hints during rendering via `WindowManager::keybindings().bottom_hints(6)`

**Files to modify**:
- `crates/term-wm-core/src/actions.rs` — add `bottom_hint_priority()` method to `Action`
- `crates/term-wm-core/src/keybindings.rs` — add `bottom_hints()` method
- `crates/term-wm-core/src/panel.rs` — add hints field, render in `render_bottom()`
- `crates/term-wm-core/src/wm_config.rs` — add `show_bottom_keybindings: bool`
- `crates/term-wm-core/src/window/window_manager/mod.rs` — compute and set hints in begin_frame or render_overlays

### 3. Keybinding Reference Overlay

**Problem**: No overlay shows all keybindings at a glance. The help overlay has some inline in markdown but it's not a structured keybinding reference.

**Solution**:
- Create `KeybindingOverlayComponent` in `crates/term-wm-ui-components/src/sys/` (alongside `help_overlay.rs`)
- Uses `DialogOverlayComponent` for the modal backdrop + dialog frame
- Uses `ScrollViewComponent<KeybindingListComponent>` (or just render a custom list with paragraph widgets)
- Render each `Action` + its key combo strings, grouped by category (Navigation, Scrolling, Windows, Dialogs)
- Register in `sys/mod.rs` and `lib.rs` exports
- Add `OpenKeybindings` Action variant and a default keybinding (e.g., `Ctrl+?` or `F1`)
- Wire through `runner.rs` event handling (similar to `OpenHelp`)
- Add `open_keybindings_overlay()` to `WindowManagerHost` trait
- The overlay accepts `&KeyBindings` at construction time and renders all entries

**Files to modify**:
- `crates/term-wm-core/src/actions.rs` — add `OpenKeybindings` action variant
- `crates/term-wm-core/src/keybindings.rs` — add default binding for `OpenKeybindings`
- `crates/term-wm-ui-components/src/sys/keybinding_overlay.rs` — new file (the overlay component)
- `crates/term-wm-ui-components/src/sys/mod.rs` — add `pub mod keybinding_overlay;`
- `crates/term-wm-ui-components/src/lib.rs` — export `KeybindingOverlayComponent`
- `crates/term-wm-core/src/runner.rs` — handle `OpenKeybindings` action in event loop
- `src/main.rs` — override `open_help_overlay` or add keybinding overlay integration

## Verification

1. `cargo clippy --workspace --all-targets --all-features -- -D warnings` — no new warnings
2. `cargo test` — all existing tests pass
3. Manual: run `cargo run` and verify:
   - Bottom bar shows keybinding hints (`Ctrl+Q Quit`, `Tab Focus`, etc.)
   - New keybinding overlay opens with custom binding (default: `Ctrl+?`)
   - Overlay shows all actions with their configured key combos
   - Custom keybindings in `WmConfig` are respected (test by passing modified config)

## Key Design Decisions

1. **Threading keybindings**: Rather than pushing `KeyBindings` through `ComponentContext` (which would be a large refactor), we store it in `WindowManager` and pass to overlay constructors as needed. The `HelpOverlayComponent` already receives a reference in its `handle_help_event_in_area` — we'll pass it from `WindowManager` which owns the config.

2. **Bottom hints priority**: Self-describing via `Action::bottom_hint_priority()`. Each `Action` variant declares its own priority level (`Some(u8)` for shown, `None` for hidden). `KeyBindings::bottom_hints(max)` reads this metadata to pick the top N. No separate hardcoded list — the `Action` enum is the single source of truth. Users who add custom actions or remove defaults automatically get correct hints.

3. **Overlay vs new screen**: Keep it as a dialog overlay (like help) so it works in both standalone and embedded modes.

# Architectural Blueprint for Configurable Keybindings and Discoverability in Terminal Window Managers

The Renaissance of Terminal User Interfaces and Window Management
The development of a generic window manager for Terminal User Interfaces (TUIs) requires navigating a complex and historically rich intersection of legacy terminal protocols, modern application architecture, and sophisticated human-computer interaction paradigms. Window managers operating within terminal emulators—historically dominated by mainstays such as tmux and GNU screen—are increasingly expected to provide the ergonomic fluidity, discoverability, and configuration capabilities typically associated with graphical desktop environments.1 As the computational landscape evolves, developers and power users are returning to the terminal for its unmatched speed and scriptability, demanding interfaces that do not sacrifice the usability standards established by decades of graphical interface evolution.
Historically, text-based interfaces from the late 1980s and early 1990s, such as Borland's Turbo Vision, set a remarkably high standard for keyboard discoverability by introducing persistent bottom status bars, robust menu systems, and context-aware event handling.3 These interfaces allowed users to easily discover that a specific function key provided help or that a specific combination exited the application, bridging the gap between raw command-line input and visual application state.5 Modern terminal applications are currently undergoing a renaissance, blending the efficiency of command-line tools with the aesthetic and structural principles of modern operating systems, particularly drawing upon robust frameworks like the Apple Human Interface Guidelines (HIG) to dictate logical interaction patterns.6
The central challenge in engineering a modern terminal window manager lies in constructing an input architecture that is highly configurable by the end-user while remaining strictly bounded by sensible, discoverable defaults. The proposed architectural plan addresses this by implementing a dynamically loaded configuration through a WmConfig structure, establishing a self-describing priority system for bottom panel hints via an Action::bottom_hint_priority() method, and providing a comprehensive, searchable overlay for keybinding reference through a newly minted KeybindingOverlayComponent.
This exhaustive architectural analysis explores the mechanisms for configuration injection in Rust, the nuances of terminal keyboard protocols (specifically the progressive CSI u protocol), historical interface precedents, and the rigorous application of standardized Human Interface Guidelines principles. By systematically dissecting these domains, this report ensures that the resulting window manager is both profoundly capable and intuitively discoverable.
Historical Paradigms in Terminal Discoverability
To engineer a modern hint strip and keybinding overlay, it is essential to understand the historical precedents that established user expectations for terminal UI discoverability. The proposed architecture draws explicit inspiration from 1990s graphical user interfaces and terminal multiplexers, necessitating a deep analysis of how these early systems managed user input and interface self-documentation.
The concept of a persistent bottom bar displaying context-aware keybindings traces its lineage directly to Borland's Turbo Vision framework, a foundational C++ UI library from the DOS era.3 In Turbo Vision, the interface was defined by an object-oriented architecture where a TStatusLine object was instantiated with a TStatusDef configuration, which in turn contained multiple TStatusItem instances.8 Each TStatusItem directly mapped a textual description—such as the string "~Alt-X~ Exit"—to a physical keyboard scan code like kbAltX and a semantic command event like cmQuit.5 When the user pressed a key, the Turbo Vision event loop evaluated the input against these defined objects, broadcasting commands via an evBroadcast mechanism to the appropriate interface components.11 This design tightly coupled the execution of a command with its visual documentation on the screen, ensuring that users were always aware of their immediate options.
Similarly, the ubiquitous GNU Nano text editor utilizes a static, hardcoded two-row bottom grid to display the most critical commands, such as ^G Get Help or ^O Write Out.12 While Nano's approach is highly visible and guarantees that novice users can immediately operate the editor, its hardcoded nature presents a severe architectural limitation.14 The shortcuts are embedded directly within the application's global C source files, preventing users from altering the layout or adapting the keybindings to alternative keyboard layouts without recompiling the application.14
In the modern context, tools like the Zellij terminal multiplexer utilize an advanced plugin architecture to render context-aware bindings at the bottom of the screen.15 Plugins such as zjstatus-hints dynamically update the bottom bar based on whether the user is in "Normal," "Pane," "Search," or "Resize" mode.15 The proposed design for the generic terminal window manager effectively synthesizes the persistent visibility of Nano with the dynamic, state-aware object modeling of Turbo Vision and the mode-specific contextual awareness of Zellij. By designing a system where the configuration defines the visual hints, the window manager prevents the documentation drift that occurs when hardcoded interfaces fall out of sync with user-customized configurations.
State Injection and Configuration Architecture in Rust
The transition from a hardcoded KeyBindings::default() initialization to a dynamically injected configuration structure is the foundational step in enabling user customization. In modern TUI application development, particularly within the Rust programming language ecosystem, decoupling physical input events from their intended semantic actions is a critical design pattern that ensures long-term maintainability.17
The proposed architectural plan necessitates adding a keybindings: KeyBindings field to the WmConfig structure. This pattern is prevalent in modern Rust terminal applications, such as the Helix text editor and the Zellij multiplexer, which rely heavily on robust configuration objects serialized from disk and passed down through the application state.19 By parsing a user-defined configuration file—typically using formats like TOML or JSON5 via the serde crate—the application can construct a comprehensive map of desired inputs to application behaviors.21
Centralizing the Source of Truth
By storing the configured KeyBindings within the core WindowManager struct and exposing a dedicated getter method (WindowManager::keybindings() -> &KeyBindings), the architecture successfully centralizes the source of truth for input resolution. The execution of the proposed plan requires replacing all legacy calls to KeyBindings::default() across disparate modules. Specifically, modifying files such as crates/term-wm-core/src/runner.rs, crates/term-wm-core/src/window/window_manager/overlays.rs, and the help overlay components ensures that any user overrides provided at startup are universally respected throughout the application's lifecycle.
This centralization eliminates the risk of disjointed behavior, where a child component might instantiate its own default keybindings while the parent window manager utilizes a customized set. In Rust, rather than pushing the KeyBindings object through a complex ComponentContext—which would trigger a massive and potentially destabilizing refactor of the application's Z-axis state management—the decision to retain ownership within the WindowManager and pass a shared reference (&KeyBindings) to overlay constructors is architecturally sound.17 This adheres to Rust's strict borrowing rules while preventing unnecessary memory allocations or cloning of the configuration map.
Managing Defaults, Overrides, and Erasure
A robust keybinding configuration system must define exactly how user-supplied inputs interact with the application's native defaults. The Zellij terminal multiplexer provides an excellent blueprint for managing these complex interactions, establishing three distinct levels of customization that the configuration parser must gracefully handle.23
Firstly, the system must support individual key overrides. When a custom configuration specifies a binding, it must implicitly overwrite any default action previously bound to that specific key combination.23 For example, if the default configuration maps Ctrl-N to NewPane, but the user configuration maps Ctrl-N to NextWindow, the configuration parser must resolve this collision in favor of the user, discarding the default mapping.
Secondly, the system must support explicit unbinding. Users often require the ability to strip a default binding without immediately assigning a new one, particularly to free up standard keystrokes for sub-applications running within the terminal.23 The parser must recognize a directive analogous to an unbind instruction, removing the target key from the application's resolution map entirely.23
Finally, advanced users require a mechanism for global or modal erasure. Implementing a directive such as clear-defaults=true allows the configuration parser to discard all predefined bindings before processing the user's custom definitions.19 This "blank slate" approach is vital for users employing non-standard keyboard layouts (such as Dvorak or Colemak) or those who wish to construct a binding schema completely divorced from the application's original vision.23 By supporting these three tiers of interaction, the WmConfig injection pipeline ensures maximum flexibility.
The Intermediate Abstraction Layer
Within the Rust TUI ecosystem, translating raw terminal input into application-specific actions relies on an intermediate abstraction layer.17 The most scalable approach involves defining an Action enumeration that represents every possible mutation of the window manager's state.
The KeyBindings struct typically wraps a hierarchical map, such as a HashMap<Vec<KeyEvent>, Action>, which associates a sequence of physical keystrokes with a target semantic action.21 When a centralized event loop captures a crossterm::event::KeyEvent, it queries this abstraction map. If a match is found, the corresponding Action variant is dispatched to the application's state mutation logic.17 This abstraction ensures that the business logic of the window manager remains entirely agnostic to the physical keys pressed. Whether a pane is closed via Ctrl-W, Alt-C, or a complex leader-key sequence, the core logic only ever receives the Action::ClosePane instruction. This separation of input from intent is the cornerstone of a stable, configurable application.
The Legacy of Terminal Input and Progressive Protocols
To support modern, highly configurable keybindings—especially those inspired by graphical operating systems and modern Human Interface Guidelines—the window manager architecture must address the severe and historically entrenched limitations of traditional terminal emulators.
The ASCII Teletype Bottleneck
For nearly five decades, terminal applications have struggled to differentiate complex keyboard combinations due to encoding behaviors dating back to the 1960s and 1970s.26 When a user depresses a key in a standard terminal emulator, the physical interaction is translated into a byte sequence and transmitted to the application over a pseudo-terminal (pty) interface.26 Regular alphanumeric characters map predictably to standard ASCII codes.26
However, the handling of modifier keys is rooted in legacy hardware constraints. The Control key acts by physically clearing bits 5 and 6 of a character's binary code, a trick dating back to the circuitry of the 1963 ASR-33 teletype machine.26 This mechanism introduces severe collisions, rendering physically distinct keystrokes entirely indistinguishable to the receiving application 26:

Modernizing Input: The CSI u (Kitty) Keyboard Protocol
To achieve the nuanced input control required by a modern, configurable window manager, the application architecture must support progressive terminal keyboard protocols, specifically the Kitty Keyboard Protocol, commonly referred to as the CSI u protocol.26 Introduced and refined to escape standard VT100 limitations, this protocol resolves legacy collisions by instructing the host terminal emulator to transmit structured, mathematically unambiguous escape sequences to the application.26
The standard sequence format dictated by the protocol is CSI keycode ; modifiers u, where CSI represents the standard escape introducer (ESC [).26 The keycode parameter represents the precise Unicode code point of the physical key pressed (for example, the Enter key is 13, Tab is 9, and the lowercase 'a' is 97).26
The brilliance of the protocol lies in the modifiers parameter, which is a calculated bitmask value plus one.26 The bitmask assigns specific values to modifier states: Shift equals 1, Alt equals 2, Control equals 4, and Super/Command equals 8.26 When multiple modifiers are held, their bit values are summed. A critical architectural consideration when implementing these progressive enhancements is crash safety. The protocol utilizes a push/pop stack mechanism to manage terminal state.26 If the window manager encounters an unhandled panic or the host SSH connection drops unexpectedly, the terminal emulator will remain in the enhanced state, expecting CSI u sequences but feeding them into a standard shell like Bash or Zsh, which do not understand the protocol.26 This results in broken keybindings for the user until the terminal is manually reset.33 The application architecture must absolutely guarantee the execution of PopKeyboardEnhancementFlags during the shutdown sequence.31 This is typically achieved by implementing custom panic hooks or by binding the terminal restoration logic to robust Drop traits on the main application wrapper, ensuring cleanup occurs even during unwinding.
Applying Apple Human Interface Guidelines to Terminal Workflows
The architectural plan explicitly states a desire for the window manager to follow system guidelines akin to modern macOS environments. Apple's Human Interface Guidelines (HIG) provide a rigorous, meticulously researched, and universally understood framework for keyboard interactions.28 Mapping these graphical desktop paradigms onto a terminal window manager establishes an immediate sense of familiarity and reduces the cognitive load for professional users.
Principles of Clarity, Deference, and Depth
The core principles of Apple's HIG include Clarity, Deference, and Depth.34 In the context of a keyboard-driven application, Clarity demands that shortcuts are logical, memorable, and rely on standard modifiers.34 Deference dictates that application-specific shortcuts should not conflict with ingrained system-wide or platform-standard shortcuts, allowing the user's primary focus to remain on the content rather than fighting the interface.34 Depth implies a logical hierarchy; standard actions require fewer modifiers, while destructive or complex actions require deeper, more deliberate keystroke combinations.34
The HIG explicitly advises against defining new keyboard shortcuts for actions that are seldom used, warning that such sprawl leads to memory fatigue and potential conflicts.35 Translating this to the TUI window manager, the KeyBindings configuration should only establish default mappings for frequent, highly consequential actions, such as creating new workspace panes, shifting focus between them, or invoking the command palette. Edge-case functionality should remain accessible via the command overlay but unmapped by default.

Translating HIG macOS paradigms into the terminal space requires mapping specific modifiers to distinct functional domains. According to Apple's guidelines, different combinations of the Command, Control, Option (Alt), and Shift keys denote different scopes of interaction.28

Collision Avoidance and Deference in Defaults
A frequent and severe pitfall in terminal window manager design is the hijacking of heavily utilized shortcuts native to the applications running inside the terminal. For example, GNU screen famously utilized Ctrl-A as its default command prefix, directly colliding with the universally established GNU Readline and Bash shortcut for "move cursor to the beginning of the line".37 Similarly, tmux defaults to Ctrl-B, which collides with the standard Emacs and Readline shortcut for "move cursor backward".13 When a window manager intercepts these keys, it breaks the user's workflow and violates the HIG principle of deference.
To adhere to HIG principles, the window manager's default bindings must minimize interference with underlying terminal processes (such as shell prompts, vim, nano, or SSH sessions). This is effectively achieved through two architectural paradigms:
Prefix Modes (The Tmux Approach): Utilizing a rare trigger key combination to enter a temporary "Command Mode," after which subsequent keystrokes act globally on the window manager rather than passing to the shell. This requires users to prefix every window management action, slowing down operation but guaranteeing zero collisions.
Unlock-First Presets (The Zellij Approach): Operating primarily in a "locked" state where all input passes directly through to the underlying application.19 The user must intentionally trigger an "unlock" binding (such as Ctrl-G) to interact with the window manager's native modes, such as "Pane Mode" or "Resize Mode".39
By adopting an unlock-first or strictly modal architecture for its default configuration, the window manager ensures that its bindings will never inadvertently trigger destructive actions in the user's sub-processes, preserving the integrity of the terminal experience while offering total control when explicitly requested.
Ergonomic Discoverability: Architecting the Bottom Panel Hint Strip
A highly configurable interface is only effective if its current configuration is readily discoverable by the user. The proposed inclusion of a keybinding hint strip in the bottom panel bar is a historically proven and highly effective HCI pattern for text interfaces, seamlessly orienting users within complex state machines.
The Problem of Static Help
As previously noted, the bottom bar currently only displays static system information: pkg · platform · hostname. This represents underutilized screen real estate. In applications without constant visual feedback, users must memorize extensive keybinding maps, creating a steep learning curve. While tools like nano solve this with a static help bar, a dynamic window manager requires a dynamic solution. If a user rebinds the "Quit" action from Ctrl-Q to Alt-X, a statically rendered help bar will display incorrect information, leading to severe user frustration and application distrust.
The Action::bottom_hint_priority() Metadata Pattern
The architectural proposal to implement a fn bottom_hint_priority(&self) -> Option<u8> method directly on the Action enumeration is a profoundly elegant solution to the problem of dynamic hint generation. It establishes a self-describing metadata framework that solves the logistical challenge of selecting which hints to display in a constrained horizontal space.
High Architectural Cohesion: By decentralizing the priority logic into the Action enum itself, the metadata remains intrinsically tied to the exact action it describes. If a developer adds a new Action::ToggleFullscreen variant in the future, they define its display priority in the exact same location, rather than updating a disconnected rendering array elsewhere in the codebase.
Loose Coupling: The rendering pipeline—encapsulated within the Panel struct—requires no hardcoded knowledge of specific actions or application logic.17 It merely requests the top prioritized actions from the active configuration.
Dynamic Adaptability: The KeyBindings::bottom_hints(max: usize) function operates by iterating over the configured keybinding map. It filters the list to include only actions that actually have a physical key sequence mapped to them, reads the Option<u8> metadata to sort them by priority (where higher u8 values indicate higher priority), and truncates the resulting vector to fit the requested max limit.
When this logic executes, the application automatically queries the active configuration, identifies the physical keystroke, retrieves its formatted string representation, and pushes a tuple (Action, Vec<String>) to the Panel's keybinding_hints field. This guarantees that the bottom bar is never out of sync with the user's configuration file, offering true, dynamic discoverability.
Rendering Constraints and Overflow Management
Terminal window managers operate within strictly enforced, dynamic grid layouts. Unlike web browsers, text cannot simply flow off the screen; it must be calculated and truncated at the character cell level. The Panel's render_bottom() method must be exceptionally resilient to arbitrary terminal resizing events.
When formatting the hints for display, the rendering engine must calculate the total string width of the concatenated text, accounting for spacing and separator characters (such as the · symbol). If the calculated width exceeds the available terminal columns—after reserving space for the right-aligned system information—the renderer must gracefully degrade the display.
This degradation sequence typically involves:
Iteratively truncating the lowest-priority hints from the left-aligned strip until the text fits within the available width.
Replacing truncated elements with a standardized ellipsis character (…) to indicate to the user that additional options exist but are currently hidden due to space constraints.
If the terminal is resized to an exceptionally narrow width (e.g., below 40 columns), the logic should hide the hint strip entirely to prevent visual overlap with critical system data, relying on the user to invoke the full Keybinding Reference Overlay.
Furthermore, the provision of a WmConfig.show_bottom_keybindings: bool configuration toggle is essential. As experienced users develop muscle memory, the persistent hint strip may become visual noise. Allowing power users to reclaim that vertical screen real estate for their terminal applications is a standard practice, mimicking the toggle capabilities found in tmux status bar configurations.2
Comprehensive Command Discovery: The Reference Overlay
While the bottom panel provides immediate context for the most critical actions, a complex window manager requires a dedicated, full-screen interface for exhaustive command discovery. The proposal to implement a KeybindingOverlayComponent alongside existing help structures is structurally sound and aligns with the expectations established by modern developer tooling.
The Modal Dialog Paradigm
Integrating the keybinding reference as a modal dialog overlay, rather than rendering it as an entirely separate screen or overtaking an active terminal workspace pane, is vital for maintaining user context. The user is not jarringly removed from their active workflow; instead, the DialogOverlayComponent acts as a Z-index layer. It visually dims the background terminal components while drawing focus to an interactive ScrollViewComponent<KeybindingListComponent>.
This modal paradigm ensures that the overlay functions correctly regardless of the window manager's current state. Whether the application is running in a standalone full-screen mode or operating as a deeply embedded component within a larger software suite, the modal overlay securely traps keyboard focus, displays the information, and then relinquishes control back to the exact state the user left.
Dynamic Generation and the Which-Key Influence
The dynamic generation of this overlay must mirror the sophisticated capabilities found in the ratatui-which-key crate, a popular Rust UI library inspired by the Neovim which-key plugin.18 Historically, TUI applications relied on hardcoded markdown files for help screens. To eliminate this flaw, the KeybindingOverlayComponent must actively iterate over the live &KeyBindings reference injected during its instantiation. By processing the active hash map, the overlay guarantees 100% fidelity between what is displayed to the user and what the event loop will actually accept as input.

Implementation Strategy and Sub-System Wiring
Executing this comprehensive architectural plan requires rigorous attention to the ownership semantics, lifetime constraints, and event routing architecture within the Rust codebase.
Injecting Configuration Through the Component Hierarchy
The provided plan accurately identifies a common architectural pitfall: attempting to push the KeyBindings struct through a massive, globally available ComponentContext. Doing so would trigger a sprawling refactor and risk cluttering the application's state management with unnecessary global references.
Instead, the decision to maintain ownership of the KeyBindings centrally within the WindowManager is the optimal path.17 During the instantiation of child components, specifically the KeybindingOverlayComponent and the legacy HelpOverlayComponent, the WindowManager borrows its internal KeyBindings structure and passes a shared, read-only reference (&KeyBindings) to the overlay's constructor. This pattern guarantees that the overlay always possesses the latest view of the configured bindings, avoids the performance overhead of cloning large data structures, and circumvents the need for complex Rc<RefCell<T>> wrapping.
Event Bubbling and Interception Architecture
The integration of custom keybindings must be woven seamlessly into the application's event propagation model. When a user strikes a key, the runner.rs event loop captures the KeyEvent. This event must then flow through a strictly enforced priority chain to ensure proper state mutation and prevent unwanted command execution in child processes.
Overlay Interception (Highest Z-Index): Active modal overlays, such as the KeybindingOverlayComponent or DialogOverlayComponent, receive the event first. If an overlay is active, standard navigation keys (arrows, scrolling) and dismissal keys (specifically Escape or q) must be intercepted and consumed to interact with or close the modal.28 If the event is consumed here, propagation halts entirely.
Window Manager Interception (Middle Z-Index): If no overlay consumes the event, it drops down to the WindowManager. The manager matches the raw input against the active KeyBindings map.21 If a match is found (for example, Ctrl-N mapped to Action::NewPane), the window manager consumes the event, executes the associated semantic action, mutates the workspace state, and halts propagation.
Application Pass-Through (Lowest Z-Index): If the event flows through both the overlays and the window manager without finding a match, the system assumes the input is intended for the active pseudo-terminal child process. The event is converted back into its raw byte sequence and passed down the PTY pipeline into the user's shell or text editor.
This hierarchical bubbling architecture ensures that the window manager remains highly responsive to administrative commands while accurately and reliably passing unaltered input to the terminal sessions running within it.
State Formatting and Presentation Logic
Finally, the visual presentation of the keybindings requires a standardized formatting utility to translate raw Rust KeyEvent structures into human-readable strings. Because the application's TOML or JSON5 configuration allows users to map inputs using string representations (e.g., "Ctrl+Shift+Tab" or "A-minus" 41), the system must be capable of translating a parsed binary event back into an aesthetically pleasing format for display in both the hint strip and the overlay.41
A robust utility function, such as format_key_event(event: &KeyEvent) -> String, must be implemented. This function should apply HIG-compliant naming conventions uniformly:
Utilize uppercase letters for base character representations (e.g., C, not c).
Spell out modifiers clearly, or use standard graphical symbols if the terminal environment permits Unicode rendering (e.g., Ctrl or ⌃, Alt or ⌥, Shift or ⇧).
Utilize distinct, standardized names for special keys (e.g., Space, Enter, Esc, PageUp, Up, Down) rather than their raw character representations.20
By ensuring consistency in formatting, the application reinforces the clarity and predictability required by modern interface guidelines, completing the bridge between robust backend configuration and polished frontend presentation.
Synthesis and Strategic Outlook
The proposed architectural plan for implementing configurable keybindings, a contextual bottom panel hint strip, and a comprehensive reference overlay represents a highly sophisticated and complete approach to terminal window manager design. It expertly navigates the historical, byte-level limitations of command-line environments while adopting the modern interaction paradigms established by Apple's Human Interface Guidelines, advanced Rust UI frameworks, and contemporary multiplexers.
By moving away from hardcoded defaults and defining a strict Action abstraction layer, the application ensures maximum user adaptability. The strategic decision to embed prioritization metadata directly within the Action enum via the bottom_hint_priority() method demonstrates a profound understanding of cohesive software design. This choice allows the user interface to react dynamically and accurately to arbitrary configuration changes without requiring fragile manual interventions in the rendering logic. Furthermore, the integration of a dynamically populated KeybindingOverlayComponent ensures that the application's documentation remains inherently synchronized with its operational state, eliminating a major source of user friction.
To guarantee the long-term success of this implementation, the following architectural imperatives must be strictly observed throughout the development lifecycle:
CSI u Protocol Compliance: The application must successfully integrate KeyboardEnhancementFlags::DISAMBIGUATE_ESCAPE_CODES via the crossterm library to bypass legacy teletype constraints. Without this, any attempt to implement complex, HIG-compliant modifier combinations (such as Control-Shift-Tab) will fail entirely due to inherent terminal byte collisions.
Crash-Resilient State Management: Explicit panic handlers and robust memory drop routines must be established to ensure PopKeyboardEnhancementFlags is executed upon unexpected termination, preventing the severe corruption of the user's global terminal input state following an application crash.
Strict Event Propagation Chains: The architecture must guarantee that active modal overlays intercept and consume keystrokes before they are evaluated by the global KeyBindings configuration, which in turn must evaluate them before passing unhandled events to the underlying terminal sub-processes.
Deference in Default Architectures: Default keybindings shipped with the application must studiously avoid universally recognized application inputs. Protecting window management actions behind a modal locking mechanism or explicit trigger prefix is necessary to prevent destructive collisions with the user's core workflow.
By executing this comprehensive design with rigorous adherence to these technical and philosophical principles, the resulting window manager will provide an exceptionally robust, deeply discoverable, and highly ergonomic user experience that successfully bridges the gap between the speed of the terminal and the usability of graphical desktop environments.
