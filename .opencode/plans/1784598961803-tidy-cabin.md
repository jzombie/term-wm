**Replace lines 196-221** (empty state block) with:

```rust
    // Render empty state if no mapped windows — clickable link opens command palette
    const EMPTY_MSG: &str = "No opened windows.";
    if wm.mapped_windows().is_empty() {
        use ratatui::style::{Color, Modifier, Style};
        use ratatui::widgets::Paragraph;
        if let Some(rb) = backend.as_any_mut().downcast_mut::<RatatuiBackend>() {
            let buf = &mut rb.buffer;
            let msg_width = EMPTY_MSG.len() as u16;
            let x = area.width.saturating_sub(msg_width) / 2;
            let y = area.height as i32 / 2;
            let text_area = ratatui::layout::Rect {
                x,
                y: y.max(0) as u16,
                width: msg_width,
                height: 1,
            };
            Paragraph::new(EMPTY_MSG)
                .style(
                    Style::default()
                        .fg(Color::Cyan)
                        .add_modifier(Modifier::BOLD),
                )
                .render(text_area, buf);

            // Register hitbox so click opens command palette
            let hitbox_id = term_wm_core::hitbox_registry::HitboxId::new();
            wm.hitbox_registry_mut().register(
                hitbox_id,
                term_wm_core::hitbox_registry::ComponentOwner::Chrome(
                    term_wm_core::chrome::ChromeTarget::EmptyStatePlaceholder,
                ),
                term_wm_layout_engine::LayoutRect {
                    x: i32::from(x),
                    y: y.max(0) as i32,
                    width: msg_width,
                    height: 1,
                },
            );
        }
    }
```

Key design decisions:
- Use `ComponentOwner::Chrome(ChromeTarget::EmptyStatePlaceholder)` for the hitbox, matching the existing chrome interception pattern for Press events
- Color changes from DIM white to BOLD Cyan to indicate it's interactive
- A fresh `HitboxId::new()` is created each frame (same pattern as content_hitbox_id at line 145)
- Hitbox is registered directly via `wm.hitbox_registry_mut().register()` — no sub-registry merge needed since this is a simple single-entry registration

## Change 2: Hide FAB when command palette is open in monocle mode

### Files to modify

1. **`src/lib.rs`** — add condition check to FAB rendering block

### Implementation

**File:** `src/lib.rs` (lines 226-240)

Add `!wm.command_menu_visible()` to the existing condition:

```rust
    // Render FAB only in monocle/mobile mode — sole mobile navigation mechanism.
    // Hidden when command palette is open; the bottom panel overlay fills the row.
    let fab_layer_id = wm
        .semantic_registry
        .get(&term_wm_core::window::ComponentTag::FloatingActionButton)
        .copied();
    if wm.is_monocle()
        && !wm.command_menu_visible()         // <-- added
        && let Some(fab) =
            wm.get_semantic_component_mut(term_wm_core::window::ComponentTag::FloatingActionButton)
        && let Some(layer_id) = fab_layer_id
    {
        let mut local_hb = HitboxRegistry::with_owner(ComponentOwner::Layer(layer_id));
        let ctx = term_wm_core::components::ComponentContext::new(true).with_screen_area(area);
        fab.render(backend, area, &ctx, &mut local_hb);
        wm.hitbox_registry_mut().merge(local_hb);
    }
```

This is sufficient because:
- The bottom panel overlay in `render_overlays` (draw_plan_renderer.rs:766-777) already renders at full width (x:0, width: full_area.width) when both `is_monocle()` and `command_menu_visible()` are true
- The FAB occupies the bottom-right 3×1 area, which overlaps with the bottom panel overlay
- No additional hitbox suppression needed: when `fab.render()` is not called, its hitbox isn't registered
- `wm.command_menu_visible()` delegates to `wm.command_palette_visible()` (via command_menu.rs:4-5)
