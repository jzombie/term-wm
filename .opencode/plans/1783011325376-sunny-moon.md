# Threading Model — Revised Implementation Plan

## Current Architecture (with flaws)

| Component | Thread | Role | Flaw |
|-----------|--------|------|------|
| `EventLoop::run()` | Main UI | Polls driver, dispatches events, renders idle ticks | ANSI parsing blocks render |
| `UnifiedEventSource::new()` | 1 thread | Polls crossterm every 100ms, sends via bounded channel | OK |
| `read_loop()` (per PTY) | N threads | Blocks on PTY read, writes raw bytes to `pending` | Raw bytes only; no parsing |
| `Pty::update()` (called in render) | Main UI | Lock `pending`, feed `vt100::Parser`, render cells | **Chokes under high PTY output** |
| `Reaper::tick()` | **Never called** | Inline zombie management | **Bug: zombies leak until exit** |
| PowerSaver | 500ms poll | ~2fps polling loop | **Wastes CPU; should block**

---

## Phase 1 — PowerProfile: 3-State Model

**Files:** `power_profile.rs`, `event_source.rs`, `unified_event_source.rs`, `console.rs`, `runner.rs`

**Enum:**
```rust
pub enum PowerProfile {
    Interactive,   // 8ms poll — user typing (<100ms since last input)
    Streaming,     // 16ms poll — PTY data flowing, no user input
    PowerSaver,    // blocks on channel — no recent activity, no dirty windows
}
```

**`profile_from_activity(last_event_at, has_dirty_windows)` → `PowerProfile`:**
- `<100ms` since last input → `Interactive`
- `<500ms` → `Streaming`
- `has_dirty_windows` → `Streaming`
- otherwise → `PowerSaver`

**`PowerSaver::poll_interval()` returns large timeout** (e.g. 1 hour via `Duration::from_secs(3600)`). The event loop blocks on `recv_timeout()` for the full interval in PowerSaver, yielding to the OS. Only PtyWakeup or input events wake it.

**In `event_loop.rs`:** Keep idle handler call always. Power savings come from blocking poll, not skipping renders.

**In `runner.rs`:** `profile_tracker` needs to consider dirty windows. The `run_app()` closure should provide `!app.windows().take_dirty_windows().is_empty()` to the profile heuristic.

Add `WindowManager::has_dirty_windows()` / `take_dirty_windows()` (or the tracker reads from `UnifiedEventSource` directly).

## Phase 2 — Output Coalescing (3ms)

**File:** `unified_event_source.rs`

Add `coalesce_deadline: Option<Instant>` field. Constant `COALESCE_DELAY = 3ms`.

**Modified `poll()` logic on `PtyWakeup(key)`:**
```
dirty_windows.insert(key)
if coalesce_deadline is None:
    coalesce_deadline = now + COALESCE_DELAY
check coalesce:
    if now >= deadline → clear, return Ok(false)  // triggers idle render
    else → remaining = min(remaining, deadline - now); continue
```

On `Input(event)`: clear coalesce, return `Ok(true)` immediately.
At timeout: clear coalesce, return `Ok(false)`.

Test: update test construction to include `coalesce_deadline: None`.

## Phase 3 — Mechanical Backpressure (documentation)

**Files:** `unified_event_source.rs`, `pty.rs`

Add doc comments: bounded channel (256 slots) blocks `send()` when full → PTY reader pauses → OS pipe buffer fills → child process `write()` blocks. This prevents memory exhaustion under extreme load (`cat /dev/urandom`).

## Phase 4 — Reaper on Background Thread

**File:** `reaper.rs`, `window_manager/mod.rs`, `main.rs`

**Problem:** Current `Reaper` stores `Vec<ZombieChild>` inline. `tick()` is never called. Even if it were, `handle.join()` blocks the main thread.

**Solution:** Spawn a dedicated reaper thread.

```rust
// reaper.rs
pub struct Reaper {
    tx: crossbeam_channel::Sender<ZombieChild>,
    _handle: std::thread::JoinHandle<()>,
    shutdown: Arc<AtomicBool>,
}

impl Reaper {
    pub fn new(shutdown_timeout: Duration) -> Self {
        let (tx, rx) = crossbeam_channel::bounded::<ZombieChild>(64);
        let shutdown = Arc::new(AtomicBool::new(false));
        let s = Arc::clone(&shutdown);
        let _handle = std::thread::Builder::new()
            .name("pty-reaper".into())
            .spawn(move || {
                let mut zombies: Vec<ZombieChild> = Vec::new();
                loop {
                    if s.load(Ordering::Acquire) { break; }
                    // Drain incoming zombies (non-blocking)
                    while let Ok(z) = rx.try_recv() {
                        zombies.push(z);
                    }
                    // Tick — same logic as current Reaper::tick() but on this thread
                    let timeout = shutdown_timeout;
                    zombies.retain_mut(|z| { /* same tick logic */ });
                    std::thread::sleep(Duration::from_millis(50));
                }
                // drain_all logic on shutdown
                for mut z in zombies.drain(..) { /* force kill + join */ }
            })
            .expect("reaper thread");
        Self { tx, _handle, shutdown }
    }

    pub fn reap(&self, zombie: ZombieChild) {
        let _ = self.tx.send(zombie);
    }
}

impl Drop for Reaper {
    fn drop(&mut self) {
        self.shutdown.store(true, Ordering::Release);
    }
}
```

**WindowManager** stores `Reaper` as before. `reaper()` accessor returns `&Reaper` (not `&mut` — channel send only needs `&`).

**`main.rs::wm_close_window()`** — uses `app.windows().reaper().reap(...)` with `&self` reaper. No `reaper()` mutability issue.

Existing `Reaper` tests are rewritten to work with the new thread-based API. `ZombieChild` moves across threads (`Send` already).

## Phase 5 — Move ANSI Parsing into PTY Reader Thread (MANDATORY)

**Files:** `pty.rs`, `pane.rs`, `terminal.rs` + `TestPane`

### Rationale

Today, `Pty::update()` (called from the main thread during every render) locks `pending`, drains raw bytes, and feeds `vt100::Parser`. When PTY output is heavy (e.g. `cat /dev/urandom`, `yes`, large git log), the main thread spends its frame budget on ANSI parsing instead of rendering. Moving parsing to the reader thread eliminates this bottleneck.

### Design

**Shared state between reader thread and main thread:**
```rust
pub struct Pty {
    // Reader-thread owned (moved into read_loop):
    //   - vt100::Parser (exclusive access in reader thread)

    // Shared state (Arc):
    shared_screen: Arc<Mutex<vt100::Screen>>,  // reader writes, main reads
    dirty: Arc<AtomicBool>,                     // set by reader after each batch
    pending_resize: Arc<Mutex<Option<PtySize>>>, // main → reader resize requests

    // Main-thread cached copy (avoids lock contention on every cell access):
    cached_screen: vt100::Screen,

    // Removed:
    //   pending: Arc<Mutex<Vec<u8>>>,
    //   parser: vt100::Parser,
    //   history: Vec<u8>,
}
```

**Reader thread (`parser_read_loop`):**
```rust
fn parser_read_loop(
    mut reader: Box<dyn Read + Send>,
    shared_screen: Arc<Mutex<vt100::Screen>>,
    dirty: Arc<AtomicBool>,
    dsr_requested: Arc<AtomicBool>,
    pending_title: Arc<Mutex<Option<String>>>,
    pending_resize: Arc<Mutex<Option<PtySize>>>,
    wakeup: WakeupFn,
    scrollback_len: usize,
    rows: u16, cols: u16,
) {
    let mut parser = vt100::Parser::new(rows, cols, scrollback_len);
    // ... DSR/tail tracking same as before ...
    loop {
        // Check for pending resize
        if let Ok(mut resize_opt) = pending_resize.lock() {
            if let Some(size) = resize_opt.take() {
                // Recreate parser with new size (no history replay needed —
                // reader thread owns all bytes internally via parser.process())
                parser = vt100::Parser::new(size.rows, size.cols, scrollback_len);
            }
        }

        match reader.read(&mut buf) {
            Ok(0) => break,
            Ok(n) => {
                bytes_received.fetch_add(n, ...);
                // DSR detection same as before
                // OSC 52 / OSC title detection same as before
                parser.process(&buf[..n]);  // ← parsing happens HERE, on reader thread

                // Publish parsed screen to main thread
                if let Ok(mut screen_guard) = shared_screen.lock() {
                    *screen_guard = parser.screen().clone();
                }
                dirty.store(true, Ordering::Release);

                // Wakeup main thread
                if let Ok(guard) = wakeup.lock() && let Some(ref cb) = *guard { cb(); }
            }
            Err(_) => break,
        }
    }
}
```

**Main thread `Pty::screen()`:**
```rust
pub fn screen(&mut self) -> &vt100::Screen {
    if self.dirty.load(Ordering::Acquire) {
        if let Ok(guard) = self.shared_screen.lock() {
            self.cached_screen = guard.clone();
        }
        self.dirty.store(false, Ordering::Release);
    }
    &self.cached_screen
}
```

**Clone cost:** `O(rows*cols)` per main-thread render (not per PTY read). For a typical 80×24 terminal this is ~1920 cells → microseconds. For larger terminals (200×60 → 12k cells) → tens of microseconds. Acceptable for initial implementation. A SoA/SIMD optimization is future work.

**vt100 compatibility:** `vt100::Screen` must implement `Clone`. If it doesn't, we add a thin wrapper that stores cell data in a Clone-friendly format. Verify by checking the vt100 version in `Cargo.toml`.

**`Pane::screen()` signature** stays `fn screen(&mut self) -> &vt100::Screen` — no API breakage.

**`Pane::screen_mut()`** — no longer needed. The reader thread owns the parser; `set_scrollback()` can work on the cached screen directly.

**Resize:** Sent to reader thread via `pending_resize: Arc<Mutex<Option<PtySize>>>`. Reader thread recreates the parser on the next loop iteration.

**`Pane::drain_pending()`** — used by the session server to forward raw bytes. Can be removed or replaced with `take_pending_bytes()` from the reader thread (store raw bytes in a separate small buffer).

**`TestPane`** changes: TestPane wraps a `vt100::Parser` directly (no reader thread). The `screen()` method simply returns `self.parser.screen()` unchanged — tests are unaffected.
