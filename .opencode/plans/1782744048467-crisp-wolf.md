# Plan: Standardize and Improve Core Window Management Architecture (Issue #48)

## Goal
Completely isolate the layout engine from views into a reusable sub-crate, standardize BSP/N-ary trees with proper heuristics, implement quadrant-based drag-and-drop, snapping previews, and edge resistance.

## Current Architecture (from exploration)

### Layout Module (`crates/term-wm-core/src/layout/`)
- `tiling.rs` ŌĆö `LayoutNode` enum (N-ary, `Vec<children>`), `TilingLayout`, free functions for split/insert/remove/layout
- `floating.rs` ŌĆö `ResizeHandle`, `HeaderDrag`, `FloatingPane`, resize drag logic
- `mod.rs` ŌĆö `RegionMap`, `RectSpec`, gap/handle helpers
- **Issues**: N-ary tree (not strictly BSP binary), no orientation heuristics, `ratatui::Rect`/`Direction` coupling

### Window Manager (`crates/term-wm-core/src/window/window_manager/`)
- `mod.rs` ŌĆö `WindowManager<Id>` struct with `managed_layout: Option<TilingLayout>`, floating windows, z-order
- `layout.rs` ŌĆö `register_managed_layout` orchestrates tiling + floating region computation
- `drag.rs` ŌĆö Header drag, `update_snap_preview`, `apply_snap`, `tile_window_id`
- **Issues**: Snap preview is basic (sensitivity zones only), no quadrant detection, no edge resistance

### Runner (`crates/term-wm-core/src/runner.rs`)
- `auto_layout_for_windows` ŌĆö creates flat N-ary splits (no heuristics)
- `run_app` / `run_window_app` ŌĆö calls `register_managed_layout` on resize

## Plan

### Phase 1: Create `term-wm-layout-engine` sub-crate

**New crate** at `crates/term-wm-layout-engine/` with zero non-std dependencies (pure layout math).

```
src/
  lib.rs          -- re-exports
  rect.rs         -- LayoutRect (i32 coords, u16 dims), quadrant enum
  node.rs         -- BspNode (strict binary BSP), NaryNode (i3-style)
  orientation.rs  -- Orientation, OrientationHeuristic trait
  split.rs        -- split_rects, split_rects_weighted, gap calculation
  insert.rs       -- insert_leaf (BSP + N-ary), remove_leaf
  hit_test.rs     -- hit_test_leaf, quadrant_detection
  snap.rs         -- SnapPreview, EdgeResistance, threshold zones
  layout.rs       -- LayoutEngine trait
```

**Key types:**
```rust
// No ratatui dependency ŌĆö pure math
pub struct LayoutRect { pub x: i32, pub y: i32, pub width: u16, pub height: u16 }

pub enum Orientation { Horizontal, Vertical }

pub enum Quadrant { North, South, East, West }

// Constraints passed down from the window manager (view-agnostic).
pub struct SizeConstraints {
    pub min_width: u16,
    pub min_height: u16,
}

pub enum LayoutError {
    ConstraintViolated(SizeConstraints),
    NotFound,
}

// Integer ratio (p, q) meaning p/(p+q) of the parent's size.
// e.g., (1, 1) = 50/50, (1, 2) = 33/66, (3, 1) = 75/25.
pub struct Ratio(pub u16, pub u16);

// Strict BSP (binary) ŌĆö uses integer ratios, NOT f32.
// Remainder isolation algorithm guarantees no dead zones:
//   child_w = floor(parent_w * p / (p+q))
//   remainder = parent_w - child_w
// The remainder is carried forward, ensuring sum(child widths) == parent_w.
pub enum BspNode<Id> {
    Leaf(Id),
    Split { orientation: Orientation, left: Box<BspNode<Id>>, right: Box<BspNode<Id>>, ratio: Ratio },
}

// N-ary (i3-style)
pub enum NaryNode<Id> {
    Leaf(Id),
    Container { orientation: Orientation, children: Vec<NaryNode<Id>>, weights: Vec<Ratio> },
}
```

**Size constraint enforcement:** All mutation functions (`insert_leaf`, `remove_leaf`, `apply_drag`) take `&SizeConstraints` and return `Result<(), LayoutError>`. On `Err(ConstraintViolated)`, the caller ignores the action ŌĆö visual feedback stops the drag dead.

### Phase 2: Implement Orientation Heuristics

**`OrientationHeuristic` trait:**
```rust
pub trait OrientationHeuristic {
    fn choose(&mut self, area: LayoutRect, depth: usize) -> Orientation;
}
```

Implement:
- `LongestSide` ŌĆö if `area.width >= area.height` ŌåÆ Horizontal, else Vertical
- `Spiral` ŌĆö cycle: Horizontal, Vertical, Horizontal, Vertical (based on depth % 2)
- Default: `LongestSide`

### Phase 3: Quadrant Detection

**Pure function** in `hit_test.rs`:
```rust
pub fn detect_quadrant(cursor: (u16, u16), target: &LayoutRect) -> Quadrant
```
- Compute center of target rect
- Calculate angle of cursor relative to center
- Map to Quadrant: East (0┬░┬▒45┬░), North (45┬░-135┬░), West (135┬░-225┬░), South (225┬░-315┬░)

### Phase 4: Edge Resistance

**`EdgeResistance` struct:**
```rust
pub struct EdgeResistance {
    pub spatial_threshold: u16,   // pixels for fast-move override
    pub temporal_threshold: Duration,  // hold duration for override
    pub magnetic_zone: u16,       // width of magnetic border
}
```

**Algorithm:**
- When a floating window approaches a boundary (within `magnetic_zone`), apply a damping factor to movement delta
- If cursor velocity > `spatial_threshold` px/frame, override resistance
- If mouse held at boundary > `temporal_threshold`, override resistance

### Phase 5: Improve Snap Preview

Enhance `update_snap_preview` to use:
1. Edge zones (existing): left/right/top/bottom edge detection for half-screen snap
2. Quadrant zones (new): when dragged *over* a tiled leaf, use `detect_quadrant` for 4-direction insertion
3. Return `SnapTarget` enum: `Edge(InsertPosition)` | `TiledInsert(TiledSnapshot)`

### Phase 6: Re-integrate with Existing Code

1. Add `term-wm-layout-engine` as a dependency of `term-wm-core`
2. Update `LayoutNode` in `tiling.rs` to delegate to `BspNode` / `NaryNode` internally (or replace entirely)
3. Update `WindowManager::register_managed_layout` to use new engine
4. Update `WindowManager::update_snap_preview` and `apply_snap` to use quadrant detection + edge resistance
5. Update `auto_layout_for_windows` in runner to use `LongestSide` heuristic

### Phase 7: Tests

All new code in `term-wm-layout-engine` must have unit tests:
- `test_longest_side_wide_space` ŌĆö width > height ŌåÆ Horizontal
- `test_longest_side_tall_space` ŌĆö height > width ŌåÆ Vertical
- `test_spiral_alternates` ŌĆö orientation changes every insertion
- `test_quadrant_detection_east` ŌĆö cursor in east quadrant returns East
- `test_edge_resistance_damping` ŌĆö movement near boundary is damped
- `test_edge_resistance_velocity_override` ŌĆö fast movement bypasses resistance
- `test_snap_preview_edge` ŌĆö cursor at screen edge returns Edge snap
- `test_snap_preview_quadrant` ŌĆö cursor over tiled leaf returns Quadrant snap
- `test_bsp_insert_and_remove` ŌĆö basic tree operations
- `test_insert_rejects_below_minimum` ŌĆö insertion violating min width/height returns `Err`
- `test_drag_rejects_below_minimum` ŌĆö resize drag below minimum returns `Err`
- Property tests with `proptest` for BSP tree invariants

## Phase 8 (Expanded Scope): Complete Layout Math Consolidation

### Goal
Move **all** pure-math types and functions from `term-wm-core` into `term-wm-layout-engine`, making the engine fully self-contained with **zero** runtime dependencies. All layout math ŌĆö tiled trees, floating resize, z-order, focus rings, scroll state, region maps, rect utilities, percentage resolution ŌĆö lives in the engine. `term-wm-core` becomes a thin rendering bridge.

1. **­¤ö┤ `insert_leaf` constraint enforcement**: `BspNode::insert_leaf` and `NaryNode::insert_leaf` accept `&SizeConstraints` but prefix it `_constraints` and never validate.
