# Plan: Floating Window Drop Shadow

## Problem

Floating windows sit flat against the background with no visual depth cue. A translucent drop shadow signals stacking order.

## Approach

Before each floating window is composited onto the main frame, directly mutate the frame buffer cells at the shadow position (offset by 2 right, 1 down) to darken them (set bg to `theme::surface()` + apply `Modifier::DIM` to foreground). Since windows are drawn bottom-to-top, each window's shadow naturally goes behind it and lower windows' shadows are overlaid by higher windows.

**Key insight**: We use direct cell mutation (not `blit_from_signed`) so the underlying text from lower-z windows remains visible but darkened — a true translucent effect. `Modifier::DIM` in ratatui affects foreground only, so it dims the text on top of the darkened background.

**Critical constraint: visual-only.** The shadow is painted directly on the frame buffer's *cells* after all layout/hit-testing/event-handling data structures have been computed. It must NOT affect:
- Hit-testing (mouse click targets, resize handles, headers)
- Region maps or occlusion detection
- Window focus or z-order
- Floating window positioning or clamping
- Any `WindowSurface` / `FloatRect` / layout computation

The shadow is a pure render-time effect — it reads `draw_shadow` from the already-computed `WindowSurface` and mutates only the terminal framebuffer.

### 2. `crates/term-wm-core/src/window/window_manager/mod.rs` (~line 108-113)
Add `draw_shadow: bool` to `WindowSurface`:
```rust
pub struct WindowSurface {
    pub full: Rect,
    pub inner: Rect,
    pub dest: crate::window::FloatRect,

```
This keeps `composite_window` decoupled from `WmConfig` / `WindowManager`.

### 3. `crates/term-wm-core/src/window/window_manager/layout.rs` (~line 483-493)
In `window_draw_plan`, set `draw_shadow: self.is_window_floating(id) && self.config.shadow_enabled`:

surface: super::WindowSurface {
    full, inner, dest,

Both `SystemWindowDraw` and `WindowDrawContext` go through this same path.

### 4. `crates/term-wm-core/src/runner.rs` (~line 710-742)
In `composite_window`, before the final `frame.blit_from_signed(&buffer, surface.dest)`:

```rust
// NEW: render drop shadow via direct cell mutation (preserves underlying text)
if surface.draw_shadow {
    let frame_area = frame.area();
    let sx = surface.dest.x.saturating_add(SHADOW_OFFSET_X);
    let sy = surface.dest.y.saturating_add(SHADOW_OFFSET_Y);
    let ex = sx.saturating_add(surface.dest.width as i32);
    let ey = sy.saturating_add(surface.dest.height as i32);
    let buf = frame.buffer_mut();
    for y in sy..ey {
        if y < frame_area.y as i32 || y >= (frame_area.y + frame_area.height) as i32 {
            continue;
        }
        for x in sx..ex {
            if x < frame_area.x as i32 || x >= (frame_area.x + frame_area.width) as i32 {
                continue;
            }
            if let Some(cell) = buf.cell_mut((x as u16, y as u16)) {
                if !cell.symbol().starts_with(' ') {
                    cell.modifier.insert(Modifier::DIM);
                }
                cell.set_bg(crate::theme::surface());
            }
        }
    }
}
frame.blit_from_signed(&buffer, surface.dest);  // existing — window paints over its own shadow
```

### 6. `crates/term-wm-core/src/theme.rs`
Add `Surface` to `BgColor` enum, match arm in `Theme::bg()`, and convenience wrapper:
```rust
pub enum BgColor {
    // ... existing ...
    Surface,
}

// in Theme::bg():
BgColor::Surface => self.surface,

// convenience wrapper:
pub fn surface() -> Color { current().bg(BgColor::Surface) }
```

Also add exclusions to `PHYSICALLY_IMPOSSIBLE` in the `exhaustive_fg_times_bg` test (the `Surface` bg is very dark — near-black text like `MenuSelectedFg` / `SelectionFg` can't reach 3.0:1 against it, but these pairs never occur in practice since the shadow is a visual effect, not a text surface):
```rust

```
