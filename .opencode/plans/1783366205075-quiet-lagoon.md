# Plan: Rename overlay.rs and Associated Terminology

## Problem

The term "overlay" is used for multiple distinct concepts, causing confusion:

1. **`overlay.rs`** - Implements both the window manager's **command menu mode** (super-key interaction) AND generic modal overlay management
2. **`Overlay` trait** - Base trait for components that appear on top of other content
3. **`OverlayId` enum** - Identifies specific modal dialogs (Help, ExitConfirm, Keybindings)
4. **`ComponentContext::overlay`** - Boolean flag indicating a component is rendered as an overlay

The core issue: `overlay.rs` mixes two concerns:
- **Command menu methods**: `open_wm_overlay()`, `close_wm_overlay()`, `wm_overlay_visible()`, `fold_menu()`, `handle_wm_menu_event()`, `wm_menu_consumes_event()`
- **Modal overlay management**: `close_exit_confirm()`, `exit_confirm_visible()`, `help_overlay_visible()`, `close_help_overlay()`, `handle_help_event()`, `handle_exit_confirm_event()`

## Proposed Changes

### 1. Split `overlays.rs` into Two Files

**Create `command_menu.rs`** - Extract super-key menu mode logic:
- `open_wm_overlay()` â†’ `open_command_menu()`
- `open_wm_overlay_no_passthrough()` â†’ `open_command_menu_no_passthrough()`
- `close_wm_overlay()` â†’ `close_command_menu()`
- `wm_overlay_visible()` â†’ `command_menu_visible()`
- `fold_menu()` (keep name - already clear)
- `handle_wm_menu_event()` (keep name - already clear)
- `wm_menu_consumes_event()` (keep name - already clear)

**Keep `overlays.rs`** - Retain generic modal overlay management:
- `close_exit_confirm()`
- `exit_confirm_visible()`
- `help_overlay_visible()`
- `close_help_overlay()`
- `handle_help_event()`
- `handle_exit_confirm_event()`

### 2. Move `render_overlays()` to `mod.rs`

`render_overlays()` draws from both the command menu and modal overlays, so it must live in the central orchestrator (`mod.rs`) to avoid cross-module circular references.

### 3. Update Field Names in `WindowManager`

In `crates/term-wm-core/src/window/window_manager/mod.rs`:
