# Plan: Rename overlay.rs and Associated Terminology

## Problem

The term "overlay" is used for multiple distinct concepts, causing confusion:

1. **`overlay.rs`** - Implements the window manager's **command menu mode** (super-key interaction)
2. **`Overlay` trait** - Base trait for components that appear on top of other content
3. **`OverlayId` enum** - Identifies specific modal dialogs (Help, ExitConfirm, Keybindings)
4. **`ComponentContext::overlay`** - Boolean flag indicating a component is rendered as an overlay

The core issue: When reading `overlay.rs`, one expects it to implement the `Overlay` trait concept, but it actually implements the window manager's super-key menu mode.

## Proposed Rename

**Rename `overlay.rs` → `command_menu.rs`** and associated terminology:

| Current | New | Rationale |
|---------|-----|-----------|
| `overlay.rs` | `command_menu.rs` | File implements super-key command menu mode |
| `overlay_visible: bool` | `command_menu_visible: bool` | Tracks whether command menu is active |
| `open_wm_overlay()` | `open_command_menu()` | Opens the command menu |
| `close_wm_overlay()` | `close_command_menu()` | Closes the command menu |
| `wm_overlay_visible()` | `command_menu_visible()` | Query if command menu is active |
| `wm_overlay_enabled` (config) | `wm_command_menu_enabled` | Config flag for this feature |

**Keep unchanged:**
- `Overlay` trait - Correctly named for the general concept
- `OverlayId` enum - Correctly names modal dialog identifiers
- `MenuOverlay` trait - Correctly names menu-specific overlay behavior
- `ComponentContext::overlay` - Correctly flags overlay rendering context
