# Plan: Debug Clipboard Copy — "Only copies first two characters"

## Problem
User reports that auto-copy on drag-end only copies the first ~2 characters of any selection, regardless of selection size or content.

## Core Files
| File | Role |
|------|------|
| `crates/term-wm-core/src/utils/selectable_text.rs` | SelectionController (anchor/cursor tracking, `selection_range`, `finish_drag`) |
| `crates/term-wm-ui-components/src/terminal.rs` (lines 560-621) | `logical_position_from_point`, `selection_text_for_range` — row_base translation, text extraction |
| `crates/term-wm-core/src/runner.rs` (lines 527-542) | `update_selection_snapshot` — auto-copy trigger on drag-end |
| `crates/term-wm-core/src/window/window_manager/mod.rs` (lines 652-665) | `copy_selection_to_clipboard` — writes to arboard + OSC 52 |
| `crates/term-wm-pty-engine/src/clipboard.rs` | Clipboard backend (arboard + OSC 52) |

## Investigation Steps

### Step 1: Add temporary debug logging
Add `tracing::info!` in `selection_text_for_range` to log:
- The input `SelectionRange` (start row, start col, end row, end col)
- The computed `row_base`, `rows`, `cols`
- The adjusted end_row/end_col after the `end_col == 0` branch
- The clamped visible-row indices
- The length of the returned string

### Step 2: Add debug logging in `copy_selection_to_clipboard`
Log text length and first 200 chars to confirm the full text reaches the WindowManager.

### Step 3: Build and reproduce
Run `cargo build` and reproduce the issue. Check the logs.

### Step 4: Targeted fixes based on findings
Based on what the logs reveal, the fix will target one of:

**Scenario A: `selection_text_for_range` returns wrong range**
- If `end_col == 0` adjustment fires incorrectly (e.g., when row maps above viewport after saturating_sub), the range could lose rows
- Fix: guard the adjustment with a check that `end_row > row_base` (i.e., the adjustment doesn't reach above the viewport)

**Scenario B: `contents_between` returns fewer chars than expected**
- If the vt100 screen size is 2 columns (unlikely but possible if PTY hasn't received resize), all selections would be 2 chars max
- Fix: ensure PTY resize is sent before selection extraction, or clamp correctly
- More likely: `cols` is correct but the start_col/end_col mapping produces a tiny range

**Scenario C: Clipboard write truncates**
- arboard or OSC 52 path has a size/encoding issue
- Fix: try writing via arboard only, bypass OSC 52, to isolate backend

**Scenario D: Row_base changed between render and extraction**
- If new output arrived between when user made the selection and when `selection_text_for_range` reads `pane.screen()`, the row_base could shift, causing the absolute logical positions to map to wrong visible rows
- Fix: snapshot the screen content at the time of selection, or re-derive row_base consistently
