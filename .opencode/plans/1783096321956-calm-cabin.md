# Plan: Channel Backpressure Tuning

## Goal
Reduce CPU under continuous PTY streaming (e.g., `cat /dev/random`) by engaging mechanical backpressure earlier. No dirty rectangles. No reader thread changes.

## Diagnosis
The bounded channel between PTY reader threads and the event loop has capacity 256. Under `cat /dev/random`, the reader processes 4KB batches at ~2ms each. It takes 256 wakeups × ~2ms = ~512ms before the channel fills and the reader blocks on `send()` — by which time ~1MB of data has been parsed. During that window, the reader spins at 100% CPU.

After the channel fills at ~512ms, the reader blocks on `send()` → `reader.read()` pauses → OS pipe buffer fills (8KB macOS, 64KB Linux) → child's `write()` blocks. This is correct mechanical backpressure — it just engages way too late.

## Fix: Reduce Channel Capacity

**Single constant change** in `crates/term-wm-core/src/io/unified_event_source.rs`:

```rust
// Before:
const EVENT_CHANNEL_CAPACITY: usize = 256;

// After:
const EVENT_CHANNEL_CAPACITY: usize = 8;
```

**No changes to the reader loop, the coalesce timer, or any other logic.**

### Why 8

| Capacity | Batches before block | Time before block | Throughput per cycle | Est. CPU/reader |
|----------|---------------------|-------------------|---------------------|-----------------|
| 256 (current) | 256 | ~512ms | ~1MB | ~100% |
| 8 | 8 | ~16ms | ~32KB | ~25-35% |
| 4 | 4 | ~8ms | ~16KB | ~15-20% |
| 1 | 1 | ~2ms | ~4KB | ~5-10% |

With capacity=8, the reader processes 8 batches (32KB, ~16ms CPU), then blocks on the 9th `send()`. The coalesce timer (3ms from first wakeup) triggers a render long before the reader blocks — the render drains the channel, keeping throughput high while naturally pacing the reader.

The channel also carries `Input(Event)`, `Signal`, and `Tick` messages — capacity=8 ensures keyboard input and signals aren't starved by PTY data. Input events are rare (<1/s during idle, maybe 10/s while typing) compared to PTY wakeups, so they won't contend significantly.

### Tuning knob

`EVENT_CHANNEL_CAPACITY` is trivially adjustable. If throughput feels sluggish under grep/log-tail workloads, bump to 16 or 32. If CPU is still too high, drop to 4. The constant is in one place.

## Tests

Existing tests should pass unchanged. The test `dirty_windows_cleared_after_poll_ok_false` creates its own channel with its own capacity. The channel capacity doesn't affect any test assertion — it only affects timing (when backpressure engages). No test changes needed.

However, check test `dirty_windows_causes_streaming_profile` — it passes a pre-built `dirty_windows` set directly, bypassing the channel. No channel dependency.
