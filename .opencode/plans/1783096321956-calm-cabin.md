## Main Thread Hotspots

| Rank | % | Location | What |
|------|---|----------|------|
| 1 | ~60% | `drop_in_place<ScrollViewComponent<TerminalComponent>>` | **Dropping component data** — 3 offsets: +200 (47%), +64 (8%), +392 (4.5%). The +200 site is `mov w2, #1` before `__rust_dealloc` on a `command_description: String` buffer. |
| 2 | ~8% | `BTreeMap::insert_recursing` | Window map manipulation |
| 3 | ~5% | `VecDeque::Iter::fold` | Terminal screen row iteration |
| 4 | ~4% | `Grid::visible_row` / `Screen::cell` / `Cell::contents` | vt100 per-cell screen reads |
| 5 | ~4% | `TerminalComponent::render` | Main component render method |
| 6 | ~3% | `Vec::drop` | Vec deallocation |
| 7 | ~2% | `WmBottomPanelComponent::fmt::Debug` | Debug vtable artifact |

### Key Finding #1: 60% in `drop_in_place` is the dominant cost

The compiler-generated drop function for `ScrollViewComponent<TerminalComponent>` is executing:
1. `Box<dyn Pane>::drop` → vtable call → `Pty::drop` → shutdown + unpark reader
2. `__rust_dealloc` for the Pane Box heap allocation
3. Loop over link overlay data (`Vec<Option<Arc<str>>>`)
4. `__rust_dealloc` for `command_description: String` buffer (the +200 site)
5. `Arc::drop` for `screen_arc` (atomic decref)
6. Rc decrement inside `cached_screen`

**Root cause unclear**: Components persist in `App.terminals BTreeMap` across frames — no explicit drop during rendering. The profiler shows 60% of samples landing in this function, which means the drop is being called frequently. Possible explanations:

- **A)** `Buffer::empty` / `Buffer::drop` (ratatui offscreen buffer) is the real cost; the profiler misattributes to `drop_in_place` due to code layout. The offscreen buffer at line 679 of `runner.rs` allocates `Vec<Cell>` sized to window dimensions (~200MB for 4K) and drops it every frame. At 60fps × 2 windows = 24GB/s memory bandwidth.
- **B)** Something in the render loop creates and destroys components (less likely — investigation confirms no ownership transfer during rendering).
- **C)** The `catch_unwind` panic recovery in the render path is dropping stack values that include component references.

### Key Finding #2: Reader threads are healthy

- **~74% parked** in `thread::park` — expected idle
- **~26% active** in `vte::Parser::advance` — PTY output parsing, necessary work

### Key Finding #3: Debug vtable artifacts

`WmMenuOverlay::fmt::Debug` (+59) and `FmtLevel::Display::fmt` (+327) appear on the hot path. These are vtable entries for `dyn Overlay` and `dyn BottomPanel` trait objects — the Debug methods are **not explicitly called** but the profiler attributes samples near vtable dispatch sites to all vtable entries.

## Plan of Action

### Step 1: Validate hypotheses (profiling instrumentation)

- Add frame pointer support to the release build (`-C force-frame-pointers=yes`) for cleaner stack traces
- Profile with:
  - 1 small terminal window (80×24) instead of 2 — to isolate Buffer allocation cost
  - Disable dimming on unfocused windows (comment out lines 685-688 in `runner.rs`) — to test if the Vec iteration is the source
- ✅ Release build, `cargo clippy -D warnings`, `cargo test` workspace

### Step 2: Optimize the offscreen Buffer allocation (if confirmed as the 60% source)

The `Buffer::empty(local_area)` at `runner.rs:679` allocates a fresh `Vec<Cell>` per window per frame. Options:

**A) Reuse a persistent scratch buffer** — Replace `Buffer::empty(area)` with a method that reuses a `Vec<Cell>` and only reallocates when the area grows:
```rust
struct ScratchBuffer {
    cells: Vec<ratatui::core::buffer::Cell>,
    area: Rect,
}
impl ScratchBuffer {
    fn resize(&mut self, area: Rect) {
        let len = area.width as usize * area.height as usize;
        if self.cells.len() < len {
            self.cells.resize(len, Cell::default());
        }
        // zero out the used portion
        for cell in self.cells[..len].iter_mut() {
            *cell = Cell::default();
        }
        self.area = area;
    }
}
```
→ **Eliminates `Vec::empty` allocation and `Vec::drop` deallocation per frame.** Only grows when the window gets larger; never frees until scratch buffer is dropped.

**B) Pool per-window scratch buffers** — Store the `ScratchBuffer` in `TerminalComponent` or `WindowManager`.

### Step 3: Reduce vt100 per-cell overhead (~10% in Grid::visible_row / Screen::cell / Cell::contents)

Current render path in `TerminalComponent::render` calls `pane.screen()` which calls `ArcSwap::load_full()`, then iterates rows and cells one-by-one. Each cell access goes through `Grid::visible_row` → `Vec::index` → `Row::cells` → `Cell::contents`.

**Options:**
- **A) Bulk cell read** — Add a method to `Pane` that returns a `Vec<&[Cell]>` (row slices) in one call, avoiding per-cell bounds checks.
- **B) Cache rendered output** — Keep a `String` or `Vec<Vec<Cell>>` cache of the last rendered screen; only re-render changed rows (requires dirty-rect tracking).
- **C) Zero-cost access** — If vt100 allows, access the Grid's internal `Vec<Row>` directly (unsafe, but avoids bounds checks in hot loop).

### Step 4: Remove Debug trait overhead (~2% in vtable artifacts)

- Remove `std::fmt::Debug` supertrait from `Overlay` and `BottomPanel` trait definitions
- Remove `#[derive(Debug)]` from `WmMenuOverlay` and `WmBottomPanelComponent`
- Write a manual `Debug` impl for `WmMenuOverlay` that doesn't require `R: Debug`:
  ```rust
  impl<R> std::fmt::Debug for WmMenuOverlay<R> {
      fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
          f.debug_struct("WmMenuOverlay")
              .field("outlined", &self.outlined)
              .field("anchor", &self.anchor)
              .finish_non_exhaustive()
      }
  }
  ```
- Drop `R: std::fmt::Debug + 'static` bounds from all `impl` blocks on `WmMenuOverlay`
- If any test uses `{:?}` on these types, update them.

### Step 5: Reduce terminal exit check overhead (~2% in has_exited)

The `has_exited()` method on `Pty` is called every idle render tick. It checks `self.exited` and may call `child.try_wait()` which reads `/proc` / `waitpid`. Consider caching the exit status in an `AtomicBool` set by the reader thread.

### Step 6: Rebuild and re-profile

- `cargo build --release`
- Profile with samply:
  ```bash
  samply record ./target/release/term-wm -n 2 cat /dev/random
  ```
- Verify: drop_in_place cost should drop from 60% to <20%
- Verify: no freeze, CPU < 5%, responsive keyboard/mouse
