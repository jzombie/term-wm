# Plan: Backpressure + Render Pipeline Performance Fixes

## Goal
Maximize performance under continuous PTY streaming (e.g., `cat /dev/random`) **without implementing dirty rectangles**.

## Problem
Two windows running `cat /dev/random` consume ~204% CPU:
- **2× PTY reader threads** (~100% each): tight loop of `read(4K)` → `parser.process(4K random bytes)` → `shared_screen.clone()` → unbounded wakeup signaling. The bounded channel (256 slots, ~1MB slack) means backpressure engages too late.
- **Main thread** (~4%): full-screen render on every coalesce tick. Not the bottleneck.

## Fix 1: Gate Wakeup Per Consumption Cycle + Park Reader

**Core idea:** The reader sends at most **one** PtyWakeup per dirty→consumed cycle. When the renderer hasn't consumed, the reader parks its thread instead of spinning.

### Changes

**`crates/term-wm-pty-engine/src/pty.rs`** — `parser_read_loop()`, lines 517-528:

Replace `dirty.store(true, Release)` + unconditional `cb()` with gated wakeup:

```rust
if let Ok(mut guard) = shared_screen.lock() {
    *guard = parser.screen().clone();
}

// Gate: only one wakeup per consumption cycle.
let was_dirty = dirty.swap(true, Ordering::Release);
if !was_dirty {
    if let Ok(guard) = wakeup.lock()
        && let Some(ref cb) = *guard
    {
        cb();
    }
} else {
    // Renderer hasn't consumed — park until Pty::screen() unparks.
    // This is NOT a spin: thread blocks until unparked.
    thread::park();
}
```

**`crates/term-wm-pty-engine/src/pty.rs`** — `Pty::screen()`, ~line 338:

After consuming, unpark reader:

```rust
pub fn screen(&mut self) -> &vt100::Screen {
    self.poll_foreground();
    if self.dirty.swap(false, Ordering::Acquire) {
        if let Ok(guard) = self.shared_screen.lock() {
            self.cached_screen = guard.clone();
        }
        if let Some(reader) = &self.reader {
            reader.thread().unpark();
        }
        // ... DSR response ...
    }
    &self.cached_screen
}
```

**`crates/term-wm-pty-engine/src/pty.rs`** — `Pty::set_wakeup()`, ~line 175:

Reader might park before `set_wakeup` is called during init. Unpark on set:

```rust
pub fn set_wakeup(&mut self, cb: Option<Arc<dyn Fn() + Send + Sync>>) {
    if let Ok(mut guard) = self.wakeup.lock() {
        *guard = cb;
    }
    // Reader might park before wakeup is set — wake it up
    if let Some(reader) = &self.reader {
        reader.thread().unpark();
    }
}
```

### Effect

```
Reader:    read(4K) → parse → publish → park (blocked)
                                          ↑
            (reader stays parked until renderer consumes)
            
Renderer:  render → pane.screen() → consumes dirty → unpark reader
                                                    ↓
Reader:    unparked → read(4K) → parse → publish → send PtyWakeup → park
```

The reader spends most time **parked** (~0% CPU). The OS pipe buffer fills (8KB macOS, 64KB Linux) and the child's `write()` blocks — true mechanical backpressure.

## Fix 2: Simplify Coalesce

With Fix 1, at most one PtyWakeup per window per cycle. The 3ms coalesce window almost never batches multiple wakeups for the same window.

**`crates/term-wm-core/src/io/unified_event_source.rs`** — `poll()`, PtyWakeup handler:

Replace the coalesce-timer loop with immediate `Ok(false)`:

```rust
Ok(UnifiedEvent::PtyWakeup(key)) => {
    self.dirty_windows.insert(key);
    return Ok(false);
}
```

Remove:
- `const COALESCE_DELAY` (line 29)
- `coalesce_deadline: Option<Instant>` field and all references
- `self.coalesce_deadline = None` in Input handler (line 186) and timeout (line 218)

The timeout path (line 219, `self.dirty_windows.clear()`) stays — it's what clears dirty_windows after streaming stops, letting the profile drop back to PowerSaver.
