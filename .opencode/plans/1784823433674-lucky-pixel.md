# Plan: Spatial-Aware Float-to-Tiled Reattachment (Corrected)

## Problem

`toggle_tiling()` inserts windows sequentially via `tile_window_key()` in arbitrary order. Each window's spatial context is evaluated against a partial layout containing only previously-inserted windows. Since floating rects are cleared during insertion, subsequent windows can't reference each other's original positions. Result: the tiled layout after float→tiled cycles drifts substantially.

The prior dominant-axis approach worsened this by ignoring the user's cursor-based spatial intent entirely. This plan restores quadrant-based intent mapping but bridges it with constraint-safe candidate exhaustion.

## Approach

Three independent changes:

### A. Add `Quadrant::to_insert_position()` to rect.rs

Map user cursor quadrant to the user-intended InsertPosition:

```rust
impl Quadrant {
    pub fn to_insert_position(self) -> InsertPosition {
        match self {
            Quadrant::North => InsertPosition::Top,
            Quadrant::South => InsertPosition::Bottom,
            Quadrant::West => InsertPosition::Left,
            Quadrant::East => InsertPosition::Right,
        }
    }
}
```

### B. Constraint-Aware `LayoutNode::insert_leaf`

Modify `LayoutNode::insert_leaf` (tiling.rs) to accept `area: Rect` and `constraints: &SizeConstraints`. In the `Leaf` case, before mutating the tree, check whether the split dimensions satisfy the constraints. Return `false` on violation to natively drive the exhaustion loop, matching `BspNode::insert_leaf` semantics.

```rust
pub fn insert_leaf(
    &mut self,
    target: Id,
    insert: Id,
    position: InsertPosition,
    area: Rect,
    constraints: &SizeConstraints,
) -> bool {
    match self {
        LayoutNode::Leaf(current) => {
            if *current != target { return false; }
            // O(1) constraint check before mutation
            let viable = match position {
                InsertPosition::Left | InsertPosition::Right =>
                    area.width / 2 >= constraints.min_width
                    && area.height >= constraints.min_height,
                InsertPosition::Top | InsertPosition::Bottom
                | InsertPosition::TopLeft | InsertPosition::TopRight
                | InsertPosition::BottomLeft | InsertPosition::BottomRight =>
                    area.height / 2 >= constraints.min_height
                    && area.width >= constraints.min_width,
            };
            if !viable { return false; }
            // ... existing split mutation ...
            true
        }
        // existing Split/Void cases unchanged
    }
}
```

Callers pass the candidate's `target_rect` as `area` and `SizeConstraints { min_width: MIN_TILE_WIDTH, min_height: MIN_TILE_HEIGHT }` as constraints.

This eliminates `split_is_viable` entirely — the engine owns constraint validation.

### C. Bulk spatial insertion in `toggle_tiling` and `apply_position_based_layout` (drag.rs)

Replace the sequential `tile_window_key` loop in `toggle_tiling` with area-descending bulk insertion. Also standardize `apply_position_based_layout` to use the same pipeline.

**`toggle_tiling`** (float→tiled path):

```rust
pub fn toggle_tiling(&mut self) {
    if self.managed_layout.is_some() {
        self.float_all_windows();
    } else {
        use crate::layout::LayoutNode;
        let keys = self.mapped_windows();
        
        // Collect windows with their floating rects
        let mut with_rects: Vec<(WindowKey, Rect)> = keys.iter()
            .filter_map(|k| {
                self.floating_rect(*k)
                    .map(|spec| (*k, spec.resolve(self.managed_area)))
            })
            .collect();
        
        // Sort by area descending — largest windows anchor the macro-geometry
        with_rects.sort_by_key(|(_, r)| {
            u32::MAX - (r.width as u32) * (r.height as u32)
        });
        
        // Build tree: first window is root, rest inserted spatially
        for (i, (key, rect)) in with_rects.iter().enumerate() {
            self.clear_floating_rect(*key);
            if i == 0 {
                self.managed_layout = Some(TilingLayout::new(LayoutNode::leaf(*key)));
                continue;
            }
            // ... spatial insertion logic ...
        }
    }
}
```

The spatial insertion for subsequent windows:
1. Capture rect center from `rect`
2. Compute current layout regions
3. Use sorted_candidates to get priority-ordered candidates
4. For each candidate: compute quadrant position via `detect_quadrant` + `Quadrant::to_insert_position()`, attempt `insert_leaf`
5. If insert_leaf fails (constraint violation or stale target), advance to next candidate
6. If all candidates fail, call `insert_window_balanced`

### D. Standardize `apply_position_based_layout`

Apply the same area-descending spatial insertion to `apply_position_based_layout` in `drag.rs:520`. Replace the existing `calculate_tiling_order` + `insert_window_balanced` loop with the area-sorted bulk pipeline, then apply the final `split_root` for the anchor window.

### E. Cleanup

Remove `dominant_axis_insert_position` and `split_constraints_for` from hit_test.rs and lib.rs if present.
