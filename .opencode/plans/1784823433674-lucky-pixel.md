# Plan: Spatial-Aware Float-to-Tiled Reattachment (Corrected)

## Problem

`toggle_tiling()` inserts windows sequentially via `tile_window_key()` in arbitrary order. Each window's spatial context is evaluated against a partial layout containing only previously-inserted windows. Since floating rects are cleared during insertion, subsequent windows can't reference each other's original positions. Result: the tiled layout after float→tiled cycles drifts substantially.

The prior dominant-axis approach worsened this by ignoring the user's cursor-based spatial intent entirely. This plan restores quadrant-based intent mapping but bridges it with constraint-safe candidate exhaustion.

## Approach

Three independent changes:

### A. Add `Quadrant::to_insert_position()` to rect.rs

Map user cursor quadrant to the user-intended InsertPosition:

```rust
impl Quadrant {
    pub fn to_insert_position(self) -> InsertPosition {
        match self {
            Quadrant::North => InsertPosition::Top,
            Quadrant::South => InsertPosition::Bottom,
            Quadrant::West => InsertPosition::Left,
            Quadrant::East => InsertPosition::Right,
        }
    }
}
```

### B. Pre-Computation Constraint Guard in Exhaustion Loop

Do NOT modify `LayoutNode::insert_leaf` (keep its original 3-arg signature). Instead, add a `split_is_viable` helper in the core and use it at the call site BEFORE calling `insert_leaf`:

```rust
fn split_is_viable(target_rect: &Rect, pos: InsertPosition) -> bool {
    match pos {
        InsertPosition::Left | InsertPosition::Right =>
            target_rect.width / 2 >= crate::constants::MIN_TILE_WIDTH
            && target_rect.height >= crate::constants::MIN_TILE_HEIGHT,
        InsertPosition::Top | InsertPosition::Bottom
        | InsertPosition::TopLeft | InsertPosition::TopRight
        | InsertPosition::BottomLeft | InsertPosition::BottomRight =>
            target_rect.height / 2 >= crate::constants::MIN_TILE_HEIGHT
            && target_rect.width >= crate::constants::MIN_TILE_WIDTH,
    }
}
```

The exhaustion loop calls `split_is_viable` for each candidate. If it returns `false`, `continue` to the next candidate. Only call `insert_leaf` when the guard passes.

Additionally, apply the same `split_is_viable` guard inside `TilingLayout::insert_window_balanced` (tiling.rs) before the `self.root.insert_leaf(largest_id, insert, pos)` call. If the guard returns `false` for `largest_rect`/`pos`, skip `insert_leaf` and fall through to `split_root` immediately. This prevents degenerate splits in the balanced fallback path.

### C. Bulk spatial insertion in `toggle_tiling` and `apply_position_based_layout` (drag.rs)

Replace the sequential `tile_window_key` loop in `toggle_tiling` with area-descending bulk insertion. Also standardize `apply_position_based_layout` to use the same pipeline.

**`toggle_tiling`** (float→tiled path):

```rust
pub fn toggle_tiling(&mut self) {
    if self.managed_layout.is_some() {
        self.float_all_windows();
    } else {
        use crate::layout::LayoutNode;
        let keys = self.mapped_windows();
        
        // Partition: windows WITH floating_rect use spatial insertion;
        // windows WITHOUT get appended via insert_window_balanced.
        let mut with_rects: Vec<(WindowKey, Rect)> = Vec::new();
        let mut without_rects: Vec<WindowKey> = Vec::new();
        for &k in &keys {
            if let Some(spec) = self.floating_rect(k) {
                with_rects.push((k, spec.resolve(self.managed_area)));
            } else {
                without_rects.push(k);
            }
        }
        
        // Sort by area descending — largest windows anchor the macro-geometry
        with_rects.sort_by_key(|(_, r)| {
            u32::MAX - (r.width as u32) * (r.height as u32)
        });
        
        // Build tree: first window is root, rest inserted spatially
        for (i, (key, rect)) in with_rects.iter().enumerate() {
            self.clear_floating_rect(*key);
            if i == 0 {
                self.managed_layout = Some(TilingLayout::new(LayoutNode::leaf(*key)));
                continue;
            }
            // ... spatial insertion logic (with split_is_viable guard) ...
        }
        // Append windows without floating rects via balanced insertion
        for &key in &without_rects {
            self.clear_floating_rect(key);
            if let Some(ref mut layout) = self.managed_layout {
                layout.insert_window_balanced(key, self.managed_area);
            } else {
                self.managed_layout = Some(TilingLayout::new(LayoutNode::leaf(key)));
            }
        }
    }
}
```

The spatial insertion for subsequent windows:
1. Capture rect center from `rect`
2. Compute current layout regions
3. Use sorted_candidates to get priority-ordered candidates
4. For each candidate: compute quadrant position via `detect_quadrant` + `Quadrant::to_insert_position()`, check `split_is_viable(target_rect, pos)`. If not viable, `continue` to next candidate.
5. Attempt `layout.root_mut().insert_leaf(target_key, key, pos)`. If it fails (stale target), advance.
6. If all candidates exhausted, call `insert_window_balanced`.

### D. Standardize `apply_position_based_layout`

Apply the same area-descending spatial insertion to `apply_position_based_layout` in `drag.rs:520`. Replace the existing `calculate_tiling_order` + `insert_window_balanced` loop with the area-sorted bulk pipeline, then apply the final `split_root` for the anchor window.

### E. Revert Unauthorized `LayoutNode::insert_leaf` Changes + Guard `insert_window_balanced`

The implementation incorrectly modified `LayoutNode::insert_leaf` in tiling.rs to add `area`/`constraints` parameters and updated all callers. These changes must be reverted:

- Revert `LayoutNode::insert_leaf` to its original 3-argument signature: `(&mut self, target: Id, insert: Id, position: InsertPosition) -> bool`
- Revert all callers of `insert_leaf` back to 3 arguments (tiling.rs tests, runner.rs, drag.rs, layout.rs)
- Remove the `SizeConstraints` import from tiling.rs
- Remove the test `test_constraints()` helper and `TEST_AREA` constant from tiling.rs tests

All constraint validation moves to `split_is_viable` at the call site — not in the tree method.

**Additionally**, add a `split_is_viable` guard inside `TilingLayout::insert_window_balanced` in tiling.rs, immediately before the `self.root.insert_leaf(largest_id, insert, pos)` call. If the guard returns false for `largest_rect`/`pos`, fall through to `split_root` without calling `insert_leaf`. The guard function itself is defined in drag.rs (shared between exhaustion loops) and imported in tiling.rs.

### F. Cleanup

Remove `dominant_axis_insert_position` and `split_constraints_for` from hit_test.rs and lib.rs if present.
