# Plan: Spatial-Aware Float-to-Tiled Reattachment

## Problem

When switching from float to tiled mode (via `toggle_tiling` → `tile_window_key`), windows are inserted into the layout tree via `insert_window_balanced`, which picks the largest region and splits it — completely ignoring each window's floating position. Same issue in `reattach_to_tiling_layout` (Iconic→Mapped), which blindly inserts next to `current_focus` with `InsertPosition::Right`.

Result: the tiled layout after float→tiled→float→tiled cycles drifts substantially from the user's original spatial arrangement.

## Approach

Two-phase spatial insertion pipeline, used in both `tile_window_key` and `reattach_to_tiling_layout`:

1. Compute the floating window's center from its stored `floating_rect` (if present).
2. **Phase 1 — exact hit-test**: `.contains()` against each rect from `layout.regions(area)`. If the center falls inside a tiled rect, that's the target.
3. **Phase 2 — Euclidean fallback**: if the center lands in a gap between tiles (common with gutter configs), compute squared Euclidean distance from center to each region's center and pick the closest. **This is the fallback, not `current_focus`.**
4. Use `detect_quadrant` on the resolved target rect → `Quadrant` → `InsertPosition` (mapping: North→Top, South→Bottom, West→Left, East→Right).
5. Execute `insert_leaf(target, key, pos)`.

The only acceptable fallback to `current_focus` / `insert_window_balanced` is when `floating_rect` is `None` (window was never floating, e.g., newly created window).

All infrastructure already exists: `LayoutRect::center()`, `detect_quadrant`, `Quadrant`, `InsertPosition`, `TilingLayout::regions(area)`.

## Changes

### 1. `crates/term-wm-core/src/window/window_manager/drag.rs` — `tile_window_key()`

Replace `insert_window_balanced` with the spatial pipeline. Before clearing `floating_rect`, capture the center. Regions from `layout.regions(self.managed_area)`. Uses the two-phase target resolution.

If `floating_rect` is `None` (impossible for a genuinely floating window, but defensive), call `insert_window_balanced` as before.

```rust
fn find_closest_region(
    (cx, cy): (i32, i32),
    regions: &[(WindowKey, Rect)],
) -> (WindowKey, Rect) {
    regions.iter()
        .map(|(k, r)| {
            let (rcx, rcy) = r.center();
            let dx = cx.saturating_sub(rcx);
            let dy = cy.saturating_sub(rcy);
            let dist = dx.saturating_mul(dx) + dy.saturating_mul(dy);
            (*k, *r, dist)
        })
        .min_by_key(|(_, _, d)| *d)
        .map(|(k, r, _)| (k, r))
        .unwrap()
}

fn resolve_target(
    (cx, cy): (i32, i32),
    regions: &[(WindowKey, Rect)],
) -> (WindowKey, Rect) {
    // Phase 1: exact hit-test
    if let Some(found) = regions.iter().find(|(_, r)| r.contains(cx as u16, cy as u16)) {
        return *found;
    }
    // Phase 2: Euclidean closest
    find_closest_region((cx, cy), regions)
}
```

### 2. `crates/term-wm-core/src/window/window_manager/layout.rs` — `reattach_to_tiling_layout()`

Replace the `current_focus` + `InsertPosition::Right` with the spatial pipeline. Only fall back to `current_focus` when `floating_rect` is `None`.

### 3. Helper placement

`quadrant_to_insert_position`, `find_closest_region`, and `resolve_target` — place in `layout.rs` (since `reattach_to_tiling_layout` lives there) and re-use from `drag.rs`, or vice versa. Pick `layout.rs` as it's the more foundational module.
