The notes strongly suggest using a highly concurrent, **strictly decoupled thread model** that completely isolates background I/O processing from the main UI rendering loop. This ensures that heavy terminal output (like compiling a large codebase or reading massive files) does not freeze the user interface or cause memory bloat.

Here are the core components of the recommended thread model:

*   **Dedicated Background PTY Threads:** Instead of handling everything on a single thread, the architecture dictates using dedicated background threads (or an asynchronous runtime pool) to continuously monitor the pseudoterminals (PTYs). These threads should utilize efficient OS-level primitives like `epoll` (Linux) or `kqueue` (macOS) to wait for data.
*   **The Main Event Loop (Screen Thread):** A separate foreground thread is responsible strictly for processing user input, managing the window state machine, calculating damage tracking, and rendering the UI frames.
*   **Hybrid Thread Communication (The Bridge):** To safely communicate between the background PTY threads and the Screen thread, the architecture avoids pushing all data through standard channels:
    *   **Control Events:** System events (like window resizes or exits) are passed through **bounded MPSC channels**. If the UI thread falls behind, the bounded channel fills up, which natively blocks the PTY thread. This exerts mechanical backpressure on the underlying child process, preventing out-of-memory crashes.
    *   **Text Streaming:** Heavy text data should *never* be sent through channels, as copying megabytes of text across threads destroys memory bandwidth. Instead, the background PTY thread should lock the window's text grid via a `Mutex`, parse the text directly into the grid, toggle a shared `AtomicBool` dirty flag, and then send a lightweight, zero-payload "Wakeup" ping to the main event loop so it knows to redraw.
*   **Advanced Thread Segregation (The Ghostty Model):** For extreme performance and power efficiency, the research highlights the architecture used by the Ghostty emulator. It allocates dedicated read, write, and render threads for *every* active terminal session. Furthermore, it uses OS-level thread scheduling to actively route non-focused or obscured background rendering threads to high-efficiency CPU cores (E-cores) to save battery life. 
*   **Plugin Threads:** [NOT CURRENTLY APPLICABLE; NO PLUGIN SUPPORT AT THE MOMENT] In highly complex multiplexers (like Zellij), the thread model may also branch out to include dedicated threads for safely executing WebAssembly (WASM) user plugins asynchronously.
