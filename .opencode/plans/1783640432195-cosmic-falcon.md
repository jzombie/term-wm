# Notification (Toast) System — Amended Plan

## Architecture

- **Pure data**: `NotificationQueue` lives in `term-wm-core` as a data-only struct (no Ratatui dependency).
- **Platform rendering**: Toast drawing lives in `term-wm-console::draw_plan_renderer` where `RatatuiBackend` is available — no downcasting across crate boundaries.
- **No Component/Overlay trait**: Notifications are simpler, display-only elements.
- **Timeout via `SystemTask::DismissNotification(id, Instant)`** — matches existing `DragSnap` pattern.
- **Read-only render pass**: `render()` takes `&self`; state eviction happens in a separate `tick()`.

### 1. New: `crates/term-wm-core/src/notification.rs`

**Data types (NO rendering logic, no Ratatui import):**

```rust
#[derive(Debug, Clone)]
pub struct Notification {
    pub id: u64,
    pub message: String,
    pub expires_at: Instant,   // when the SystemTask fires
}

/// Owns the notification state. Pure data — no rendering.
#[derive(Debug, Default)]
pub struct NotificationQueue {
    notifications: VecDeque<Notification>,
    next_id: u64,
    max_capacity: usize,        // default 5
}

impl NotificationQueue {
    pub fn push(&mut self, message: String, ttl: Duration) -> u64;
    pub fn dismiss(&mut self, id: u64) -> bool;
    /// Purge expired/dismissed notifications. Called BEFORE render.
    pub fn tick(&mut self) -> Vec<u64>;
    /// Read-only iteration (newest first) for the render pass.
    pub fn renderable(&self) -> impl Iterator<Item = &Notification>;
    pub fn len(&self) -> usize;
}
```

`push` evicts the oldest notification when at max capacity.

### 5. New rendering function: `crates/term-wm-console/src/draw_plan_renderer.rs`

Add `render_notifications` — this file already has `RatatuiBackend` and uses downcasting internally:

```rust
pub fn render_notifications(
    backend: &mut dyn term_wm_render::RenderBackend,
    notifications: &NotificationQueue,
    area: LayoutRect,
) {
    let backend: &mut RatatuiBackend = /* existing downcast helper */;
    let full = layout_rect_to_rect(area);
    let buffer = &mut backend.buffer;

    const TOAST_WIDTH: u16 = 40;
    const MARGIN: u16 = 2;
    const GAP: u16 = 1;

    let mut y = MARGIN;

    for notification in notifications.renderable() {
        // --- Accurate height calculation via unicode-width ---
        let inner_w = (TOAST_WIDTH.saturating_sub(2)) as usize; // minus borders
        let total_lines: usize = notification
            .message
            .lines()
            .map(|line| {
                let w = unicode_width::UnicodeWidthStr::width(line);
                // ceil division: how many wrapped rows this line needs
                w.saturating_add(inner_w).saturating_sub(1) / inner_w.max(1)
            })
            .sum::<usize>()
            .max(1); // at least 1 line of content
        let h = (total_lines as u16).saturating_add(2) as u16; // + borders
        let h = h.min(full.height.saturating_sub(y.saturating_add(MARGIN)));
        if h < 3 { break; }

        let x = full.width.saturating_sub(TOAST_WIDTH).saturating_sub(MARGIN);
        let rect = ratatui::layout::Rect { x, y, width: TOAST_WIDTH, height: h };

        // --- Render the toast ---
        Clear.render(rect, buffer);
        let (fg, bg) = notification_style(notification); // returns ratatui::style::Color
        let block = Block::default()
            .borders(Borders::ALL)
            .border_style(Style::default().fg(fg))
            .style(Style::default().bg(bg).fg(fg));
        let paragraph = Paragraph::new(notification.message.as_str())
            .block(block)
            .wrap(Wrap { trim: true });
        paragraph.render(rect, buffer);

        y = y.saturating_add(h).saturating_add(GAP);
    }
}

fn notification_style(_n: &Notification) -> (ratatui::style::Color, ratatui::style::Color) {
    // Use theme colors: dark bg, bright fg
    (Color::White, Color::DarkGray)
}
```

All arithmetic uses `saturating_*` per AGENTS.md.

### 6. Modify: `src/lib.rs` (render_app)

Add a **read-only tick + render** pass. The tick happens BEFORE the render phase starts, and the render happens after overlays:

```rust
// Before rendering: prune expired notifications (state mutation)
wm.dismiss_expired_notifications();

// In the rendering pipeline, after render_overlays:
render_notifications(backend, wm.notifications(), area);
```

This follows the strict phase separation: state mutation → rendering.

## Edge Cases & Verification

| Flaw from Review | Fix in This Plan |
|---|---|
| Downcasting across crates | Rendering lives in `term-wm-console` where `RatatuiBackend` is native |
| State mutation in render | `tick()` called pre-render for expiry; `render()` takes `&self` |
| Missing TTL params | `SystemTask::DismissNotification(id, Instant)` carries the deadline |
| Text wrapping desync | `unicode_width::UnicodeWidthStr::width` for measurement (same library Ratatui uses internally) |
| DrawPlan bypass | `render_notifications` called as the penultimate layer in the existing pipeline (after `render_overlays`) |
