# Fix: Per-window hitbox registration + cached screen_area

## Root Causes

1. **Z-order fragmentation**: Content hitboxes (`HitTarget::Window`) registered in Loop 1, chrome hitboxes (`ChromeResize`, `ChromeHeader`) in Loop 2. ALL chrome entries float above ALL content entries in reverse scan. A background window's chrome can intercept clicks on a foreground window's content.

2. **Coordinate drift**: Capture routing re-queries the live hitbox registry for `screen_area`, but the registry changes each frame.

## Fix

### 1. `src/lib.rs` — Per-window hitbox registration

Move chrome hitbox registration into the main window loop, interleaved with content registration. After rendering each window and registering its content hitbox, immediately register its chrome handles.

Current structure (broken):
```
Loop 1: for window in plan_regions:
    render window
    register content hitbox

Loop 2: for window in plan_regions:
    register chrome handles
```

New structure:
```
for window in plan_regions:
    render window
    register content hitbox
    register chrome handles
```

This ensures a foreground window's chrome is registered AFTER its content but BEFORE the background window's content — correct Z-ordering.

Tiling split handles stay AFTER the window loop (global high-priority for layout boundaries).

### 2. `window_manager/mod.rs` — Cached screen_area in capture state

Add `screen_area: LayoutRect` to `ComponentInteraction`, store `hit_rect` at Press time, use it during capture routing instead of querying the registry.
