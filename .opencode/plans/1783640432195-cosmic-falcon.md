# Fix: Use local area for selection overlay rendering

## Root Cause

`TextRendererComponent::render` passes `ctx.screen_area()` (ABSOLUTE screen coordinates) to `render_selection_overlay`. The buffer is a scratch buffer in LOCAL coordinates (from `composite_window`). For floating windows, `screen_area.x` can be > 100 while `buffer.area` starts at (0,0). The intersection produces an empty rect, breaking selection rendering.

Even for tiled windows, `screen_area` has chrome offsets (1,2) while `buffer.area` starts at (0,0). Using the local `area` parameter (which already matches the buffer's coordinate space) fixes both cases.

## Fix

In `crates/term-wm-ui-components/src/text_renderer.rs`, change `render_selection_overlay` call to use the local `area` parameter, not `ctx.screen_area()`:

```rust
// OLD: uses absolute screen coordinates from context
let screen_area = ctx.screen_area().map(layout_rect_to_rect).unwrap_or(area);
// ...
self.render_selection_overlay(&mut backend.buffer, screen_area, &ctx.config().theme);

// NEW: uses local area that matches buffer coordinate space
self.render_selection_overlay(&mut backend.buffer, area, &ctx.config().theme);
```

Also remove the now-unused `screen_area` variable if it's only used for this call.
