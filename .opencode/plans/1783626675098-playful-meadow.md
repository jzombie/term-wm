# Fix: PowerSaver Profile UI Update Bug

## Bug Description

When the PowerSaver profile is active (after 500ms of inactivity), PTY output data may not trigger UI updates reliably. The root cause is that `dirty_windows` in `UnifiedEventSource` are never consumed in production code — `take_dirty_windows()` is defined but only called in tests. This leads to two failure modes:

1. **Path A (wasteful 60fps loop)**: A PtyWakeup arrives, `poll()` arms the frame pacer, returns `Ok(false)` via `try_expire()`, but dirty_windows are NOT cleared. The profile stays at Streaming (16ms poll), causing 60fps rendering forever even when idle.

2. **Path B (missed updates)**: Multiple PtyWakeups arrive in quick succession, shrinking `remaining` below the frame deadline. `recv_timeout` fires before the deadline → `try_expire()` returns false → `break` → **`dirty_windows.clear()`** on line 358 of `unified_event_source.rs`. The poll returns `Ok(false)`, a render fires, but the next poll cycle sees empty dirty_windows → profile drops to PowerSaver (3600s timeout). If more PTY data arrives, it won't be rendered for up to 3600s.

## Fix

Add `take_dirty_windows()` to the `EventSource` trait and call it from the runner after each idle render, so dirty_windows are properly consumed and the profile can correctly calibrate between Streaming and PowerSaver.

### Files to Modify

1. **`crates/term-wm-core/src/io/event_source.rs`** — Add `take_dirty_windows()` method to trait with default empty return.

2. **`src/unified_event_source.rs`** — Promote the inherent `take_dirty_windows()` to implement the trait method. Also remove the `dirty_windows.clear()` on line 358 (the fallthrough path) since consumption is now the runner's responsibility.

3. **`crates/term-wm-core/src/runner.rs`** — Call `driver.take_dirty_windows()` in the idle handler AFTER `output.draw(...)` and BEFORE `flush_state_changes(app, ControlFlow::Continue)`. This ensures dirty windows are consumed after rendering, allowing `current_profile()` in `flush_state_changes` to see the correct state.

### Design Rationale

- After each render, the displayed terminal state is up-to-date for all windows that had pending PTY data. Consuming dirty_windows after render signals "we've shown these updates."
- If new PTY data arrives DURING the render, a new PtyWakeup is sent to the channel. The next `poll()` drains it via `drain_pending()`, sets dirty_windows, arms the frame pacer, and triggers a render within 16ms. No data is lost.
- The default trait implementation returns an empty set, so other `EventSource` impls (ConsoleEventSource, test stubs) are unaffected.
