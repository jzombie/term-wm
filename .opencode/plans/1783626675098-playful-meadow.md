# Fix: PowerSaver Profile UI Update Bug

## Bug Description

The profile is stuck at Streaming and never reaches PowerSaver. Two interacting root causes:

### Root Cause A: Dirty-windows never consumed (`runner.rs` + `unified_event_source.rs`)

`dirty_windows` in `UnifiedEventSource` are never consumed in production code — `take_dirty_windows()` is defined but only called in tests. This means the set grows without bound and `current_profile()` always sees `has_dirty_windows == true`, preventing PowerSaver.

### Root Cause B: Logging feedback loop / Heisenbug (`power_profile.rs`)

Even after fixing Root Cause A, the debug log creates a circular dependency:

1. Profile transitions to PowerSaver
2. `report_change()` logs `tracing::info!("profile: PowerSaver")`
3. Log subscriber writes to debug log component, dirtying its `WindowKey`
4. Next `poll()` finds the dirty log window → profile back to Streaming
5. Step 1 fires again → infinite 60fps loop

The subscriber ceiling is `Level::DEBUG` (`tracing_sub.rs:94`), so `info!` always passes through. Downgrading to `trace!` breaks the cycle.

## Fix

Add `take_dirty_windows()` to the `EventSource` trait and call it from the runner after each successful idle render, so dirty_windows are properly consumed and the profile can correctly calibrate between Streaming and PowerSaver.

### Files to Modify

1. **`crates/term-wm-core/src/io/event_source.rs`**
   - Add `take_dirty_windows()` to the `EventSource` trait with default empty return.
   - **Must use fully qualified paths** (`std::collections::HashSet`, `crate::window::WindowKey`) since these types are not in scope in this file.
   - **CRITICAL (SEV-1 fix):** Add explicit forwarding in the blanket impl `impl<T: EventSource + ?Sized> EventSource for &mut T`:
     ```rust
     fn take_dirty_windows(&mut self) -> std::collections::HashSet<crate::window::WindowKey> {
         (**self).take_dirty_windows()
     }
     ```
     Without this, calls through `&mut D` silently resolve to the default no-op.

2. **`src/unified_event_source.rs`**
   - Move the inherent `take_dirty_windows()` into the `impl EventSource for UnifiedEventSource` block.
   - Drop the `pub` keyword since it now fulfills a trait contract. Signature uses `std::collections::HashSet<WindowKey>` (both types in scope).
   - Remove `self.dirty_windows.clear()` on line 358 (the fallthrough path) since consumption is now the runner's responsibility.

3. **`crates/term-wm-core/src/runner.rs`**
   - **CRITICAL: Swap order in `flush_state_changes`** — consume dirty windows BEFORE checking the profile, not after. If the profile check happens first, `current_profile()` always sees non-empty dirty_windows (from the just-completed render cycle) and the profile never drops to PowerSaver.
   - **CRITICAL: Only consume dirty in idle path** — use `event.is_none() && !did_panic` as the consume flag. The event path (keyboard/mouse) does not render, so it must not consume dirty_windows.
     ```rust
     let mut flush_state_changes = |app: &mut A, flow: ControlFlow, consume_dirty: bool| {
         // ... existing mouse_capture, clipboard, selection checks ...
         
         // Consume BEFORE profile check so current_profile() sees post-render state
         if consume_dirty {
             driver.take_dirty_windows();
         }
         if let Some(profile) = profile_tracker.poll(driver.current_profile()) {
             app.wm().set_power_profile(profile);
         }
         Ok(flow)
     };
     ```
   - The final call site becomes:
     ```rust
     flush_state_changes(app, ControlFlow::Continue, event.is_none() && !did_panic)
     ```

4. **`crates/term-wm-core/src/power_profile.rs`** (NEW — Heisenbug fix)
   - Change `tracing::info!` to `tracing::trace!` in `report_change()` (line 131).
   - The tracing subscriber ceiling is `Level::DEBUG` (`tracing_sub.rs:94`), so `trace!` is discarded before reaching the subscriber. This breaks the **logging feedback loop** where:
     1. Profile changes to PowerSaver → `info!` logged
     2. Debug log component catches the event, dirties its window
     3. Next `poll()` finds the dirty log window → profile back to Streaming
     4. Step 1 fires again → infinite 60fps loop

### Design Rationale

- After each render, the displayed terminal state is up-to-date for all windows that had pending PTY data. Consuming dirty_windows after render signals "we've shown these updates."
- If new PTY data arrives DURING the render, a new PtyWakeup is sent to the channel. The next `poll()` drains it via `drain_pending()`, sets dirty_windows, arms the frame pacer, and triggers a render within 16ms. No data is lost.
- If the render panics, dirty_windows are preserved so the next successful render will pick up the missed data.
- The default trait implementation returns an empty set, so other `EventSource` impls (ConsoleEventSource, test stubs) are unaffected.
- **Heisenbug fix**: `tracing::trace!` is discarded by the subscriber (ceiling at `DEBUG`), so profile change logs no longer trigger dirty-window updates on the debug log component. This breaks the observer-effect feedback loop.
