Architectural Redesign Blueprint for term-wm: Eradicating API Friction, Double-Trait Coupling, and Lifecycle Circularity
Architectural Critique of the Legacy Paradigm
The architectural design of the legacy term-wm framework 1 exhibits several structural anti-patterns that create severe borrow-checker friction, force manual state synchronization, and increase the cognitive load for systems developers. These friction points stem from high-level, imperative orchestrators, double-trait coupling, and an inverted lifetime and ownership model.






+------------------------------------------------------------+
|                       Application                          |
+------------------------------------------------------------+
      |                                              ^
      | (Forced Dual-Trait Implementation)           |
      v                                              |
+-------------------------+                           |
|   WindowManagerHost     |                           |
+-------------------------+                           |
      |                                              |
      v                                              |
+-------------------------+                           |
|     WindowProvider      |                           |
+-------------------------+                           |
      |                                              |
      v                                              | (Manual Mapping)
+-------------------------------------------------+   |
| window_component(key)                           |   |
|   - Forces manual match { Key => &mut Component } --+
+-------------------------------------------------+

Double-Trait Coupling and the Manual Routing Anti-Pattern
The legacy framework enforces a strict dual-trait implementation, requiring a developer to implement both WindowManagerHost<Id> 3 and WindowProvider<Id> 4 on a central application state manager (such as App or AppState). The WindowProvider trait dictates that the host must expose two critical methods: enumerate_windows 4 to return a list of active handles, and window_component 4 to return a mutable reference to a trait object representing the window component.
This coupling introduces a central bottleneck:






Rust
fn window_component(&mut self, key: WindowKey) -> Option<&mut dyn Component> {
   if key == self.trials_key { Some(&mut self.trials_scroll_view) }
   else if key == self.charts_key { Some(&mut self.charts_scroll_view) }
   //... manual branching scales O(N) with the number of windows
}

This pattern forces developers to write tedious, boilerplate-heavy index-based routing blocks. It turns a central orchestrator into a manual dispatcher, breaking encapsulation. Components cannot be treated as self-contained, modular elements because their ownership remains bound to the root application struct while their routing is delegated to key-based matching on every tick.
Borrow Checker Friction and State Synchronization
Because the central application state owns both the data (such as database states or background caches) and the UI components, and must also yield mutable references to those components to the window manager via window_component 4, the Rust borrow checker enforces strict aliasing rules. A developer cannot easily pass a mutable reference of the application context down to a component if the window manager is already holding a mutable borrow of that component via the same application state.
To circumvent this, developers must resort to manual data synchronization, cloning state on every frame tick:






Rust
let charts_trials = self.trials.clone();
self.charts_scroll_view.content.trials = charts_trials;

This dynamic results in high CPU overhead and unnecessary allocations, which directly degrades performance targets tested by term-bench.2 It also breaks the single-source-of-truth principle, leading to potential state-desynchronization bugs where components render stale data.
The Type Erasure Trap and Lifecycle Circularity
To support polymorphic window types, components are boxed and type-erased as Box<dyn Component>. However, components that manage long-running tasks—such as pseudo-terminal (PTY) engines 4 or network streams—need to know their own assigned WindowKey to communicate back to the window manager (such as requesting redraws, resizing, or initiating a graceful shutdown sequence).
This requirement creates a circular dependency:
1. The component must be created and boxed to be registered.
2. The window manager registers the component and generates a unique WindowKey for its internal mapping.
3. The component needs this WindowKey to construct its callback closures before being boxed.
To solve this lifecycle circularity, the legacy API forces developers to use atomic interior mutability hacks:






Rust
let key_holder = Arc::new(Mutex::new(None::<WindowKey>));
//... build callback utilizing key_holder inside box...
let key = self.windows.create_window(Box::new(sv));
*key_holder.lock().unwrap() = Some(key);

This approach introduces runtime coordination overhead (locking a Mutex inside critical rendering paths), leaks memory structures through shared ownership (Arc), and makes asynchronous operations difficult to reason about due to temporal initialization gaps where the key_holder is accessed before the lock is populated.
Embedded vs. Standalone Fragmentation
term-wm serves two distinct modalities: Standalone Mode (a fully self-contained window-managed terminal application) and Embedded Mode (a modular layout manager inside a larger Ratatui application).2 In the legacy paradigm, these two modes share too much structural boilerplate.
A developer who simply wants to embed a dynamic grid of layout nodes into a larger canvas must configure custom layout trees, construct panel overrides, and manage window registration manually.2 This lack of progressive disclosure limits the utility of the library, as there is no easy way to scale from a single drop-in layout widget to a highly customized, multi-layered window management system.
The Target API Design Blueprint
To address these architectural issues, this blueprint introduces an ergonomic, idiomatic, and highly extensible Rust API for term-wm. This design removes trait-coupling, utilizes the StatefulWidget pattern 6, solves lifecycle circularity through deferred context-passing, and supports declarative tiling layouts.






+---------------------------------------------------------+
|                  Ratatui Render Loop                    |
+---------------------------------------------------------+
                           |
                           | renders stateful widget
                           v
+---------------------------------------------------------+
|                WindowManager (Widget)                   |
+---------------------------------------------------------+
                           |

                           v
+---------------------------------------------------------+
|             WindowManagerState (Registry)               |
|  - Holds SlotMap<WindowKey, Box<dyn Component>>         |
|  - Manages Focus and Layout Trees                       |
|  - Dispatches to self-contained components              |

A. Progressive Disclosure & Fluent Builders
To eliminate boilerplate in Standalone Mode, term-wm introduces a fluent builder API. This builder abstracts the complex initialization of the terminal backend, PTY event loops, panel rendering, and system chrome behind a declarative interface.2






Rust
use term_wm::prelude::*;

fn main() -> Result<(), Box<dyn std::error::Error>> {
   // Launch a complete, standalone multiplexer session with sensible defaults
   // in under 10 lines of code.
   StandaloneApp::new("term-wm-session")
      .with_chrome(WmConfig::standalone()) // Enables top panels, overlays, and status bars 
      .with_default_terminal()              // Spawns a default shell PTY pane
      .with_command_palette()              // Registers the global search & actions overlay 
      .with_power_profile(PowerProfile::Adaptive) // Enables power-aware polling rates 
      .run()?;                             // Runs the PTY-aware event loop blocking call

   Ok(())
}

This API progressively discloses complexity. A developer can start with the simple code block above and gradually customize its behavior by appending configuration methods (such as adding custom menu overlays, overriding keymaps, or registering background event handlers) without having to rewrite the terminal loop.
B. Eradicating the Dual-Trait Manual Router Pattern
To eliminate the double-trait coupling 3, the redesigned architecture shifts the responsibility of window management into a standard Ratatui StatefulWidget.6 The WindowManager struct becomes a lightweight, stateless configuration blueprint, while all active state, focus tracking, and owned components reside inside WindowManagerState.






Rust
use ratatui::prelude::*;
use slotmap::SlotMap;

// Unique identifier for registered windows, using a type-safe SlotMap key
slotmap::new_key_type! { pub struct WindowKey; }

/// Lightweight blueprint for configuring window layout and style.
pub struct WindowManager {
   pub layout_spec: LayoutSpec,
   pub border_style: Style,
}

/// The state container holding owned components, avoiding manual trait-routing.
pub struct WindowManagerState {
   windows: SlotMap<WindowKey, Box<dyn Component>>,
   layout_tree: LayoutTree,
   active_focus: Option<WindowKey>,
}

impl WindowManagerState {
   pub fn new() -> Self {
       Self {
           windows: SlotMap::with_key(),
           layout_tree: LayoutTree::new(),
           active_focus: None,
       }
   }

   /// Registers a component directly into the manager, returning its key.
   pub fn register<C>(&mut self, component: C) -> WindowKey 
   where
       C: Component + 'static,
   {
       self.windows.insert(Box::new(component))
   }
}

// Implement standard Ratatui StatefulWidget to allow easy integration inside external loops 
impl StatefulWidget for WindowManager {
   type State = WindowManagerState;

   fn render(self, area: Rect, buf: &mut Buffer, state: &mut Self::State) {
       // Resolve dynamic tiling layout bounds
       let layouts = state.layout_tree.compute_rects(area, &state.windows);

       for (key, window_area) in layouts {
           if let Some(component) = state.windows.get_mut(key) {
               // Construct a temporary context for safe, read-only app state sharing
               let ctx = ComponentContext::new(key, state.active_focus == Some(key));
               
               // Draw decorative window frame / chrome if specified
               let inner_area = self.render_window_chrome(buf, window_area, &ctx);
               
               // Direct polymorphic rendering without manual index matching
               component.render(inner_area, buf, &ctx);
           }
       }
   }
}

This pattern turns the window manager into a standard, composable widget.6 Developers can render WindowManager inside any arbitrary sub-layout or nested panel of a parent Ratatui application, resolving the embedded vs. standalone fragmentation.
C. Unifying and Streamlining Component Lifecycles
To resolve the Type Erasure Trap and lifecycle circularity without atomic locks, term-wm introduces a deferred context hook system. This pattern passes context to components when performing operations, rather than storing stateful identifiers inside the components themselves.






Rust
use std::sync::mpsc::Sender;

/// Event channel wrapper to safely communicate back to the central event loop.
#[derive(Clone)]
pub struct AppContext {
   event_tx: Sender<AppEvent>,
}

impl AppContext {
   pub fn trigger_redraw(&self) {
       let _ = self.event_tx.send(AppEvent::Redraw);
   }

   pub fn close_window(&self, key: WindowKey) {
       let _ = self.event_tx.send(AppEvent::CloseWindow(key));
   }
}

/// Transient execution context passed down during event handling and rendering.
pub struct ComponentContext<'a> {
   pub key: WindowKey,
   pub is_focused: bool,
   pub app: &'a AppContext,
}

impl<'a> ComponentContext<'a> {
   pub fn new(key: WindowKey, is_focused: bool, app: &'a AppContext) -> Self {
       Self { key, is_focused, app }
   }
}

/// Unified UI component trait.
pub trait Component {
   /// Optional hook invoked immediately after the component is registered.
   /// This resolves the lifecycle circularity by providing the assigned key and context
   /// before any frame rendering or event polling occurs.
   fn on_mount(&mut self, _key: WindowKey, _app: &AppContext) {}

   /// Polished handle_event function utilizing non-allocating actions.
   fn handle_event(&mut self, event: &crossterm::event::Event, ctx: &ComponentContext) -> EventHandled;

   /// Standard render hook containing contextual awareness.
   fn render(&mut self, area: Rect, buf: &mut Buffer, ctx: &ComponentContext);
}

#
pub enum EventHandled {
   Consumed,
   Ignored,
}

By decoupling registration from execution, the window manager can allocate a type-safe WindowKey from its SlotMap during registration, instantiate a temporary execution context ComponentContext, and invoke on_mount on the newly registered window immediately:






Rust
pub fn spawn_window<C>(&mut self, mut component: C, app: &AppContext) -> WindowKey
where
   C: Component + 'static,
{
   let key = self.windows.insert_with_key(|k| {
       component.on_mount(k, app);
       Box::new(component)
   });
   self.layout_tree.add_node(key);
   key
}

This pattern completely avoids reference cycles, thread-unsafe interior mutability, and temporal initialization gaps.
D. Layout Trees & Progressive Complexity
To replace the verbose, imperative layout building (LayoutNode::split(...)), term-wm introduces a declarative layout specification. This system supports both cooperative tiling heuristics (such as Binary Space Partitioning or dynamic column tiling) and explicit constraint overrides.






Rust
/// Declarative layout strategies that adapt automatically to new window allocations.
#
pub enum LayoutSpec {
   /// Dynamically splits space vertically or horizontally based on aspect ratio (e.g., BSPWM).
   BinarySpacePartition,
   /// Traditional stack-and-master layout (e.g., DWM).
   StackAndMaster { master_ratio: f32, master_count: u32 },
   /// Completely unconstrained manual layouts with layering (z-ordering).
   Floating,
   /// Absolute percentage or constraint-driven split grids.
   CustomGrid(Vec<Constraint>),
}

/// Dynamic Layout Tree that implements layout heuristics.
pub struct LayoutTree {
   spec: LayoutSpec,
   nodes: Vec<WindowKey>,
}

impl LayoutTree {
   pub fn new(spec: LayoutSpec) -> Self {
       Self { spec, nodes: Vec::new() }
   }

   pub fn add_node(&mut self, key: WindowKey) {
       self.nodes.push(key);
   }

   pub fn remove_node(&mut self, key: WindowKey) {
       self.nodes.retain(|&k| k!= key);
   }

   /// Computes terminal cell boundary rectangles for each active window node.
   pub fn compute_rects(&self, total_area: Rect, active_windows: &SlotMap<WindowKey, Box<dyn Component>>) -> Vec<(WindowKey, Rect)> {
       if self.nodes.is_empty() {
           return Vec::new();
       }

       match self.spec {
           LayoutSpec::BinarySpacePartition => self.compute_bsp_layout(total_area),
           LayoutSpec::StackAndMaster { master_ratio, master_count } => {
               self.compute_stack_master(total_area, master_ratio, master_count)
           }
           LayoutSpec::Floating => {
               // Fall back to stored custom sizes or cascade default coordinates
               self.compute_floating_layout(total_area, active_windows)
           }
           LayoutSpec::CustomGrid(ref constraints) => {
               self.compute_custom_grid(total_area, constraints)
           }
       }
   }

   fn compute_bsp_layout(&self, area: Rect) -> Vec<(WindowKey, Rect)> {
       let mut results = Vec::new();
       if self.nodes.is_empty() { return results; }

       // Recursive helper to split partitions dynamically based on cell dimensions
       fn split_recursive(nodes: &, target: Rect, results: &mut Vec<(WindowKey, Rect)>) {
           match nodes {
               => {}
               [single] => {
                   results.push((*single, target));
               }
               _ => {
                   let mid = nodes.len() / 2;
                   // Determine split direction: split horizontally if height is large, vertically otherwise
                   let split_horizontal = target.height > (target.width / 2);
                   let sub_rects = if split_horizontal {
                       let half_height = target.height / 2;
                       (
                           Rect::new(target.x, target.y, target.width, half_height),
                           Rect::new(target.x, target.y + half_height, target.width, target.height - half_height)
                       )
                   } else {
                       let half_width = target.width / 2;
                       (
                           Rect::new(target.x, target.y, half_width, target.height),
                           Rect::new(target.x + half_width, target.y, target.width - half_width, target.height)
                       )
                   };
                   split_recursive(&nodes[..mid], sub_rects.0, results);
                   split_recursive(&nodes[mid..], sub_rects.1, results);
               }
           }
       }

       split_recursive(&self.nodes, area, &mut results);
       results
   }
   
   // Stub implementations for compilation validation
   fn compute_stack_master(&self, _area: Rect, _ratio: f32, _count: u32) -> Vec<(WindowKey, Rect)> { Vec::new() }
   fn compute_floating_layout(&self, _area: Rect, _windows: &SlotMap<WindowKey, Box<dyn Component>>) -> Vec<(WindowKey, Rect)> { Vec::new() }
   fn compute_custom_grid(&self, _area: Rect, _constraints: &[Constraint]) -> Vec<(WindowKey, Rect)> { Vec::new() }
}

E. Handling Third-Party Widgets and Component Control
To integrate third-party Ratatui widgets 6, the redesign provides a generic adapter structure. This adapter wraps any standard Widget or StatefulWidget and implements our Component trait, ensuring it can be used natively inside our window framework.






Rust
/// Generic adapter to transform standard Ratatui widgets into managed term-wm components.
pub struct WidgetAdapter<W, S> 
where
   W: StatefulWidget,
{
   widget_factory: Box<dyn Fn() -> W>,
   state: S,
}

impl<W, S> WidgetAdapter<W, S>
where
   W: StatefulWidget + 'static,
   S: 'static,
{
   pub fn new<F>(factory: F, initial_state: S) -> Self 
   where
       F: Fn() -> W + 'static,
   {
       Self {
           widget_factory: Box::new(factory),
           state: initial_state,
       }
   }

   /// Exposes read-only access to inner widget state
   pub fn state(&self) -> &S {
       &self.state
   }

   /// Exposes mutable access to inner widget state
   pub fn state_mut(&mut self) -> &mut S {
       &mut self.state
   }
}

impl<W, S> Component for WidgetAdapter<W, S>
where
   W: StatefulWidget + 'static,
   S: 'static,
{
   fn handle_event(&mut self, event: &crossterm::event::Event, _ctx: &ComponentContext) -> EventHandled {
       // Here, input events can be routed. For example, mouse events can be translated 
       // using relative coordinates, or keys can be passed to specific state mechanisms.
       match event {
           crossterm::event::Event::Key(_key_event) => {
               // If the wrapped widget handles events (using third-party extensions [9, 10]), 
               // dispatch them directly here.
               EventHandled::Ignored
           }
           _ => EventHandled::Ignored,
       }
   }

   fn render(&mut self, area: Rect, buf: &mut Buffer, _ctx: &ComponentContext) {
       let widget = (self.widget_factory)();
       // Native rendering onto the character grid buffer
       widget.render(area, buf, &mut self.state);
   }
}

	Host Coordination
	Hard coupled through manual implementations of WindowManagerHost and WindowProvider.3

	High allocations and copies due to constant state cloning, degrading the 120 FPS targets.2

	Input Event Routing and Coordinate Mapping Pipeline
To process mouse clicks and direct keyboard inputs according to the "No-Conflict Esc Super Key" philosophy 2, input routing uses a layered event-propagation pipeline:






+----------------------------------------------------------------+
|                   crossterm Mouse Event                        |
|                     (Absolute: X, Y)                           |
+----------------------------------------------------------------+
                               |
                               v
+----------------------------------------------------------------+
|             Hit Test against Computed Layout Rects             |
|   Matches absolute coordinate with relative component boundary  |
+----------------------------------------------------------------+
                               |
                               v
+----------------------------------------------------------------+
|                  Active Focus Selection                        |
|   - Updates active_focus key inside WindowManagerState         |
|   - Re-sorts Z-Index layering (floating)                       |
+----------------------------------------------------------------+
                               |
                               v
+----------------------------------------------------------------+
|               Relative Coordinate Offset Mapping               |
|   - Modifies Mouse coordinates to local window bounds (0, 0)   |
|   - Calls target component's handle_event(relative_event)      |
+----------------------------------------------------------------+

1. Calculated Pass Layout Generation: When rendering, StatefulWidget::render computes the layout tree coordinates and stores the calculated bounds map HashMap<WindowKey, Rect> in the WindowManagerState for that frame.
2. Absolute Hit Testing: When a mouse event occurs, the coordinates (mouse.column, mouse.row) are queried against the stored bounds map of the active frame.
3. Z-Order Evaluation: If multiple overlapping windows intersect the target coordinate, the hit tester evaluates candidates in reverse z-order (topmost floating window first) to find the correct recipient.
4. Coordinate Mapping and Offsets: The system translates absolute mouse coordinates into relative coordinates bounded by the component's rect (where the top-left cell of the component is mapped to (0, 0)), and passes the relative event down to the component's handle_event implementation.
5. Focus Invalidation: Clicking a component's chrome or interior region automatically updates the active_focus reference inside WindowManagerState, triggering a layout update to elevate the focused window in z-index order and adjust boundary borders.

}

pub struct ArgtunerDashboard {
   wm_state: WindowManagerState,
   app_context: AppContext,
   tuning_db: SharedTuningDb,
}

impl ArgtunerDashboard {
   pub fn new(db: TuningDatabase) -> Self {
       let mut wm_state = WindowManagerState::new();
       let db_shared = SharedTuningDb { inner: Arc::new(db) };

       // Pass the shared reference to components during instantiation.
       // Components can read from this database directly during their execution context,
       // eliminating the need to manually copy data on every frame tick.
       let trials = TrialsTableComponent::new(db_shared.clone());
       let charts = ChartsComponent::new(db_shared.clone());

       wm_state.register(trials);
       wm_state.register(charts);

       Self {
           wm_state,
           app_context: AppContext { event_tx: std::sync::mpsc::channel().0 },
           tuning_db: db_shared,
       }
   }

   pub fn draw(&mut self, frame: &mut Frame) {
       // Render the components using the standard StatefulWidget pattern.
       // This is clean, safe, and avoids all borrow-checker friction.
       frame.render_stateful_widget(
           WindowManager {
               layout_spec: LayoutSpec::StackAndMaster { master_ratio: 0.6, master_count: 1 },
               border_style: Style::default(),
           },
           frame.area(),
           &mut self.wm_state,
       );
   }
}

pub struct TrialsTableComponent {
   db: SharedTuningDb,
}

impl TrialsTableComponent {
   pub fn new(db: SharedTuningDb) -> Self {
       Self { db }
   }
}

impl Component for TrialsTableComponent {
   fn handle_event(&mut self, _event: &crossterm::event::Event, _ctx: &ComponentContext) -> EventHandled {
       EventHandled::Ignored
   }

   fn render(&mut self, area: Rect, buf: &mut Buffer, _ctx: &ComponentContext) {
       // Query the shared database directly during render passes.
       // No redundant clones or state synchronization steps are needed on ticks.
       let trials_data = self.db.inner.fetch_latest_runs();
       
       let mut y_offset = area.y;
       for trial in trials_data.iter().take(area.height as usize) {
           let info = format!("Trial ID: {} | Status: {}", trial.id, trial.status);
           buf.set_string(area.x, y_offset, &info, Style::default());
           y_offset += 1;
       }
   }
}

Architectural Conclusions
The redesigned architecture of term-wm addresses the core API friction points of the legacy codebase:
* Separation of Concerns: Replacing WindowManagerHost and WindowProvider 3 with a standard Ratatui StatefulWidget 6 simplifies the API and makes it easy to embed the window manager inside any external render loop.
* Streamlined Component Ownership: Moving component ownership into WindowManagerState and using a type-safe SlotMap simplifies event dispatching and rendering, eliminating the need for manual routing boilerplates.
* Safe Lifecycle Coordination: The deferred on_mount context hook resolves lifecycle circularity cleanly, allowing components to access their context and identifiers safely without resorting to interior mutability hacks.
* Ergonomic and Flexible Layouts: Declarative layout specifications (LayoutSpec) replace imperative split configurations with automatic, cooperative tiling heuristics.
* Ecosystem Compatibility: The WidgetAdapter wrapper enables native integration of any third-party Ratatui widget 6, bringing the benefits of the term-wm window system to the broader Ratatui ecosystem.
