# Workspace Split Plan: `term-wm` → `term-wm-core` + `term-wm` (binary)

## Overview

Split the monolith into a workspace with two crates:
- **`term-wm-core`** (library) — everything except `main.rs` and `tracing_sub.rs`
- **`term-wm`** (binary) — `main.rs` + `tracing_sub.rs`, depends on `term-wm-core`

## Step 2: Cargo.toml changes

**Critical changes:**
1. Add `"term-wm-core"` to workspace `members`
2. Remove all library dependencies — move them to `term-wm-core`
3. Add `term-wm-core` as a path dependency for the binary crate
4. Keep only deps used by `main.rs` + `tracing_sub.rs`

## Step 5: Critical refactoring — `tracing_sub::init_default()` call

### Problem

`window/window_manager/mod.rs:373` calls `crate::tracing_sub::init_default();` inside `WindowManager::with_config()`. This code moves to `term-wm-core`, but `tracing_sub` stays in the binary crate. The library crate cannot depend on the binary crate.

### Solution

1. **Remove** line 373 from `WindowManager::with_config()`:
   ```rust
   // BEFORE (window/window_manager/mod.rs:373):
   crate::tracing_sub::init_default();
   
   // AFTER:
   // (removed)
   ```

2. **Add** `init_default()` call in `src/main.rs` at the start of `main()` (before any `WindowManager` is created):
   ```rust
   fn main() -> io::Result<()> {
       tracing_sub::init_default();  // ← NEW: initialize tracing early
       // ... rest of main ...
   }
   ```

3. **Add** a simple tracing init in `term-wm-core/examples/dual_image.rs` since it no longer gets it implicitly:
   ```rust
   fn main() -> io::Result<()> {
       tracing_subscriber::fmt().init();  // ← NEW: simple stdout init
       // ... rest of main ...
   }
   ```
   (The example can use the `tracing-subscriber` crate as a dev-dependency in `term-wm-core` or inline a minimal init.)

## Step 6: Visibility changes

**Key insight:** Almost no visibility changes needed. All `pub(crate)` items stay within `term-wm-core` since only `main.rs` and `tracing_sub.rs` leave. The `pub(crate)` visibility still works correctly because all cross-references between modules remain in the same crate.

### Items reviewed and confirmed fine:

| File | `pub(crate)` items | Status |
|------|-------------------|--------|
| `component_context.rs` | `ViewportHandle.shared`, `ViewportSharedState.*`, `max_offset_x/y` | OK — accessed by `components/scroll_view.rs` and `components/text_renderer.rs`, both in same crate |
| `ui.rs` | `from_parts`, `safe_set_string`, `truncate_to_width` | OK — accessed by `panel.rs`, `confirm_overlay.rs`, `text_renderer.rs`, `selection_preview_overlay.rs`, all in same crate |
| `window/window_manager/mod.rs` | `WindowManager` fields (16 items) | OK — only accessed within `window/window_manager/*` submodules and `#[cfg(test)]` in same file, all in same crate |
| `io/mod.rs` | `pub(crate) mod utils` | OK — accessed by `theme.rs` via `crate::io::utils::term_color`, both in same crate |
| `io/utils/mod.rs` | `pub(crate) mod keyboard_normalizer`, `pub(crate) mod term_color` | OK — same as above |

### Only change needed:

Make `io/utils/mod.rs` modules **pub** instead of `pub(crate)` so that `theme.rs` can access them across module boundaries within `term-wm-core`:

Actually, no — this already works. `pub(crate)` means "visible to the current crate". After the split, `theme.rs` and `io` are both in `term-wm-core`, so `pub(crate)` still works.

**Conclusion: Zero visibility changes needed.** The split is purely structural with no API surface changes.

## Step 7: Root binary crate — does it need `lib.rs`? **NO**

Decision: **No `lib.rs` in the root binary crate.**

- The root crate has only `main.rs` (binary target) and `tracing_sub.rs`.
- Integration tests and examples move to `term-wm-core`.
- The binary crate doesn't need a library target.
- This avoids maintaining a thin re-export layer and keeps things simple.

Note: This means `term-wm` as a crate name can no longer be used as a library dependency by external code. Since the user's goal is to make `term-wm-core` the reusable library, this is correct.
