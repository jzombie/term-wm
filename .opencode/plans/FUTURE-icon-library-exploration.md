# FUTURE: Terminal Icon Library & Palette Viewer

## Goal
Build a separate icon library crate that catalogs **every icon/glyph renderable in any terminal** (no special fonts), and optionally display the palette as a grid viewer inside term-wm.

## Principles
- **Zero external font dependency** — no Nerd Fonts, no Font Awesome, no icon font files
- **Tiered fallback** — every icon has a representation at multiple compatibility levels
- **Standalone crate** — usable by term-wm and any other Rust TUI project
- **Cross-platform** — works on macOS, Linux, Windows terminals

## Phase 1: Cataloging

### 1a. Research Unicode Blocks
Identify and test Unicode blocks that reliably render across terminals (Terminal.app, iTerm2, Alacritty, Kitty, WezTerm, Windows Terminal, tmux, etc.):

| Tier | Range | Glyph Count | Compatibility |
|------|-------|-------------|---------------|
| ASCII | U+0020–U+007E | 95 | 100% of terminals |
| Latin-1 Supplement | U+00A0–U+00FF | 96 | ~99% |
| Box Drawing | U+2500–U+257F | 128 | ~99% |
| Block Elements | U+2580–U+259F | 32 | ~98% |
| Geometric Shapes | U+25A0–U+25FF | 96 | ~95% |
| Arrows | U+2190–U+21FF | 112 | ~95% |
| Dingbats | U+2700–U+27BF | 192 | ~90% |
| Misc Symbols | U+2600–U+26FF | 256 | ~90% |
| Misc Technical | U+2300–U+23FF | 256 | ~85% |
| Braille | U+2800–U+28FF | 256 | ~80% |
| CJK Symbols | U+2E80–U+2EFF | ~128 | variable |
| Emoticons | U+1F600–U+1F64F | 80 | ~80% |

### 1b. Build a Test Harness
- Write a small CLI binary (`icon-scout` or similar) that prints candidate glyphs to the terminal
- Renders a machine-readable grid: glyph + codepoint + hex + UTF-8 bytes
- Manual test across terminal emulators to establish real-world compatibility tiers

### 1c. Define Category Taxonomy
Organize icons into semantic categories:
- **WindowControls**: close, maximize, minimize, restore, pin, menu
- **Navigation**: arrow-up/down/left/right, chevrons, carets, breadcrumbs
- **Status**: check, cross, warning, info, error, dot, spinner chars
- **Media**: play, pause, stop, record, eject, fast-forward, rewind
- **Files**: folder, file, document, image, music, video, archive
- **Power**: battery, plug, power-on/off, sleep
- **Network**: wifi, signal, bluetooth, globe, link
- **UI**: resize-handle, scrollbar-thumb, grip, ellipsis, bullet
- **Misc**: star, heart, flag, clock, calendar, search, lock, key

## Phase 2: Library Crate Design

### 2a. Crate Structure
```
crates/term-wm-icon-library/
├── Cargo.toml          # standalone; optionally depends on term-wm-render for integration
├── src/
│   ├── lib.rs          # re-exports
│   ├── icon.rs         # Icon enum + IconTier + IconCategory
│   ├── registry.rs     # IconRegistry — indexed lookup by category/name/tier
│   ├── catalog.rs      # Full built-in catalog of all icons
│   ├── render.rs       # Render helpers (set_symbol, set_string, style)
│   └── tiers.rs        # Compatibility tier definitions
```

### 2b. Core Types
```rust
pub enum IconTier {
    Ascii,            // U+0020–U+007E — guaranteed everywhere
    Latin1,           // U+00A0–U+00FF
    CommonUnicode,    // U+2000–U+27BF — box drawing, geometric, dingbats  
    ExtendedUnicode,  // U+2300–U+26FF, U+2800–U+28FF — misc technical, braille
    ModernUnicode,    // U+1F300–U+1F9FF — emoji, symbols (variable support)
}

pub enum IconCategory {
    WindowControls,
    Navigation,
    Status,
    Media,
    Files,
    Power,
    Network,
    Ui,
    Misc,
}

pub struct Icon {
    pub name: &'static str,          // e.g., "close", "maximize"
    pub category: IconCategory,
    pub glyphs: &'static [(IconTier, &'static str)], // tier -> glyph mapping
    pub description: &'static str,   // human-readable
    pub codepoint: u32,              // canonical Unicode codepoint
}
```

### 2c. Fallback Logic
```rust
impl Icon {
    /// Pick the best glyph for a given terminal capability tier.
    pub fn select_glyph(&self, terminal_tier: IconTier) -> &'static str;
}
```

### 2d. Registration / Auto-generation
- Catalog should be generated by a build script (`build.rs`) or macro from a data file
- Data file format: TOML/YAML — one entry per icon with all tier glyphs
- Ensures the catalog is comprehensive and consistent

## Phase 5: CI Snapshot Testing — Cross-Platform Visual Verification

### 5a. Motivation
An icon that renders at 1 cell wide on macOS might render at 2 cells (or as a tofu `□`) on Linux or Windows. To catch this, we build a CI pipeline that renders the full icon catalog to PNG images on each platform, checks size consistency, and flags mismatches.

### 5b. Snapshot CLI Tool (`icon-snapshot`)
A small binary / library in the icon crate that converts a ratatui `Buffer` to a pixel PNG:

```
crates/term-wm-icon-library/src/
├── snapshot/
│   ├── mod.rs           # SnapshotConfig, render_to_image()
│   ├── rasterizer.rs    # fontdue-based per-cell glyph rasterizer
│   ├── compare.rs       # Pixel diff + size consistency check
│   └── cli.rs           # Binary entry point (optional)
```

**Core logic:**
1. Render the full icon catalog into a ratatui `Buffer` (existing headless pattern: `Buffer::empty(area)`)
2. For each non-empty cell, rasterize the character using `fontdue` + `image`:
   - Pick a representative monospace font (configurable path)
   - Render each glyph at the appropriate cell size (e.g. 8×16 px)
   - Composite into a single pixel grid
3. Save as PNG with labeled grid lines showing icon names and codepoints

**Platform-specific fonts to test:**
| Platform | Default Monospace Font | Font Path |
|----------|----------------------|-----------|
| macOS    | SF Mono             | `/System/Library/Fonts/SF-Mono-Regular.otf` |
| Linux    | DejaVu Sans Mono    | `/usr/share/fonts/truetype/dejavu/DejaVuSansMono.ttf` |
| Windows  | Consolas            | `C:\Windows\Fonts\consola.ttf` |

### 5c. Size Consistency Algorithm
For each icon glyph, measure its rendered pixel bounding box:

```
fn measure_glyph_bounds(font: &Font, glyph: char, px: f32) -> (u32, u32)
```

Then across platforms, compare:
- **Glyph width (px)**: Should match expected cell width (1 cell = `font.advance_width`)
- **Glyph height (px)**: Should match `font.height` 
- **Cross-platform delta**: `|width_A - width_B|` — any difference > 0 means the glyph renders differently

**Icon suitability scoring:**
```rust
pub struct IconCompatibility {
    pub icon_name: &'static str,
    pub tier: IconTier,
    pub widths: HashMap<Platform, u32>,  // rendered pixel widths
    pub heights: HashMap<Platform, u32>,
    pub is_consistent: bool,             // same width across all platforms
    pub renders_as_tofu: bool,           // renders as □ / missing glyph
}
```

### 5d. GitHub Actions Workflow
File: `.github/workflows/icon-snapshot.yml`

```yaml
jobs:
  snapshot:
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - run: cargo run -p term-wm-icon-library --bin icon-snapshot
      - uses: actions/upload-artifact@v4
        with:
          name: icon-snapshot-${{ matrix.os }}
          path: snapshots/*.png

  verify-consistency:
    needs: snapshot
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
      - run: cargo run -p term-wm-icon-library --bin icon-verify --snapshots ./
        # Fails with exit code 1 if any icon has inconsistent width
        # Outputs a machine-readable report of all mismatches
      - if: failure()
        run: cat inconsistency-report.md >> $GITHUB_STEP_SUMMARY
```

### 5e. CI Artifacts per Run
Each platform CI job produces:
- `catalog-grid-{os}.png` — Full icon grid with labels
- `catalog-grid-{os}-annotated.png` — Same but with bounding box overlays and size annotations
- `icon-glyphs-{os}/` — Individual per-icon glyph images for detailed inspection

The verification job then produces:
- `inconsistency-report.md` — Summary of icons with size mismatches, including rendered width per platform and screenshot references

### 5f. Failure & Triage Workflow
When CI fails on size inconsistency:
1. Developer downloads the per-platform PNG artifacts from the CI run
2. Reviews the inconsistency report to see which icons differ
3. If an icon renders at different widths:
   - **Downgrade its tier**: Move the problematic glyph to a lower `IconTier` where a narrower fallback exists
   - If no working fallback exists at any tier, the icon gets flagged as `cross-platform: false` in the catalog
4. The icon library exports `Icon::is_cross_platform()` so consumers can skip unreliable glyphs
