# Plan: Purge Duplicated Layout Math — Migrate Core to Use `term-wm-layout-engine`

## Background

The code review revealed **SEV-1** architectural violations:
1. **Floating-point math** — `LayoutNode::Split` stores `weights: Vec<f32>`; `split_rects_weighted` uses `floor()`/`round()` with f32 arithmetic, causing pixel drift in terminal grid.
2. **Complete boundary failure** — 5 private functions in `tiling.rs` duplicate the exact public API already exposed by `term-wm-layout-engine/src/split.rs`.

`Rect` is already a type alias for `LayoutRect` (same struct, `crates/term-wm-core/src/lib.rs:36`), so no geometry conversion is needed — only weight type and function call changes.

## Files to Modify

| File | Role |
|---|---|
| `crates/term-wm-core/src/layout/tiling.rs` | Primary target — purge 5 functions, change `Vec<f32>` → `Vec<u16>` |
| `crates/term-wm-core/src/layout/mod.rs` | Remove duplicated `gap_size` if it's redundant with engine's version |

## Important: Constraint-based Layout Handling

The deleted `split_rects` function had a fallback:
```rust
if constraints.len() == child_count {
    return Layout::default().direction(direction).constraints(constraints.to_vec()).split(area).to_vec();
}
```

The engine doesn't have constraint-based layout. The 3 call sites (`layout_recursive`, `void_regions_recursive`, `split_area_for_path`) all have `constraints` available. Each call site should check:

```rust
if constraints.len() == children.len() {
    // use constraint-based layout (call Layout::default()...)
    let rects = Layout::default().direction(*direction).constraints(constraints.clone()).split(area).to_vec();
    // For split_rects_with_gaps, also compute gap rects...
} else {
    // use engine split_rects_with_gaps
}
```

However, examining the call sites: `layout_recursive` at line 563 previously always called `split_rects_with_gaps`, which internally called `split_rects` which checked constraints. So we need to hoist the constraint check into each caller.

Actually, wait — `layout_recursive` is called from `layout_with_handles` which is called from places that presumably always have weights (since `Split` always stores `weights`). Let me re-examine:

The constraint fallback only activates when `weights.len() != child_count || !weights.iter().any(|v| *v > 0.0)` AND `constraints.len() == child_count`. Since every `Split` node always has non-zero weights (or empty weights that get filled to `vec![1.0; n]`), the constraint path is rarely hit — it's only for code that explicitly sets constraints without weights.

**Decision**: Hoist the constraint check to call sites. Keep the pattern simple — if constraints are given and match child count, use `Layout::default().direction(direction).constraints(constraints).split(area)`. Otherwise, use the engine.

Potential risks:
- Weight semantics change: f32 `1.0` represents "equal share", u16 `1` also represents "equal share" because the engine normalizes by `total_weight`. But the engine's `split_rects_weighted` uses integer division, which may produce different pixel distributions than f32 floor-division. Verify with the existing test `split_rects_weighted_even`.
- The engine's `split_rects_with_gaps` takes a pre-computed `gap` parameter; make sure all call sites compute gap correctly with `gap_size()` before calling.
