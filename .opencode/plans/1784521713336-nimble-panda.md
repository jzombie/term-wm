# Chrome Fragmentation Fix — Dumb Core Architecture

## Problem
Chrome metrics (border widths, header heights, button positions) are hardcoded in 4 locations across `term-wm-core` and `term-wm-console`, creating implicit coupling. A border thickness change requires synchronized edits across crate boundaries.

## Solution: Dumb Core
The core defines the **vocabulary** (`ChromeTarget`). The console owns ALL chrome math — it draws, registers hitboxes, and returns the inner content rect.

## Core Changes (`term-wm-core`)

### 1. `src/chrome/target.rs` — Only the Vocabulary
```rust
pub enum ChromeTarget {
    Resize(WindowKey, ResizeEdge),
    Drag(WindowKey),
    CloseButton(WindowKey),
    MaximizeButton(WindowKey),
    SplitHandle(HitboxId),
}
```
No metrics. No border widths. No button placements. Purely semantic.


## AMENDMENT 2: Fix Remaining Chrome Interaction Bugs

### Bug 1: Content hitbox registered over full window

`src/lib.rs` registers content hitbox with `surface.inner = full` covering chrome too. Clicks at (0,0) hit content instead of chrome.

**Fix**: Move content hitbox registration into `render_window_chrome` — console generates its own `HitboxId` via `registry.register()`. No `HitboxId` persistence in core.

`render_window_chrome` already takes a `registry: &mut HitboxRegistry` and knows the inner bounds. After registering chrome hitboxes, register:
```rust
let inner_id = HitboxId::new();
registry.set_active_owner(ComponentOwner::Window(key));
registry.register(inner_id, screen_space_inner_bounds);
```

| File | Change |
|------|--------|
| `crates/term-wm-console/src/draw_plan_renderer.rs` | Register content hitbox inside `render_window_chrome` using fresh `HitboxId::new()` + inner bounds |
| `src/lib.rs` | Remove the `set_active_owner`+`register` content hitbox block (no `window_content_hitbox_id` needed) |

### Bug 2: Resize handles don't light up

`hovered_resize_handle()` returns `None`. Must not add geometric checks to core (SEV-1).

**Fix**: Query `HitboxRegistry` directly. The console already registers `ChromeTarget::Resize(key, edge)` hitboxes; the core can check the registry at hover time.

```rust
pub fn hovered_resize_handle(&self) -> Option<(WindowKey, ResizeEdge)> {
    let (col, row) = self.hover?;
    let pos = MousePosition { column: col as i16, row: row as i16, space: CoordSpace::Screen };
    if let Some((_, ComponentOwner::Chrome(ChromeTarget::Resize(key, edge)), _)) =
        self.hitbox_registry.hit_test(pos)
    {
        Some((key, edge))
    } else {
        None
    }
}
```

Update callers expecting `&ResizeHandle<WindowKey>` to use `(WindowKey, ResizeEdge)`.

| File | Change |
|------|--------|
| `crates/term-wm-core/src/window/window_manager/mod.rs` | Replace no-op `hovered_resize_handle` with `HitboxRegistry` query |
| `src/lib.rs` | Update `render_resize_outline` call to use `(WindowKey, ResizeEdge)` signature |

### Bug 3: `ChromeTarget::SplitHandle` payload type

**Fix**: Change `SplitHandle(HitboxId)` → `SplitHandle(SplitHandleId)` (the layout engine's domain key).

| File | Change |

### Bug 4: Buttons beyond Close don't work

Fixed by Bug 1 (content hitbox no longer occludes button positions).

### Bug 5: Frame-phase ordering — hover queries must run after registration

`hovered_resize_handle()` queries the registry. The registry must be fully populated before this runs.

**Fix**: Ensure `render_window_chrome` completes for all windows before `hovered_resize_handle` is called. The current `render_app` loop already processes all windows before reaching `render_resize_outline`, so ordering is naturally correct as long as chrome hitbox registration happens inside `render_window_chrome`.

### Bug 6: Split handle identification — `SplitHandleId` not `HitboxId`

`ChromeTarget::SplitHandle(HitboxId)` prevents dispatch_mouse from resolving the layout handle. Fix: use the layout engine's `SplitHandleId` domain type.

**File**: `crates/term-wm-core/src/chrome/target.rs`
```rust

    Resize(WindowKey, ResizeEdge),
    Drag(WindowKey),
    CloseButton(WindowKey),
    MaximizeButton(WindowKey),

}
```

The console's `register_layout_handle_hitboxes` and `layout_chrome.rs` will use `handle.split_id` instead of `handle.hitbox_id` for the `ChromeTarget::SplitHandle` payload.

### AMENDMENT 3: Z-Layer Priority + Clear Sync + Event Routes

**Z-Layer Priority in `HitboxRegistry::hit_test()`**: Segment the registry into overlay vs standard entries. `hit_test` evaluates entries in LIFO (reverse insertion) order within each bucket — overlays first, then standard. This preserves correct child-over-parent priority (child widgets registered during `comp.render()` match before the content background registered during `render_window_chrome()`). The existing `HitboxRegistry` already uses `.iter().rev()` — no change to evaluation order, just bucket isolation.

**Clear synchronization**: `HitboxRegistry::clear()` must purge both overlay and standard collections.

**Geometry consolidation**: Move `contains`/`intersects` onto `LayoutRect` in `term-wm-layout-engine`. Replace local duplicates.

**Modal overlay backdrop**: Each modal overlay (Command Palette, Help Overlay) must register a full-screen blocker hitbox in `overlay_hitboxes` covering `0..screen_width × 0..screen_height`. `dispatch_mouse` handles it: dismissible → `TermWmAction::CloseOverlay`, non-dismissible → `EventResult::Consumed`.

**Event routes**: Add overlay backdrop handler in `dispatch_mouse`. Add reverse focus cycling matching both `KeyCode::BackTab` and `KeyCode::Tab` + `KeyModifiers::SHIFT`.

| File | Change |
|------|--------|
| `crates/term-wm-core/src/hitbox_registry.rs` | Split into overlay/standard buckets. `clear()` resets both. `hit_test` checks overlays LIFO first. |
| `crates/term-wm-layout-engine/src/lib.rs` | Add `LayoutRect::contains()` and `LayoutRect::intersects()`. |
| `crates/term-wm-console/src/draw_plan_renderer.rs` | Replace local `rect_intersects` with `LayoutRect::intersects()`. |
| `crates/term-wm-core/src/window/window_manager/mod.rs` | Use `LayoutRect::contains()`. Add `KeyCode::BackTab` + `Tab`/`Shift` for `focus_previous_window()`. Fix overlay backdrop in `dispatch_mouse`. |
| `crates/term-wm-console/src/draw_plan_renderer.rs` or overlay renderers | Register full-screen overlay backdrop hitbox in `overlay_hitboxes` bucket. |

### Bug 8: Missing corner resize hitboxes

### Corrections from Review

- `ResizeEdge` already has corner variants (`TopLeft`, `TopRight`, `BottomLeft`, `BottomRight`) — use them directly. No new variants needed.
- `SplitHandle.hitbox_id` is `HitboxId` — `ChromeTarget::SplitHandle(HitboxId)` is correct. No `SplitHandleId` type exists.
- Phase-1 capture handler for `LayoutHandle` must return `EventResult::Consumed`, not `(bool, bool)`. Use `layout.handle_event(&core_event, area)` and return `EventResult::Consumed`. On Release, set `self.mouse_capture = None`.
- Z-stack elevation: `dispatch_mouse` must call `self.bring_floating_to_front_key(key)` on `MouseEventKind::Press(_)` for any floating window target (`ComponentOwner::Window(key)` or `ComponentOwner::Chrome(target)` where target belongs to a floating window). Without this, clicking a buried floating window updates focus without reordering the Z-stack, leaving it visually obscured.

### Bug 11: Floating window Drag hitbox registered at wrong screen Y

**Root cause**: `register_window_chrome_hitboxes` computes `drag_rect` in local coords (y=1) and applies `screen()` to translate to screen space. If `dest_offset` is missing or (0,0), the floating window's Drag hitbox is registered at screen y=1 instead of at y = dest.y + 1, making it overlap with every tiled window's Drag hitbox.

**Fix**: Audit the `dest_offset` passed through the call chain. In `composite_window`, confirm `render_window_chrome` receives `(surface.dest.x, surface.dest.y)`. In `register_window_chrome_hitboxes`, confirm `screen()` translation is applied to EVERY hitbox rect including Drag, buttons, and content hitbox. All of these already use `screen()` — the `dest_offset` itself (the `(dx, dy)` values) must be verified.

**Investigation needed**: Add `tracing::trace!` to print `dest_offset`, `drag_rect`, and `screen(drag_rect)` at runtime. Once the coordinate values are logged, the discrepancy will identify whether `window_dest` returns (0,0) for floating windows or `dest_offset` gets dropped somewhere in the pipeline.

### Bug 13: Non-Press events on chrome buttons leak to underlying windows

`dispatch_mouse` returns `EventResult::Ignored` for `MouseEventKind::Release` and `MouseEventKind::Drag` on chrome control buttons (CloseButton, MaximizeButton, MinimizeButton, ToggleDirectMode). This leaks release-phase mouse events through to lower-layer windows.

**Fix**: Change button match arms to return `EventResult::Consumed` for non-Press events:

```rust
ChromeTarget::CloseButton(key) => match kind {
    MouseEventKind::Press(_) => EventResult::Action((Some(*key), TermWmAction::CloseWindow)),
    _ => EventResult::Consumed,
},
```

Same pattern for MaximizeButton, MinimizeButton, ToggleDirectMode.

### Bug 14: `HitboxRegistry::register` is stateful — `set_active_owner` causes owner pollution

### Bug 15: `register_window_chrome_hitboxes` signature allows double-translation

`local_bounds: LayoutRect` + `dest_offset` permits passing screen-space rects as local bounds, causing 2x/2y double-translation.

**Fix**: Replace with `frame_size: (u16, u16)` + `screen_origin: (i16, i16)`.

**`crates/term-wm-console/src/draw_plan_renderer.rs`**:
```rust
pub fn register_window_chrome_hitboxes(
    registry: &mut HitboxRegistry,
    key: WindowKey,
    frame_size: (u16, u16),     // (width, height)
    screen_origin: (i16, i16),  // (screen_x, screen_y)
    content_hitbox_id: HitboxId,
) {
    let (width, height) = frame_size;
    let (ox, oy) = screen_origin;
    let to_screen = |lx: u16, ly: u16, lw: u16, lh: u16| -> LayoutRect {
        LayoutRect { x: ox + lx as i16, y: oy + ly as i16, width: lw, height: lh }
    };
    // ... use to_screen for all rects
}
```

Update `render_window_chrome` and `composite_window` callers to pass `(dest.width, dest.height)` and `(dest.x as i16, dest.y as i16)`.

### Bug 12: Shadow hitbox leakage

If shadow cells intercept clicks, the `register_window_chrome_hitboxes` function is receiving bounds that include shadow offsets. Fix: verify that `full_local` and `dest_offset` produce strict frame geometry without shadow expansion. The shadow is drawn at `surface.dest + SHADOW_OFFSET` which is outside the `surface.dest` bounds. Content and chrome hitboxes use only `surface.dest` area. If shadow coordinates overlap with an adjacent window's hitbox, the adjacent window's hitbox (registered later) takes priority via LIFO.

Audit `composite_window` to confirm `render_window_chrome` receives only `full_local = (0, 0, dest.width, dest.height)` — no shadow expansion. The shadow (`render_drop_shadow`) runs after hitbox registration on the main buffer and never touches the hitbox registry. No fix needed if this invariant holds.

### Bug 16: Two-click drag requirement on unfocused windows

**Cause**: `dispatch_mouse` sets focus/Z-stack elevation on `Press` for `ChromeTarget::Drag` and `ChromeTarget::Resize`, but does NOT initialize `self.mouse_capture` atomically on the first press. The user must click once to focus, then again to drag.

**Fix**: Atomic capture on first press — set `self.mouse_capture` immediately when `ChromeTarget::Drag` or `ChromeTarget::Resize` is matched on a press event, regardless of focus state.

### Bug 17: Tiled window drag initiation fails silently

**Cause**: `ChromeTarget::Drag(key)` calls `init_window_drag` which sets absolute floating coordinates. For tiled windows, the layout engine overwrites these coordinates on the next frame, so drag has no visible effect. Also, the tiled window has no `floating_rect` set, so `init_window_drag` may fail to acquire starting coordinates.

**Fix**: When `ChromeTarget::Drag` matches a tiled window, first convert it to floating (`detach_from_tiling_layout(key)`) then set `floating_rect` to the current region bounds. Then initiate drag capture. This matches the old behavior where dragging a tiled window's header detached it from the tiling layout.

### Bug 18: Tiled windows behind floating windows (constraint, not a bug)

When a floating window's frame visually covers a tiled window's header, Z-occlusion dictates the top window receives the click. This is correct WM behavior. No fix needed — the user must move the floating window first.

If the floating window's hitboxes extend beyond its visual frame (shadow padding, inflated dest bounds), fix the coordinate math in `register_window_chrome_hitboxes` — not the dispatch layer.
