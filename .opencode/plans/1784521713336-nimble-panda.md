# Chrome Fragmentation Fix — Dumb Core Architecture

## Problem
Chrome metrics (border widths, header heights, button positions) are hardcoded in 4 locations across `term-wm-core` and `term-wm-console`, creating implicit coupling. A border thickness change requires synchronized edits across crate boundaries.

## Solution: Dumb Core
The core defines the **vocabulary** (`ChromeTarget`). The console owns ALL chrome math — it draws, registers hitboxes, and returns the inner content rect.

## Core Changes (`term-wm-core`)

### 1. `src/chrome/target.rs` — Only the Vocabulary
```rust
pub enum ChromeTarget {
    Resize(WindowKey, ResizeEdge),
    Drag(WindowKey),
    CloseButton(WindowKey),
    MaximizeButton(WindowKey),
    SplitHandle(HitboxId),
}
```
No metrics. No border widths. No button placements. Purely semantic.

### 2. `src/hitbox_registry.rs`
`ComponentOwner::Chrome(ChromeTarget)` replaces `ComponentOwner::System` (already done).

### 3. `src/window/decorator.rs`
- Remove ALL metric constants (`LEFT_BORDER_WIDTH`, `TOP_BORDER_HEIGHT`, etc.)
- Remove `header_buttons()` function
- Remove `WindowDecorator` trait entirely
- Remove `WindowRenderCtx`, `DefaultDecorator`, `ButtonRenderInfo`, `HeaderRenderInfo`, `BorderRenderInfo`
- Remove `HeaderAction` enum (replaced by individual `ChromeTarget` variants: `CloseButton`, `MaximizeButton`)

### 4. `src/window/window_manager/mod.rs`
- Remove `resize_handles`, `floating_headers`, `handles` fields (console registers these hitboxes during render)
- Remove `register_window_chrome_hitboxes()` and `register_layout_handle_hitboxes()`
- Remove `resize_map`, `drag_map`, `split_ids` (already done)
- `dispatch_mouse`: already matches `ComponentOwner::Chrome(target)` with direct `TermWmAction` routing — update match arms from `Button(key, HeaderAction::Close)` pattern to `CloseButton(key)` etc.
- Remove `decorator()` and `set_decorator()` from `chrome.rs` (no `WindowDecorator` to set)

### 5. `src/window/window_manager/layout.rs`
- Remove `resize_handles` and `floating_headers` computation
- Console computes these from its own geometry during render

### 6. Downstream Fixes for Removed Types

**`src/wm_config.rs`**: Remove `decorator: Option<Arc<dyn WindowDecorator>>` field and `decorator()` method. Remove `DefaultDecorator` import. If no replacement needed, delete the field entirely.

**`src/config.rs`**: Remove `.decorator(mut self, decorator)` builder method and `WindowDecorator` import.

**`src/window/window_manager/chrome.rs`**: Remove `decorator()` and `set_decorator()` methods. Keep `close_window()`, `minimize_window()`, `toggle_maximize()`.

**`src/chrome/target.rs`**: Replace `Button(WindowKey, HeaderAction)` with `CloseButton(WindowKey)` and `MaximizeButton(WindowKey)`. Remove `HeaderAction` import.

**`src/chrome/mod.rs`**: Remove re-export of `HeaderAction` and the `window_chrome`/`layout_chrome` submodules.

## Console Changes (`term-wm-console`)

### 7. `src/draw_plan_renderer.rs` — Owns ALL Chrome Math
- Add `render_window_chrome()` that draws borders/header/buttons using locally-defined constants, registers hitboxes via `ComponentOwner::Chrome(...)`, and returns the inner `LayoutRect`
- Update `render_handles_masked()` → `render_split_handles()` that also registers split handle hitboxes
- Update `render_resize_outline()` to register resize edge hitboxes
- Remove imports of `WindowRenderCtx`, `WindowDecorator`, `HeaderAction` — console builds its own render context
- Replace `header_buttons()` usage with direct button registration via `ChromeTarget::CloseButton(key)` / `ChromeTarget::MaximizeButton(key)`

### 8. `src/lib.rs` `render_app()`
Simpler orchestration — chrome rendering bakes in hitbox registration:
```
For each window:
  inner = render_window_chrome(buffer, window_rect, registry, key, title, focused, theme)
  component.render(backend, inner, ...)  // content in remaining space

render_split_handles(buffer, handles, hovered, registry, ...)
render_resize_outline(buffer, ...)
render_panels(backend, wm) // already registers its own hitboxes
```
