# Codebase Exploration Report: Chrome Fragmentation Analysis

## Overview
**Project**: term-wm ‚Äî Rust terminal window manager  
**Stats**: 134 files, 2996 nodes, 21529 edges  
**Crates**: `term-wm-core` (layout, window mgmt, events), `term-wm-console` (rendering), `term-wm-ui-components` (TerminalComponent, scroll, etc.), `term-wm-sys-ui-components` (top/bottom panels), `term-wm-layout-engine` (node tree), `term-wm-pty-engine`, `term-session-client/server`

## Chrome Fragmentation (SEV-2)

Four locations hold implicit coupling for window chrome:

### 1. Metric Constants ‚Äî `crates/term-wm-core/src/window/decorator.rs`
Border widths/heights (`LEFT_BORDER_WIDTH`, `RIGHT_BORDER_WIDTH`, `TOP_BORDER_HEIGHT`, `BOTTOM_BORDER_HEIGHT`, `HEADER_HEIGHT`, etc.), `WindowDecorator` trait (stub `render_window` is a no-op), `WindowRenderCtx`, `HeaderAction`, `header_buttons()` computes button positions.

### 2. Chrome Geometry Construction ‚Äî `crates/term-wm-core/src/window/window_manager/layout.rs`
Computes `resize_handles` (resize edge hitboxes) and `floating_headers` (drag header hitbox) from window layout. Relies on same border metrics.

### 3. Hitbox Registration & Event Dispatch ‚Äî `crates/term-wm-core/src/window/window_manager/mod.rs`
- `register_window_chrome_hitboxes()`: Puts resize/drag handles into `resize_map`/`drag_map` + `HitboxRegistry`
- `register_layout_handle_hitboxes()`: Puts split handles into `split_ids` + registry
- `dispatch_mouse()`: Phase-2 chrome interception checks `resize_map` / `drag_map` / `split_ids` with hardcoded button-position logic (line 1476-1556 in mod.rs)

### 4. Chrome Rendering ‚Äî `crates/term-wm-console/src/draw_plan_renderer.rs`
- `render_window()` (line 664): Draws borders, title bar, header buttons with box-drawing chars
- `render_handles_masked()` (line 818): Draws tiling split handles with occlusion masking
- `render_resize_outline()` (line 965): Draws floating resize outlines (double-line box chars)
- `composite_window()` (line 572): Orchestrates per-window chrome + content compositing

### 5. Orchestration ‚Äî `src/lib.rs` `render_app()` (line 23)
Calls the chain: `register_panel_hitboxes` ‚Üí `register_layout_handle_hitboxes` ‚Üí loop over windows: `register_window_chrome_hitboxes` + `composite_window` ‚Üí `render_panels` ‚Üí `render_handles_masked` ‚Üí `render_resize_outline` ‚Üí `render_overlays`

### The Coupling Problem
A change to, e.g., border thickness requires synchronized edits in:
- `decorator.rs` (constants)
- `layout.rs` (handle geometry computation)
- `draw_plan_renderer.rs` (rendering)
- `window_manager/mod.rs` `dispatch_mouse()` (button-position logic)

## Refactoring Directives (AMENDED)

### Keep Chrome Managers Pure ‚Äî No Rendering in Core
`term-wm-core` is a pure geometry/state crate. Do NOT define rendering traits inside it. The chrome managers output only static geometric metadata; the console crate consumes it.

- `WindowChromeManager` and `LayoutChromeManager` compute `LayoutRect` metrics and register hitboxes
- `term-wm-console` queries the managers for metrics during the draw pass and executes its own imperative buffer mutations
- No `ChromeRenderer` trait in core ‚Äî `term-wm-core` remains domain-pure

### Split Into Two Domains
Chrome is not homogeneous. Split into two explicit managers:

**`WindowChromeManager`** (per-window):
- Computes local window geometry: borders, headers, buttons, resize edges
- Owns metric constants (moved from `decorator.rs`)
- Uses metrics directly (no intermediate AST)
- Exposes `content_area(window_rect)` and chrome region rects

**`LayoutChromeManager`** (global):
- Computes tiling split handle geometry from the layout node tree
- Executes occlusion masking against the full floating window Z-stack
- Cannot be per-window; split handles are topological seams between nodes

### Integrate `ChromeTarget` Into `ComponentOwner`

`ChromeTarget` must nest inside `ComponentOwner` to preserve the `HitboxRegistry` invariant:

```rust
pub enum ComponentOwner {
    Window(WindowKey),
    Overlay(OverlayKey),
    Layer(LayerId),
    Chrome(ChromeTarget),  // Replaces the generic System variant
    #[cfg(test)]
    Test,
}

pub enum ChromeTarget {
    Resize(WindowKey, ResizeEdge),
    Drag(WindowKey),
    Button(WindowKey, HeaderButtonAction),
    Split(SplitHandleId),
}
```

This guarantees `HitboxRegistry::hit_test()` yields the exact payload for O(1) dispatch. `dispatch_mouse` matches directly on `ComponentOwner::Chrome(target)` without separate `resize_map`/`drag_map`/`split_ids` lookups or hardcoded button-position logic.

### Not a `Component` Trait
Managers remain pure synchronous data pipelines, executed during the top-down layout/render pass. No async queue, no `TermWmAction`.

- `compute_metrics(window_bounds, focused, floating, theme) -> WindowChromeMetrics` ‚Äî pure geometric projection, no `Vec`
- `register_chrome_hitboxes(registry, key, &metrics)` ‚Äî registers bounded chrome elements (4 edges, 4 corners, 1 header, 4 buttons) directly into `HitboxRegistry`
- `content_area(window_rect) -> LayoutRect` (moved from `WindowDecorator`)

### `LayoutChromeManager` responsibilities (in `term-wm-core`):

Zero-allocation callback API for split handles:

```rust
impl LayoutChromeManager {
    /// Zero-allocation: calls `callback` for each computed split metric.
    /// `floating_geometries` provides resolved bounding boxes for Z-depth occlusion masking.
    pub fn for_each_split_metric<F>(
        &self,
        handles: &[SplitHandle],
        floating_geometries: &[LayoutRect],
        hovered: Option<&SplitHandle>,
        mut callback: F,
    ) where
        F: FnMut(SplitHandleMetric),
    {
        for handle in handles {
            // Compute is_obscured by intersecting handle.rect against each
            // floating_geometry in floating_geometries (sorted by Z-order).
            callback(SplitHandleMetric { rect: ..., is_hovered: ..., ... });
        }
    }
}
```

- `resize_outline_region(key, edge, floating_panes, bounds) -> Option<LayoutRect>` ‚Äî resize highlight bounds

### Rendering Stays in `term-wm-console` (no trait in core):
`draw_plan_renderer.rs` directly queries chrome managers for `WindowChromeMetrics` and renders:
```rust
// In draw_plan_renderer.rs (or a new module within term-wm-console):
render_window_borders(buffer, metrics.outer_bounds, focused, floating, theme);
render_window_header(buffer, metrics.title_area, title, focused, &metrics.buttons, theme);
render_split_handle(buffer, metric.rect, is_hovered, theme);
render_resize_outline(buffer, bounds, theme);
```

These are free functions (like the current `render_window`, `render_handles_masked`, `render_resize_outline`) operating on `&mut Buffer` ‚Äî no trait, no `Vec`, no allocation.

### Integration into `render_app` (src/lib.rs):
```
For each window:
  metrics = WindowChromeManager::compute_metrics(window_bounds, ...)
  WindowChromeManager::register_chrome_hitboxes(registry, key, &metrics)
  render_window_borders(buffer, metrics.outer_bounds, ...)
  render_window_header(buffer, metrics.title_area, ...)
  for btn in metrics.buttons.iter().flatten():
    render_header_button(buffer, btn.rect, btn.action, ...)
  // (inner content rendering unchanged)

For tiling layout:
  floating_geometries: Vec<LayoutRect> = wm.floating_geometries_sorted()  // Z-sorted bounding boxes
  LayoutChromeManager::for_each_split_metric(handles, &floating_geometries, hovered, |metric| {
    hitbox_registry.register(ChromeTarget::Split(metric.id), metric.rect, ComponentOwner::Chrome);
    render_split_handle(buffer, metric.rect, metric.is_hovered, theme);
  })

For floating windows:
  render_resize_outline(buffer, bounds, ...)
```

### Changes to `dispatch_mouse`:
Remove `resize_map` / `drag_map` / `split_ids` fields and hardcoded button-position arithmetic. The `HitboxRegistry::hit_test()` already returns `(HitboxId, ComponentOwner, LayoutRect)`. Replace `ComponentOwner::System` match arm with:

```rust
ComponentOwner::Chrome(target) => match target {
    ChromeTarget::Resize(key, edge) => {
        if !matches!(kind, MouseEventKind::Press(_)) {
            return EventResult::Ignored;
        }
        if !config.floating_windows_enabled || !self.is_window_floating(*key) {
            return EventResult::Ignored;
        }
        // Init resize capture state
        self.bring_floating_to_front_key(*key);
        self.mouse_capture = Some(MouseCaptureState::ResizingWindow {
            key: *key, edge: *edge, ...
        });
        EventResult::Consumed
    }
    ChromeTarget::Drag(key) => {
        if !matches!(kind, MouseEventKind::Press(_)) {
            return EventResult::Ignored;
        }
        // Init drag capture state (handles floating detach, double-click maximize)
        self.mouse_capture = Some(MouseCaptureState::DraggingWindow { key: *key, ... });
        self.arm_drag_snap_timer();
        EventResult::Consumed
    }
    ChromeTarget::Button(key, action) => {
        if !matches!(kind, MouseEventKind::Press(_)) {
            return EventResult::Ignored;
        }
        let wm_action = match action {
            HeaderButtonAction::Close => TermWmAction::CloseWindow,
            HeaderButtonAction::Maximize => TermWmAction::MaximizeWindow,
            HeaderButtonAction::Minimize => TermWmAction::MinimizeWindow,
            HeaderButtonAction::ToggleDirectMode => TermWmAction::ToggleDirectMode,
        };
        EventResult::Action((Some(*key), wm_action))
    }
    ChromeTarget::Split(id) => {
        if !matches!(kind, MouseEventKind::Press(_)) {
            return EventResult::Ignored;
        }
        self.mouse_capture = Some(MouseCaptureState::LayoutHandle(*id));
        EventResult::Consumed
    }
}
```
