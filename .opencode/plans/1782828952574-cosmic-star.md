# Plan: Replace Custom TCP Protocol with Muxio IPC Streaming

## Overview

Replace the custom 5-byte-header + `bitcode` framing protocol in `term-session-server`/`term-session-client` with `muxio`'s layered multiplexing and RPC framework. Create a shared service-definitions crate for a single source of truth between client and server.

## Background

The same author (jzombie) wrote both `term-wm` and `muxio`. The muxio ecosystem at 0.11.0-alpha includes:

| Crate | Purpose |
|---|---|
| `muxio-core` | Binary framing, stream multiplexing, callback-driven primitives |
| `muxio-rpc-service` | `RpcMethodPrebuffered` trait, `rpc_method_id!()` macro, service contracts |
| `muxio-rpc-service-caller` | `RpcServiceCallerInterface` — generic client invocation |
| `muxio-rpc-service-endpoint` | `RpcServiceEndpointInterface` — server handler registration |
| `muxio-tokio-rpc-ipc-server` | Local-socket IPC server (interprocess crate) |
| `muxio-tokio-rpc-ipc-client` | Local-socket IPC client (interprocess crate) |

The existing workspace already uses `tokio`, `bitcode`, and `tracing` — all shared deps with muxio.

## Step-by-step

### Step 1 — Create `crates/term-session-muxio-service-definitions`

A new workspace member that defines the RPC contract using muxio's `RpcMethodPrebuffered` trait. Each method is a unit struct with `METHOD_ID` (from `rpc_method_id!`), `Input`/`Output` types, and encode/decode impls using `bitcode`.

**Client→Server request methods (called by client, handled by server):**
- `Spawn` — Input: `(Option<Vec<String>>, cols: u16, rows: u16)`, Output: `id: u64`
- `WriteInput` — Input: `(id: u64, data: Vec<u8>)`, Output: `()`
- `ResizePty` — Input: `(id: u64, cols: u16, rows: u16)`, Output: `()`
- `CloseSession` — Input: `id: u64`, Output: `()`
- `ListSessions` — Input: `()`, Output: `Vec<(u64, String, bool)>`

**Server→Client push methods (called by server, handled by client):**
- `WelcomePush` — Input: `Vec<(u64, String, bool)>`, Output: `()`
- `RawOutputPush` — Input: `(id: u64, data: Vec<u8>)`, Output: `()`
- `SnapshotPush` — Input: `(id: u64, data: Vec<u8>)`, Output: `()`
- `SessionExitedPush` — Input: `(id: u64, status: i32)`, Output: `()`
- `TitleChangedPush` — Input: `(id: u64, title: String)`, Output: `()`

**Cargo.toml:** dep on `muxio-rpc-service = "0.11.0-alpha"`, `bitcode`

### Step 2 — Refactor `term-session-server`

- Rewrite `session_server.rs` using `muxio_tokio_rpc_ipc_server::RpcIpcServer`
- Register handler closures for each request method on the server endpoint
- Per-connection: store the `RpcServiceCallerInterface` handle to call client push methods
- Keep `session.rs` unchanged (Session struct, Pty + parser logic)
- Remove `protocol.rs` entirely; update `lib.rs` to drop its re-export
- `main.rs` becomes async (`#[tokio::main]`), starts the IPC server
- **Cargo.toml:** remove `bitcode`; add `muxio-tokio-rpc-ipc-server`, `muxio-rpc-service`, `term-session-muxio-service-definitions`, `tokio`

### Step 3 — Refactor `term-session-client`

- Remove `connection.rs` entirely (TCP background thread, mpsc channels, custom `SessionServerConnection`)
- Rewrite `remote_pane.rs` to hold an `RpcIpcClient` and implement `Pane` by making muxio RPC calls
- Rewrite `lib.rs` to:
  1. Connect via `muxio_tokio_rpc_ipc_client::RpcIpcClient::new()`
  2. Register push handlers on the client endpoint
  3. Run the async event loop
- `main.rs` becomes async (`#[tokio::main]`), passes IPC socket path
- **Cargo.toml:** remove `bitcode`; add `muxio-tokio-rpc-ipc-client`, `muxio-rpc-service`, `term-session-muxio-service-definitions`, `tokio`

### Step 4 — Update workspace root `Cargo.toml`

- Add workspace dependency entries for muxio crates (`0.11.0-alpha`)
- Add `"crates/term-session-muxio-service-definitions"` to workspace members
- Add workspace dep entries for the new shared crate

### Step 5 — Update callers

- `term-session-client/Cargo.toml` currently depends on `term-session-server` (for the protocol types). After the refactor, it depends on `term-session-muxio-service-definitions` instead, so remove the server dep.
- The main `term-wm` app creates PTY sessions directly (via `TerminalComponent::spawn_default`), not through the session server, so it's unaffected.

## Critical Design Decisions

1. **IPC transport (local sockets) not WebSocket** — using `muxio-tokio-rpc-ipc-*` (v0.11.0-alpha) with `interprocess` for cross-platform local sockets, matching the user's "IPC streaming" requirement.
2. **Push events become server→client RPC calls** — muxio's bidirectional architecture lets both sides register endpoints; pushes are client-side method handlers called by the server over the same IPC connection.
3. **`RpcMethodPrebuffered` for everything** — all messages are small and complete (no streaming responses needed), so prebuffered (full-request-before-dispatch) semantics are correct.
4. **Async rewrite of server and client** — the muxio IPC crates are Tokio-based, so both binaries become `#[tokio::main]` async functions.
5. **Scope: only `term-session-server` and `term-session-client`** — the main `term-wm` app creates PTY sessions in-process via `TerminalComponent::spawn_default()` and is unaffected.
