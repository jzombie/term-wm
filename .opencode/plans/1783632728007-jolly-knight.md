## Architecture Overview

**term-wm** is a cross-platform terminal multiplexer and Ratatui-based window manager (~2533 nodes, 113 Rust files, 12 crates).

### Crate Dependency Flow (bottom→top)
```
term-wm-layout-engine (pure math, no_std)
    → term-wm-render (RenderBackend, RenderTarget traits)
    → term-wm-pty-engine (PTY, Pane trait)
    → term-wm-core (Component trait, WindowManager, events, Engine)
        → term-wm-ui-components (leaf components: terminal, scroll, markdown, etc.)
        → term-wm-sys-ui-components (WM chrome: panels, menu, debug)
        → term-wm-console (RatatuiBackend, DrawPlanRenderer, ConsoleRenderTarget)
    → src/ (main binary, TermWmApp, render_app, UnifiedEventSource)
```

## Key Architectural Patterns

### Component System
- **Trait**: `Component<Msg>` (~term-wm-core/src/components/mod.rs) — 5-phase lifecycle: init → handle_events → update → render → destroy
- **Facade pattern for events**: Default `handle_events` converts screen-space mouse to local coords via `ctx.screen_area()`, dispatches to `on_mouse_press/release/drag/scroll/move` and `on_key`
- **Context**: `ComponentContext` carries `hover_pos`, `screen_area`, `focused`, etc.
- **Overlay trait**: Extends Component with `visible()`, `shadow_rect()`
- **WmComponent trait**: For WM chrome components (panels, menu)

### WindowManager (central orchestrator)
- Stores all windows in `SlotMap<WindowKey, Window>`
- Manages tiling layout, floating windows, overlays, focus ring, z-order
- **Key fields**: `hover: Option<(u16, u16)>` (cursor position), `mouse_capture: Option<MouseCaptureState>`, `hitbox_registry: HitboxRegistry`
- **`dispatch_mouse()`** — 4-phase dispatch: active capture → scroll → moved → press
- **`prepare_draw()`** — clears per-frame state each frame (regions, handles, hitbox_registry)

### HitboxRegistry
- Flat array of entries + clip stack for scroll containers
- `hit_test()` does O(n) reverse scan (front-to-back = top-most first)
- `HitTarget` enum: `Window`, `Component`, `Overlay`, `TopPanel`, `BottomPanel`, `ChromeResize`, `ChromeHeader`, `LayoutHandle`

### Mouse Position Tracking
- `WindowManager.hover: Option<(u16, u16)>` — set on every mouse event, cleared on FocusLost
- `MousePosition { column: i16, row: i16, space: CoordSpace }` — only `Screen` variant currently
- `ComponentContext.hover_pos: Option<(u16, u16)>` — currently set ONLY for overlays, not for window components

### Rendering Pipeline
1. `begin_frame()` → `prepare_draw()` before each draw
2. `register_managed_layout(area)` → projects tiling layout
3. `engine.project_draw_plan()` → produces `DrawPlan` (sorted `Vec<RenderRegion>`)
4. For each region: `composite_window()` with decorator + content rendering
5. Register chrome hitboxes (resize, header buttons)
6. `render_panels()`, `render_handles_masked()`, `render_resize_outline()`, ghost preview, dim target borders
7. Overlay drop shadows + `render_overlays()`
8. Final buffer swap in `ConsoleRenderTarget::draw()` → `Frame::buffer_mut()`

## Mouse Hover State Analysis

### Current State
- **No general cursor overlay exists** — `Modifier::REVERSED` is only used for selection rendering, cursor rendering, and list item selection
- Hover feedback is limited to: header buttons, tiling split handles, floating resize outlines
- `ComponentContext.hover_pos` is NOT propagated to window components (only overlays)
- `WindowManager.hover` is the single source of truth, set in `handle_focus_event` and `dispatch_mouse`

### What Needs to Change for CursorOverlayComponent
1. A new post-render step that applies `Modifier::REVERSED` to the cell under the cursor
2. Interaction states: Idle (no change), Hovering (reversed cell), Active (reversed + badge)
3. Must run AFTER all rendering (last step before buffer swap)
4. Needs access to `Buffer` via `RatatuiBackend::buffer_mut()`
5. No new Component needed — can be a simple function called at end of `render_app()`

## Next Steps for Implementation

1. Add a `cursor_state` or `interaction_mode` field to `WindowManager` to track current interaction phase (Idle/Hovering/Active)
2. Create a cursor overlay rendering function that:
   - Reads `wm.hover_pos()` to get cursor location
   - Checks interaction state
   - Applies `Modifier::REVERSED` to the cell at cursor position via `buffer_mut()`
   - For Active state, also renders a badge at (X+1, Y+1)
3. Call this function as the very last rendering step in `render_app()` (after `render_overlays()`)
4. Update `ComponentContext` to propagate `hover_pos` to all components (for potential component-level hover handling)
5. Add test coverage for the overlay rendering
