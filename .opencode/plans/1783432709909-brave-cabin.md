# Plan: Sticky Bottom Auto-Scroll for ScrollViewComponent

## Problem

`WmDebugLogComponent` wraps `ScrollViewComponent<TextRendererComponent>`. When new log lines arrive and the user is scrolled to the bottom, the viewport stays at its absolute offset instead of snapping to the new bottom. This leaves the user one line behind the latest content.

The existing `follow_tail` mechanism in `WmDebugLogComponent` works at the component level (calling `scroll_vertical_to(usize::MAX)` before render), but the underlying `ScrollViewComponent` has no awareness of this pattern. We want to push sticky-bottom logic into `ScrollViewComponent` so any child component can opt-in.

## Files to Modify

1. **`crates/term-wm-core/src/component_context.rs`** — add `sticky_bottom` to `ViewportSharedState`, modify `set_content_size`
2. **`crates/term-wm-ui-components/src/scroll_view.rs`** — add `set_sticky_bottom` setter, update render loop retry condition
3. **`crates/term-wm-sys-ui-components/src/wm_debug_log.rs`** — enable `sticky_bottom` on the inner scroll view

## Changes

### 1. Add `sticky_bottom` to `ViewportSharedState` and modify `set_content_size`

**File:** `crates/term-wm-core/src/component_context.rs`

Add the field to the struct (~line 63):

```rust
#[derive(Debug, Default)]
pub struct ViewportSharedState {
    pub offset_x: usize,
    pub offset_y: usize,
    pub width: usize,
    pub height: usize,
    pub content_width: usize,
    pub content_height: usize,
    pub pending_offset_x: Option<usize>,
    pub pending_offset_y: Option<usize>,
    pub sticky_bottom: bool,
}
```

Replace `set_content_size` (~line 96) with the snap logic:

```rust
pub fn set_content_size(&self, width: usize, height: usize) {
    let mut inner = self.shared.borrow_mut();

    // Check if we were at the bottom BEFORE updating content dimensions
    let old_max_y = inner.content_height.saturating_sub(inner.height);
    let was_at_bottom = inner.offset_y >= old_max_y;

    inner.content_width = width;
    inner.content_height = height;

    // If sticky mode is on and we were at the bottom, snap to the new bottom
    if inner.sticky_bottom && was_at_bottom {
        let new_max_y = inner.content_height.saturating_sub(inner.height);
        if new_max_y > inner.offset_y {
            inner.offset_y = new_max_y;
            inner.pending_offset_y = Some(new_max_y);
        }
    }
}
```

**Why this works without 1-frame lag:** `set_content_size` is called during the child's `render()` pass. It mutates `offset_y` in the shared state. After the child renders, the `ScrollViewComponent` render loop reads the updated offset and detects the change, triggering a retry that re-renders the child with the correct offset.

### 2. Expose setter and update render loop retry condition

Add setter after `set_keyboard_mode` (~line 198):

Update the render loop's retry condition (~line 453). Add `offset_changed` to detect when `set_content_size` snapped the offset:

```rust
let state = self.shared_state.borrow();
let content_w = state.content_width;
let content_h = state.content_height;
let off_x = state.offset_x;
let off_y = state.offset_y;
drop(state);

// Detect if the child's measurement triggered a sticky auto-scroll
let offset_changed = off_x != info.offset_x || off_y != info.offset_y;

let needs_vertical = inner_area.height > 0 && content_h > inner_area.height as usize;
let has_vertical_reserved = inner_area.width < area.width;
let needs_horizontal = inner_area.width > 0 && content_w > inner_area.width as usize;
let has_horizontal_reserved = inner_area.height < area.height;

let drop_vertical = has_vertical_reserved && !needs_vertical && area.width > 0;
let drop_horizontal = has_horizontal_reserved && !needs_horizontal && area.height > 0;
let retry_vertical =
    (needs_vertical && !has_vertical_reserved && area.width > 0) || drop_vertical;
let retry_horizontal =
    (needs_horizontal && !has_horizontal_reserved && area.height > 0)
        || drop_horizontal;

// Trigger a re-render in the exact same frame if offsets snapped
if (retry_vertical || retry_horizontal || offset_changed) && attempt + 1 < max_attempts {
    attempt += 1;
    continue;
}
```

### 3. Enable sticky_bottom on the debug log

**File:** `crates/term-wm-sys-ui-components/src/wm_debug_log.rs`

In the constructor `new()` (~line 268), after creating the `ScrollViewComponent`, enable sticky_bottom:

```rust
let mut scroll_view = ScrollViewComponent::new(renderer);
scroll_view.set_sticky_bottom(true);
```

This replaces the need for the `follow_tail` / `last_total` / `last_view` tracking in `WmDebugLogComponent` — the scroll view now handles it natively. However, to minimize risk, we keep the existing `follow_tail` mechanism as a fallback during the transition. The two mechanisms are complementary: `follow_tail` calls `scroll_vertical_to(usize::MAX)` before render, while `sticky_bottom` snaps inside `set_content_size` when content grows. Both converge on the same behavior.

## Behavior Matrix

| Scenario | sticky_bottom | User at bottom? | New content arrives | Result |
|----------|--------------|-----------------|---------------------|--------|
| Debug log, tailing | true | yes | yes | Viewport snaps to new bottom |
| Debug log, inspecting old lines | true | no | yes | Viewport stays at current position |
| Debug log, scrolled back to bottom | true | yes (manually) | yes | Auto-scroll resumes |
| Help overlay | false (default) | n/a | n/a | No change |
| Terminal shell | false (default) | n/a | n/a | No change |
