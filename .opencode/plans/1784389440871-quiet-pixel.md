# Fix: Scroll, click targeting, gap

## Fix 1: MenuComponent needs viewport support

### 1a. Add `show_header: bool` field to `MenuComponent`
Default `true` (backward compatible). When `false`, items render starting at
row 0 instead of row 1 — no header gap.

**File:** `crates/term-wm-ui-components/src/menu.rs`
- Add `pub show_header: bool` to struct, init `true`
- In `render_items()`: compute `start_row = if self.show_header { 1 } else { 0 }`
- In `handle_events()`: adjust `start_row` accordingly
- Provide `pub fn set_show_header(&mut self, show: bool)`

### 1b. Add viewport offset support to MenuComponent
In `MenuComponent::render()`, read `ctx.viewport().map(|v| v.offset_y).unwrap_or(0)`.
Then render only items `offset_y..offset_y + visible_count` instead of `0..visible_count`.

In `MenuComponent::handle_events()`, read `ctx.viewport().map(|v| v.offset_y).unwrap_or(0)`.
Add `offset_y` to the computed item index.

Both changes must use the SAME offset_y from the same viewport — symmetric math.

## Fix 2: Mouse click coordinate geometry
The palette is centered via PlacementContainerComponent. The `ctx.screen_area()`
passed to handle_events is the full managed_area, not the palette's actual position.

**Fix:** Cache the list_area rect during render:

**File:** `crates/term-wm-ui-components/src/command_palette.rs`
- Add `last_list_area: Option<LayoutRect>` field
- In `render()`, after computing list_area, store it in `self.last_list_area`
- In `handle_events()` for mouse events:
  ```rust
  if let Some(list_rect) = self.last_list_area {
      let list_ctx = ctx.clone().with_screen_area(list_rect);
      let result = self.list_scroll.handle_events(event, &list_ctx);
      self.sync_selected();
      return result;
  }
  ```

## Fix 3: Header gap
Suppressed by Fix 1a — set `show_header = false` on the MenuComponent used
inside the palette.
