## Issue 2: Scrollbar when items overflow
**File:** `crates/term-wm-ui-components/src/command_palette.rs`

### 2a. Add `scroll_offset: usize` field
### 2b. In `render_content`, after rendering items, call `render_scrollbar` if items
    exceed visible height.
### 2c. In `update` for MenuUp/MenuDown, adjust `scroll_offset` when selection
    exits the visible window.
### 2d. **CRITICAL — Fix pointer geometry in mouse handler:**
    The `handle_events` mouse click branch must apply `scroll_offset`:
```rust
let visual_idx = (my.saturating_sub(area.y.max(0) as u16).saturating_sub(1)) as usize;
let actual_idx = visual_idx.saturating_add(self.scroll_offset);
if actual_idx < self.filtered_items.len() {
    self.selected = actual_idx;
    return EventResult::Action(TermWmAction::MenuSelect);
}
```
    This replaces the current `EventResult::Consumed` return — mouse clicks
    now emit `Action(MenuSelect)` instead of `Consumed`, so the event pipeline
    correctly propagates the execution intent to the runner.

## Issue 3: Mouse click doesn't trigger action
**File:** `crates/term-wm-sys-ui-components/src/wm_command_palette.rs`

Revert the previous hack on `EventResult::Consumed`. With Issue 2d fixed,
the inner palette now returns `Action(MenuSelect)` on mouse click. The existing
`Action(MenuSelect)` branch in `WmCommandPaletteComponent::handle_events`
already handles this correctly:
```rust
TermWmAction::MenuSelect => {
    self.palette.update(action, ctx, &mut VecDeque::new());
    self.last_action = self.palette.inner().selected_action().cloned();
    EventResult::Consumed  // runner queries SelectedAction → dispatches
}
```
No change needed to the `Consumed` branch. Leave it as:
```rust
EventResult::Consumed => EventResult::Consumed,
```

    has_focused_window: bool,  // NEW

        items.extend_from_slice(&[
