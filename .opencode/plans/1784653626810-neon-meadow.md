# Fix: Synchronous Parser Resize on Terminal Dimension Change

## Bug

When the terminal is resized to a smaller dimension, the shell prompt can
cascade/duplicate down the screen. Root cause: after PTY resize, the
`vt100::Parser` used for rendering is **not resized synchronously** — the
resize is deferred to the reader thread via `pending_resize`, which processes it
asynchronously on the next PTY read. Meanwhile, `render_screen()` immediately
reads the old parser at wrong dimensions, producing incorrect cursor positioning
and cell access.

## Fix

Add a synchronous `set_size()` call on `shared_parser` in `Pty::resize()`,
matching the pattern already used by `RemotePane::resize()` and `TestPane`.

### File to modify

**`crates/term-wm-pty-engine/src/pty.rs`** — in `Pty::resize()` (line ~234),
after `self.apply_resize(size)`, lock `self.shared_parser` and call
`parser.screen_mut().set_size(size.rows, size.cols)`.

This ensures that when `TerminalComponent::render_screen()` locks the parser
(immediately after `pane.resize()` returns), it sees a parser whose dimensions
match the current rendering area.

### What stays unchanged

- The reader thread's `pending_resize` → history-replay path is kept as-is for
  scrollback content accuracy across resize boundaries.
- `TerminalComponent`, `Pane` trait, all other `Pane` impls — no changes needed.
- `RemotePane` already does synchronous `set_size` in its `resize()`.
