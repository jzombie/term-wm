# Plan: PlacementContainerComponent + Panel Overlay Rendering

## Problems
1. Monocle mode sets `panel_active() = false` → top panel gets zero height → hamburger icon rect unavailable → `anchor` is `None` → command palette's `render_dropdown` bails out. FAB menu invisible.
2. The `render_modal` free function violates the component architecture.

## Solution
1. **Create `PlacementContainerComponent<C>`** — a proper Component that encapsulates BOTH layout strategies (centered modal + anchored dropdown) and always delegates to inner `C::render`
2. **Remove `render_dropdown` from overlay** — overlay unconditionally uses the container's unified `render()` and `handle_events()`
3. **No `Deref`** — overlay accesses inner menu via explicit `inner()`/`inner_mut()` methods
4. **No `panel_active()` change** — instead, render panels explicitly from the overlay rendering phase

## Architecture

### 1. `PlacementContainerComponent<C>` in `crates/term-wm-ui-components/src/placement_container.rs`

```rust
pub enum Placement {
    Centered { width: u16, height: u16 },
    Anchored { x: u16, y: u16, managed_area: LayoutRect, content_width: u16, content_height: u16 },
}

pub struct PlacementContainerComponent<C> {
    content: C,
    placement: Placement,
    content_rect_cache: Cell<Option<LayoutRect>>,
}
```

**`Component<TermWmAction>` impl — `render()`:**
1. Match on `self.placement`:
   - `Centered { width, height }`: compute rect via `Layout::vertical([Constraint::Length(h)]).flex(Flex::Center)` + horizontal
   - `Anchored { x, y, managed_area, content_width, content_height }`: compute rect at (x,y), clamp to `managed_area`
2. Render backdrop via `DialogOverlayComponent::render_backdrop` over managed_area, excluding content_rect
3. Apply `Clear` on content_rect
4. Cache content_rect in `content_rect_cache`
5. **Register hitbox** for `content_rect` in `registry` so `dispatch_mouse` sets `ctx.screen_area()` to the menu bounds on mouse events
6. Delegate to `self.content.render(backend, content_layout_lr, ctx, registry)` — inner component registers item-level hitboxes if needed

**`Component<TermWmAction>` impl — `handle_events()`:** delegate to `self.content.handle_events(event, ctx)`

**`Component<TermWmAction>` impl — `update()`/`destroy()`:** delegate to content

**Additional methods:**
- `new(content: C) -> Self`
- `set_placement(&mut self, placement: Placement)`
- `content_rect(&self) -> Option<LayoutRect>`
- `inner(&self) -> &C` / `inner_mut(&mut self) -> &mut C` (explicit, not Deref)

### 2. Modify `MenuComponent::render` and `handle_events` (`menu.rs`)

**`render()` (line 183-194):**
Compute hovered_idx from `ctx.hover_pos()` instead of `None`. Hover is based on `ctx.hover_pos()` (screen coordinates) vs the component's render area (screen coordinates), transforming to local item index. This enables hover highlighting via Component::render when the container delegates to it.

**`handle_events()` (line 196-202):**
Add mouse event handling alongside existing keyboard handling:
- When `Event::Mouse(mouse)` with `MouseEventKind::Press(_)`:
  - Use `ctx.screen_area()` to get the menu's screen-space bounds
  - Compute item index from `mouse.row` relative to `screen_area.y`
  - Set `self.selected` and return `EventResult::Action(TermWmAction::MenuSelect)`
- Keyboard events continue through existing `self.handle_key_event(event)` path

This makes `MenuComponent` the sole authority for mouse-to-item mapping, eliminating the need for the parent overlay to reconstruct geometry.

### 3. Refactor `WmCommandPaletteOverlay` (`wm_menu_overlay.rs`)

- Replace `menu: MenuComponent` field with `menu: PlacementContainerComponent<MenuComponent>`
- **Remove** `item_hits: RefCell<Vec<(usize, LayoutRect)>>` field (no longer tracked at overlay level)
- **Remove** `hit_test_item()` method
- Replace the entire `render_dropdown` method with `PlacementContainerComponent` usage:
  - Compute placement (Anchored from `self.anchor` or Centered when anchor is None)
  - Call `self.menu.set_placement(placement)`
  - Call `self.menu.render(backend, full_area, ctx, registry)` — backdrop + content + hitbox registration handled by container
- In `render()` (Component trait): compute content dimensions, set placement, call `self.menu.render(...)`
- In `handle_events()`:
  - Delegate to `self.menu.handle_events(event, ctx)`
  - On `EventResult::Action(TermWmAction::MenuSelect)`: restore outline, capture `last_action` from `self.menu.inner().selected_action()`, return `Consumed`
  - On keyboard events: delegate to `self.menu.handle_events(event, ctx)`, let MenuComponent update selected index via `MenuUp`/`MenuDown` actions
- In `update()`: delegate to `self.menu.update(action, ctx, actions)`
- In `process_action()`: set menu items via `self.menu.inner_mut().set_items(items)`
- In `query()`: read selected action via `self.menu.inner().selected_action()`

### 4. Panels render as overlays from overlay phase (not layout)

**Do NOT modify `WindowManager::panel_active()`** (defined at `mod.rs:1996`).

**Current render order** (`src/lib.rs`):
1. Windows + decorators (lines 180-217)
2. `render_panels(backend, wm)` (line 221) — this sets `active=false` on top panel in monocle mode
3. FAB component (line 224)
4. Tiling split handles (line 233)
5. Notification toasts (line 386)
6. Drop shadows (line 396)
7. `render_overlays(backend, wm)` (line 407) — command menu + overlays
8. Notification hitboxes (line 409)
9. Cursor overlay (line 420)

**Modification to `render_overlays` (in `draw_plan_renderer.rs:480`):**
- When `wm.is_monocle() && wm.command_menu_visible()`: re-render the top panel at the top of the screen with height 1
- The panel's area is computed as `LayoutRect { x: 0, y: 0, width: full_area.width, height: 1 }` (not from `top_claimed_area` which is zero)
- **CRITICAL**: Do NOT rely on `ComponentContext::new(true)` — `WmTopPanelComponent` uses internal `self.active` state set via `ComponentAction::SetPanelActive`. The `ComponentContext`'s `active` field has no effect on the panel's render guard (`wm_top_panel.rs:462`: `if !self.active { return; }`).
- Instead, **inject synchronous state mutations** via explicit `process_action` calls:
  ```rust
  // Force active state for overlay render
  top_panel.process_action(&ComponentAction::SetPanelActive(true));
  top_panel.render(backend, top_area, &ctx, &mut local_hb);
  // Revert to layout-derived state (will be preserved for next frame)
  top_panel.process_action(&ComponentAction::SetPanelActive(false));
  ```
- This renders panels ON TOP of the maximized content without triggering a layout recalculation. The panel component already handles re-rendering gracefully.

Note: The anchor for the menu will still be None (hamburger icon rect unavailable since layout didn't compute it), so the `PlacementContainerComponent` will use `Placement::Centered` in monocle mode. The panel overlay rendering is a visual enhancement to show the top bar context.
