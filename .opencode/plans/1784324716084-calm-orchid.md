# Plan: ModalContainerComponent + Panel Overlays in Mobile Mode

## Problem
1. When viewport is narrow (<80 cols), monocle mode sets `panel_active() = false`, giving the top panel zero height. The hamburger icon rect becomes unavailable → `anchor` is `None` → command palette's `render_dropdown` bails out. The FAB menu is invisible.
2. The previous `render_modal` free function violates the component architecture.

## Solution
1. **Create `ModalContainerComponent<C>`** — a proper Component (generic wrapper, like `CenterComponent`)
2. **Modify `MenuComponent::render`** to compute hover from `ctx.hover_pos()`
3. **Convert `WmCommandPaletteOverlay`** to store `ModalContainerComponent<MenuComponent>` and use it in mobile mode
4. **Allow panels to show as overlays** when command palette is visible in monocle mode

## Architecture

### 1. `ModalContainerComponent<C>` in `crates/term-wm-ui-components/src/modal_container.rs`

Generic wrapper around `C: Component<TermWmAction>` (same pattern as `CenterComponent<C>`):

```rust
pub struct ModalContainerComponent<C> {
    content: C,
    target_width: u16,
    target_height: u16,
    content_rect_cache: Cell<Option<LayoutRect>>,
}
```

**`Component<TermWmAction>` impl:**
- `render()`: compute centered rect via `Layout::vertical([Constraint::Length(h)]).flex(Flex::Center)` + horizontal; render backdrop via `DialogOverlayComponent::render_backdrop`; apply `Clear`; delegate to `self.content.render(backend, content_layout, ctx, registry)`; cache `content_layout` in `content_rect_cache`
- `handle_events()`: delegate to `self.content.handle_events(event, ctx)`
- `update()`: delegate to `self.content.update(action, ctx, actions)`
- `destroy()`: delegate to `self.content.destroy()`

**Additional methods:** `new(content, w, h)`, `set_target_size(w, h)`, `content_rect() -> Option<LayoutRect>`

**`Deref`/`DerefMut` to `C`:** enables transparent access to inner component methods (e.g., `self.menu.items()`, `self.menu.render_items()`).

### 2. Modify `MenuComponent::render` (`menu.rs:183-194`)

Change from passing `None` for `hovered_idx` to computing it from `ctx.hover_pos()`:

```rust
fn render(&mut self, backend, area, ctx, _registry) {
    let area = layout_rect_to_rect(area);
    let backend = downcast_ratatui(backend);
    let hovered_idx = ctx.hover_pos()/* compute relative hover */;
    self.render_items(&mut backend.buffer, area, hovered_idx, &ctx.config().theme);
}
```

This enables hover highlighting via Component::render (used by the modal container's delegation in mobile mode).

### 3. Modify `WmCommandPaletteOverlay` (`wm_menu_overlay.rs`)

- Change `menu: MenuComponent` → `menu: ModalContainerComponent<MenuComponent>`
- **Desktop mode** (anchor is `Some`): unchanged — calls `self.menu.render_items(...)` via Deref, same custom positioning + hover + hit recording
- **Mobile mode** (anchor is `None`): set target dimensions via `self.menu.set_target_size(w, h)`, call `self.menu.render(backend, self.managed_area, ctx, registry)` (ModalContainerComponent's render), read content rect from `self.menu.content_rect()`, store in `menu_bounds_cache`, record hit rects (same code as desktop but using content rect instead of anchor rect)
- Remove `render_modal` import
- Remove `MIN_MODAL_WIDTH` const (moved into modal_container)

### 4. Show panels as overlays when command palette is open

Modify `WindowManager::panel_active()` in `crates/term-wm-core/src/window/window_manager/mod.rs:1996-2001`:

```rust
pub fn panel_active(&self) -> bool {
    if self.is_monocle() && !self.command_menu_visible() {
        return false;
    }
    self.config.panel_enabled && self.top_component.as_ref().is_some_and(|p| p.visible())
}
```

When the command palette is open in monocle mode, `panel_active()` returns `true`. This makes the top panel claim its normal height and render on top of the maximized content (exactly what the user wants — "show as overlays on top of existing content").
