# Plan: Better Application Lifecycle Management (Item 2 — Revised)

## Goal
Overhaul application lifecycle management to prevent zombie processes, memory leaks, phantom rendering, and deadlocks. Introduce a Generational Arena (Slotmap) as the single source of truth, a dedicated **Reaper** for async process teardown, a unified event proxy to eliminate polling duality, and fix the event loop to not deadlock on IO boundaries.

---

## Phase 1: Slotmap as the Single Source of Truth (No Split-Brain)

Replace all `WindowId<Id>` references across the entire app with `SlotMap`'s `WindowKey`. Delete the `BTreeMap<Id, WindowKey>` indirection — the `WindowKey` *is* the external ID. This eliminates source-of-truth fragmentation entirely.

### SlotMap Key Design
```rust

// It's Copy, Send, Sync, Debug, 8 bytes. Perfect for this use case.
type WindowKey = slotmap::DefaultKey;

// WindowId becomes a simple wrapper or we just use WindowKey directly everywhere.
// Since slotmap has its own generation tracking, we don't need WindowId<Id> at all.
```

### Danger Zones

- **External API break**: Every crate boundary that passes `Id` or `WindowId<Id>` now uses `WindowKey`. This affects `WindowProvider`, `WindowManagerHost`, `WindowDrawContext`, and all user code. The compiler will flag every call site — treat the error list as the migration checklist.
- **System windows**: `SystemWindowId` (DebugLog) becomes a regular `WindowKey` allocated from the same SlotMap. No separate storage.
- **Closed-window tracking**: Remove `closed_app_windows` Vec. When `close_window()` removes from SlotMap, the `Result<_, SlotMapError>` is handled immediately. The embedded app (main.rs) observes the missing key on the next render cycle and removes from `self.terminals` naturally.

## Phase 2: The Reaper — Async Process Teardown (Not in Drop)

Create a dedicated `Reaper` struct that owns the child process handles and PTY file descriptors for recently-closed windows. The main loop periodically polls the Reaper to reap dead processes asynchronously.

**Why not `Drop`?** Blocking in `Drop` freezes the UI during a 3-second SIGHUP→SIGKILL chain. Spawning threads inside `Drop` creates orphans during bulk teardown. The Reaper keeps teardown off the main thread and non-blocking.

### Design

```rust
// New file: crates/term-wm-core/src/reaper.rs

struct ZombieChild {
    child: Box<dyn Child + Send + Sync>,
    reader_handle: Option<JoinHandle<()>>,
    pty_fd: RawFd,  // or the reader/sender channel
    killed_at: Instant,
    sigkill_sent: bool,  // true after SIGHUP→timeout→SIGKILL
}

struct Reaper {
    zombies: Vec<ZombieChild>,
    shutdown_timeout: Duration,
}

impl Reaper {
    /// Take ownership of a child and its reader thread.
    /// Called by WindowManager when a window is closed.
    fn reap(&mut self, zombie: ZombieChild);

    /// Non-blocking. Called every idle tick (~16ms).
    /// 1. Calls child.try_wait() on each zombie.
    /// 2. If child exited: close PTY FD → reader thread unblocks → join reader
    /// 3. If timeout elapsed since killed_at && !sigkill_sent: send SIGKILL
    /// 4. Remove fully reaped zombies from the Vec.
    fn tick(&mut self);

    /// Called during app exit. Blocking but bounded.
    /// Forces SIGKILL on all remaining children, joins all threads.
    fn drain_all(&mut self);
}
```

### ⚠ SIGHUP vs SIGKILL Timing

`killed_at` must be recorded **after** the initial SIGHUP is issued, not when the `ZombieChild` struct is created. If a user closes 10 windows at once, 10 `ZombieChild` entries are created nearly simultaneously. The grace period (e.g., 3 seconds) starts counting from `killed_at`, so each process gets its full 3-second window regardless of how many close at once.

```rust
struct ZombieChild {
    child: Option<Box<dyn Child + Send + Sync>>,
    reader_fd: Option<RawFd>,
    reader_handle: Option<JoinHandle<()>>,
    sighup_sent: bool,     // false initially; set to true when first SIGHUP sent
    sigkill_sent: bool,    // true after timeout → SIGKILL
    killed_at: Option<Instant>, // recorded when sighup_sent transitions false→true
}
```

On first tick where `!sighup_sent`:
1. Send SIGHUP to child
2. Set `sighup_sent = true`
3. Set `killed_at = Some(Instant::now())` — grace period starts NOW

On subsequent ticks where `!sigkill_sent && killed_at + timeout < Instant::now()`:
1. Send SIGKILL
2. Set `sigkill_sent = true`

### Reaper Flow

```
Window closed by user
  ↓
close_window(key) removes from SlotMap
  ↓
Window data moved to Reaper::zombies (child, reader_fd, reader_handle)
  ↓
main loop idle tick (~16ms intervals)
  ↓
Reaper::tick():
  for each zombie:
    if !sighup_sent:
      child.kill() → SIGHUP      // portable-pty sends SIGTERM on Unix
      sighup_sent = true
      killed_at = Some(Instant::now())
      continue                    // give it time, check next tick

    if let Some(status) = child.try_wait():
      // Child exited naturally
      close_fd(reader_fd)         // close FD → unblocks reader thread
      reader_handle.join()        // safe — FD closed
      remove from zombies

    if !sigkill_sent && killed_at + timeout < Instant::now():
      child.kill() → SIGKILL      // force kill
      sigkill_sent = true
      // next tick: try_wait() will succeed, FD close + join
```

## Phase 3: Unified Event Proxy (No Polling Duality)

Currently, the main thread blocks on `crossterm::event::read()` for keyboard input but also needs to receive PTY update pings. This forces either polling (`poll()`) or a split-brain event source.

**Fix**: A dedicated input thread reads crossterm events and forwards them into a unified `Receiver`. The PTY reader threads also send their wakeup pings into the same `Receiver`. The main loop blocks on a single `recv()` call.

### Design

```rust
// Channel primitive: crossbeam_channel::bounded(cap)
// - Multi-producer: Sender is Send + Sync + Clone — each thread gets an owned clone
// - Bounded: if PTY reader floods faster than main loop drains, sender blocks →
//   backpressure propagates through OS pipe to child process
// - Blocking recv() on main thread: sleep until event arrives, zero polling

// In crates/term-wm-core/src/io/mod.rs or new file

enum UnifiedEvent {
    Input(crossterm::event::Event),   // from crossterm
    PtyWakeup(WindowKey),              // from PTY reader (zero-payload)
    Signal(SignalKind),                // from signal handler
    Tick,                              // periodic idle tick timer
}

struct UnifiedEventSource {
    rx: Receiver<UnifiedEvent>,
    // internal: crossterm reader thread handle,
    //            signal handler registration,
    //            tick timer channel
}

impl EventSource for UnifiedEventSource {
    fn poll(&mut self) -> bool {
        // non-blocking check of rx
    }
    fn read(&mut self) -> Option<crossterm::event::Event> {
        // Block on rx, convert UnifiedEvent back or
        // directly return UnifiedEvent as the new Event type
    }
}
```

### ⚠ Crossterm Input Thread — Shutdown Safety

The dedicated input thread must not block forever on `crossterm::event::read()`. If it does, joining the thread during shutdown requires a coincidental keystroke — otherwise it hangs indefinitely.

**Fix**: Use `crossterm::event::poll(timeout)` in a loop instead of blocking `read()`:

```rust
// Input thread pattern — exits within 100ms of shutdown signal
let shutdown = Arc::new(AtomicBool::new(false));
// clone shutdown flag, pass clone to UnifiedEventSource for signaling on quit

thread::spawn(move || {
    loop {
        if shutdown.load(Ordering::Acquire) {
            break;  // exit within 100ms
        }
        if crossterm::event::poll(Duration::from_millis(100)).unwrap_or(false) {
            let event = crossterm::event::read().unwrap();
            if event_tx.send(UnifiedEvent::Input(event)).is_err() {
                break;  // receiver dropped — app is shutting down
            }
        }
        // poll returning false is fine — loop back and check shutdown flag
    }
});
```

The `100ms` timeout balances responsiveness (~10 polls/sec overhead when idle, which is negligible) against guaranteed thread exit during shutdown. The `shutdown` `AtomicBool` is set by the event loop on `ControlFlow::Quit` — the input thread observes it within one poll cycle and exits.

### ⚠ PtyWakeup Flood Protection

Reader threads send zero-payload `UnifiedEvent::PtyWakeup(key)` on every mutation cycle. A background process dumping megabytes of text can spam thousands of wakeups per second into the channel.

**Do NOT render on every wakeup.** The main loop must:

1. Drain all pending events in a batch (non-blocking `try_recv` loop).
2. Accumulate `dirty` flags per-window as wakeups arrive — do not render inline.
3. Issue a **single composite render** at the end of the batch drain.

This is the same coalescing principle from the WezTerm 3ms delay pattern — batch the erratic IO bursts into one visual frame. The `dirty` flags survive across event batches so that even if no event arrives between PTY mutations, the next wakeup triggers a single render that picks up all accumulated changes.

Rationale: We use `crossbeam_channel` (not `std::sync::mpsc`) because:
- `Sender` is `Send + Sync + Clone` — each PTY reader thread gets an owned clone with zero shared-state coordination
- Bounded send + `Sync` == safe, zero-unsafe, well-audited concurrency
- `crossbeam::select!` available if we later need to multiplex a tick timer

## Phase 4: Reader Thread Safe Teardown (Close FD Before Join)

The reader thread blocks on `read(fd, buf)`. Sending SIGKILL to the child does NOT close the master FD. The thread stays blocked.

**Fix**: Before joining any reader thread, close the PTY master FD. This causes `read()` to return `EIO` (EOF on master) or `EBADF`, which unblocks the thread. Only then call `join()`.

### ⚠ FD Ownership — Prevent Double-Close

The underlying `portable-pty` wrapper or `std::fs::File` may have a `Drop` impl that also closes the same FD. If we call `libc::close()` while the wrapper still holds the FD, the `Drop` of the wrapper will later close the same FD number again. Worse, by that time the OS may have reassigned that FD number to a newly-opened PTY, so the double-close silently closes an active connection.

**Fix**: Use `IntoRawFd` to take full ownership of the raw descriptor away from the wrapper, consuming it into a `std::fs::File` or calling `libc::close()` as the sole owner. The original wrapper must be left in a state where its `Drop` is a no-op (or the wrapper is consumed/mem::forgotten).

```rust
// Safe pattern — single ownership of the FD
use std::os::unix::io::{IntoRawFd, FromRawFd};

// Take ownership of the FD from the reader wrapper
let raw_fd = reader.into_raw_fd();  // wrapper is consumed, Drop won't close it
// Close the FD — reader thread's read() gets EBADF and exits
unsafe { libc::close(raw_fd); }
// Now join the reader thread (it has already unblocked)
reader_handle.join().ok();
```

On Windows, the equivalent is `IntoRawHandle` + `CloseHandle`.

| File | Change |
|---|---|
| `crates/term-wm-pty-engine/src/pty.rs` | `read_loop()` checks for `EIO`/`EBADF` as normal exit conditions (not just `read() == 0`). Expose reader FD via `IntoRawFd` — consume it to prevent double-close. |
| `crates/term-wm-pty-engine/src/pane.rs` | `fn into_reader_fd(self) -> RawFd` — consumes the reader wrapper, returns owned FD. |

### read_loop Exit Conditions
```rust
// Current: break on read() == 0 (EOF) or error
// Fixed: also break on EIO (master closed) and EBADF (FD closed)
loop {
    match reader.read(&mut buf) {
        Ok(0) => break,            // EOF — child exited
        Ok(n) => process_bytes(&buf[..n]),
        Err(e) if e.kind() == ErrorKind::Interrupted => continue,
        Err(e) if e.kind() == ErrorKind::UnexpectedEof => break,  // EIO on Unix (master closed)
        Err(e) => break,           // EBADF (FD explicitly closed by reaper)
    }
}
```
