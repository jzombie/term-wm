# Plan: Better Application Lifecycle Management (Item 2)

## Goal
Overhaul application lifecycle management to prevent zombie processes, memory leaks, phantom rendering, and deadlocks. Introduce a Generational Arena (Slotmap) as the single source of truth for window state, proper thread lifecycle for PTY readers, integrated signal handling for graceful shutdown, and an async PTY teardown chain.

---

## Phase 1: Generational Arena for Window Storage

Replace `BTreeMap<WindowId<Id>, Window>` with a `SlotMap`-based generational arena for all authoritative window data. This eliminates zombie windows by making stale keys mathematically impossible to resolve.

### Changes

| File | Change |
|---|---|
| `crates/term-wm-core/Cargo.toml` | Add `slotmap = "1.1"` dependency |
| `crates/term-wm-core/src/window/entry.rs` | Convert `Window` into a plain data struct (no complex cleanup), add `Drop` impl that sends SIGHUP/SIGKILL |
| `crates/term-wm-core/src/window/window_manager/mod.rs` | Replace `windows: BTreeMap<WindowId<Id>, Window>` with `windows: SlotMap<WindowKey, Window>` and `window_ids: BTreeMap<Id, WindowKey>` for mapping app IDs to keys. Use `generational_key2()` for 2-generational safety. |
| `crates/term-wm-core/src/window/window_manager/chrome.rs` | Update `close_window()` to remove from SlotMap (instantly invalidates all keys). Remove `closed_app_windows` Vec — handle teardown synchronously via `Window::drop`. |
| `crates/term-wm-core/src/window/window_manager/focus.rs` | Update focus tracking to use `WindowKey` references where possible |
| `crates/term-wm-core/src/window/window_manager/layout.rs` | Layout trees hold `WindowKey` (or `Id`), never actual `Window` data |
| `crates/term-wm-core/src/window/window_manager/overlays.rs` | Update overlay rendering to use `SlotMap` lookups |
| `crates/term-wm-core/src/runner.rs` | Remove `take_closed_app_windows()` flow — `Window::drop` now handles teardown atomically |

### Design

```rust
// New in entry.rs
struct Window {
    id: Id,
    title: String,
    child_pid: u32,
    pty_handle: Box<dyn Child + Send + Sync>,
    text_grid: Arc<Mutex<TextGrid>>,
    // ... metadata fields
}

impl Drop for Window {
    fn drop(&mut self) {
        // 1. Send SIGHUP (or SIGTERM on Windows)
        // 2. Spawn a short timeout thread that sends SIGKILL if not dead
        // 3. Close PTY file descriptors (via portable_pty)
    }
}
```

### Danger Zones

- **WindowId<Id> currently wraps algorithm IDs**: The `WindowId<Id>` enum has `App(Id)` and `System(SystemWindowId)` variants. We need to map these to SlotMap keys. Strategy: keep `WindowId<Id>` as the external API, maintain an internal mapping `BTreeMap<Id, WindowKey>` and `BTreeMap<SystemWindowId, WindowKey>`.
- **Closed_app_windows removal**: Currently `close_window()` pushes to `closed_app_windows` which is drained in the next idle tick's `flush_state_changes()`. With Slotmap, removing from the arena is instant — dependent systems get `None` on lookup. But we still need to call `app.wm_close_window(id)` for application-level cleanup (removing from `self.terminals`). Solution: keep the deferred cleanup for app-level resources, but let the SlotMap entry be removed immediately. The `containing_apps` vec or similar can track which app slots need clearing.
- **Iteration patterns**: Replace `.windows.iter()` with `.windows.iter()` on the SlotMap, filtering through `window_ids` for app-id lookups.

## Phase 2: PTY Thread Lifecycle Management

Currently, each PTY spawns a reader thread stored as `_reader: JoinHandle<()>` which is **never joined** (detached on drop). This means threads can outlive their parent `Pty` struct.

### Changes

| File | Change |
|---|---|
| `crates/term-wm-pty-engine/src/pty.rs` | Change `_reader: JoinHandle<()>` → `reader: Option<JoinHandle<()>>`; add `join_reader()` method that panics if thread panicked; call on teardown |
| `crates/term-wm-pty-engine/src/pane.rs` | Add `fn join_reader(&mut self) -> io::Result<()>` to `Pane` trait |
| `crates/term-wm-ui-components/src/terminal.rs` | Call `pane.join_reader()` in `terminate()` after `kill_child()` |
| `crates/term-wm-pty-engine/src/lib.rs` | Export new `Pane` method |

### Bounded Channel for Thread Communication

Introduce a bounded channel for control events between PTY reader and main thread:
- Reader thread sends **zero-payload wakeup pings** via a `Sender<()>` (bounded to ~64 items)
- Reader thread mutates text grid **in place** via `Arc<Mutex<TextGrid>>` (shared with main thread)
- Reader thread sets `AtomicBool` dirty flag after each mutation
- Main thread receives wakeup pings, checks dirty flags, and drains

This replaces the current `Arc<Mutex<Vec<u8>>> pending` pattern where bytes are copied across threads.

### Changes

| File | Change |
|---|---|
| `crates/term-wm-pty-engine/src/pty.rs` | Add `control_tx: crossbeam::bounded::Sender<()>`; reader thread sends ping after each mutation; remove `pending: Arc<Mutex<Vec<u8>>>` |
| `crates/term-wm-pty-engine/src/pane.rs` | Add `fn drain_updates(&mut self)` to `Pane` trait — called from main thread to apply grid changes |

## Phase 3: Integrated Signal Handling for Graceful Shutdown

Currently, SIGINT/SIGTERM is handled by crossterm as a regular `Event::Key(Ctrl+C)`. There's no OS-level signal handler integration, and no graceful shutdown sequence.

### Changes

| File | Change |
|---|---|
| `src/main.rs` | Install `install_sigint_handler()` before `run_window_app`; check `SigintHandle::received()` in the event loop |
| `crates/term-wm-core/src/runner.rs` | Add SIGINT/SIGTERM check in the idle path (before the empty-windows quit check). When received: if no exit confirm is open, call `app.open_exit_confirm()`. On second SIGINT within 3s, force quit immediately. |
| `crates/term-wm-core/src/io/console.rs` | Implement graceful shutdown in `exit()`: disable raw mode, leave alt screen, show cursor — already done, but ensure it's called even on `SIGKILL` by keeping `Drop` impl. |

### Shutdown Sequence

```
SIGINT/SIGTERM received
  ↓
If exit confirm not visible → show it
If exit confirm visible (second signal within 3s) → force quit
  ↓
User confirms or force quit:
  1. Set AtomicBool "shutting_down" = true
  2. Close all windows → triggers Window::drop → SIGHUP each child
  3. Wait up to 3s for children to exit
  4. SIGKILL remaining children
  5. Join all reader threads (with timeout)
  6. Return ControlFlow::Quit
  7. output.exit() restores terminal state
```

## Phase 4: Window State Machine

Define canonical window lifecycle states and enforce transitions through the Slotmap. Based on the plan documents:

### States
- **Realized**: Window metadata allocated in SlotMap, no rendering intent (invisible)
- **Mapped**: Window visible, geometry assigned, rendered in layout
- **Unmapped/Withdrawn**: Programmatically hidden, no spatial footprint, but still in SlotMap memory
- **Iconic/Minimized**: Mapped but hidden from primary workspace layout
- **Destroyed**: Removed from SlotMap, all resources freed

### Changes

| File | Change |
|---|---|
| `crates/term-wm-core/src/window/entry.rs` | Add `WindowState` enum to `Window` struct with transition methods |
| `crates/term-wm-core/src/window/window_manager/chrome.rs` | Enforce state transitions in `close_window()`, `minimize_window()`, `toggle_maximize()` |
| `crates/term-wm-core/src/window/window_manager/mod.rs` | State transitions go through a single `set_window_state()` method that validates transitions |
| `crates/term-wm-core/src/runner.rs` | Skip render for non-Mapped windows (occlusion culling) |

### State Machine
```rust
enum WindowState {
    Realized,     // allocated, not visible
    Mapped,       // visible in layout
    Unmapped,     // hidden, no spatial footprint
    Iconic,       // minimized
    Destroyed,    // about to be freed
}

// Valid transitions:
// Realized → Mapped | Unmapped
// Mapped → Unmapped | Iconic | Destroyed
// Unmapped → Mapped | Destroyed
// Iconic → Mapped | Destroyed
// Destroyed → (terminal, cannot transition out)
```
