# Plan: Fix Terminal Resize Line Duplication Bug

## The Bug
When a terminal pane is resized smaller, lines get duplicated (the same line appearing dozens of times). The user specifically sees this when shrinking past a `%` character — the line containing `%` gets repeated.

## Root Cause
**File:** `crates/term-wm-pty-engine/src/pty.rs`, lines 509-517

The reader thread handles resize by creating a **completely new `vt100::Parser`** at the new (smaller) size and **replaying all accumulated PTY history** through it:

```rust
let mut new_parser = vt100::Parser::new(size.rows, size.cols, scrollback_len);
new_parser.process(&history);
let mut shared = shared_parser.lock().unwrap();
*shared = new_parser;
```

This is fundamentally broken because:
- The `history` buffer contains ALL PTY data since spawn (shell output, pico/nano content, etc.)
- Escape sequences in `history` (cursor positioning, scroll regions, clear-screen) assumed the OLD terminal size
- Replaying at a smaller size produces **different visual content** — lines that fit at 80 cols now wrap at 40 cols, and the vt100 reflow logic can place the same content on multiple rows
- The replacement happens asynchronously, so the render thread may read from a parser whose scrollback offset, screen content, and dimensions are all inconsistent with what the previous frame used

## The Fix
Replace the history-replay with vt100's built-in `Screen::set_size()`:

```rust
// Instead of creating new parser and replaying history
let mut shared = shared_parser.lock().unwrap();
shared.screen_mut().set_size(size.rows, size.cols);
```

**Why this works:** vt100 0.16.2's `Screen::set_size()` properly resizes the existing terminal screen, reflows content correctly, and preserves the scrollback buffer. This is the API vt100 provides for resize — it's already used correctly in `remote_pane.rs:80` and `TestPane::set_parser_size` in `terminal.rs:1106-1108`.

## Files to Modify
1. **`crates/term-wm-pty-engine/src/pty.rs`** (lines 509-517) — Replace the history-replay resize with `set_size()`
