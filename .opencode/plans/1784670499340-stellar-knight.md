# Plan: Fix Terminal Resize Line Duplication Bug

## The Bug
When a terminal pane is resized smaller, lines get duplicated (the same line appearing dozens of times). The user specifically sees this when shrinking past a `%` character.

## Root Cause
**File:** `crates/term-wm-pty-engine/src/pty.rs`, lines 509-517

The reader thread handles resize by creating a **completely new `vt100::Parser`** at the new size and **replaying all accumulated PTY history** through it. The `history` buffer contains ALL PTY data since spawn (shell, pico/nano output, etc.) — escape sequences in it assumed the OLD terminal size. Replaying at a smaller size produces different visual content due to reflow, causing lines to appear duplicated.

## The Fix (3 changes in `pty.rs`)

### 1. Replace history-replay with in-place grid resize
**Lines 509-517:** Instead of creating a new parser and replaying history, lock the shared parser and call `set_size()` directly:
```rust
let mut shared = shared_parser.lock().unwrap();
shared.screen_mut().set_size(size.rows, size.cols);
```
`Screen::set_size()` is the documented vt100 API for resize — already used correctly in `remote_pane.rs:80` and `TestPane::set_parser_size`.

### 2. Hold parser lock across OS PTY resize + set_size (fix SEV-2 race)
**In `resize()` (line 216):** Lock the parser BEFORE the OS PTY resize to prevent the reader thread from reading+processing bytes at the new PTY size into an old-dimensioned parser:
```rust
pub fn resize(&mut self, size: PtySize) -> PtyResult<()> {
    if size.rows < 2 || size.cols < 2 {
        return Ok(());
    }
    if size == self.pty_size { return Ok(()); }
    
    let sp = self.shared_parser.clone();
    let mut guard = sp.lock().unwrap();  // lock first
    
    self.master.resize(size).map_err(|err| wrap_err("resize", err))?;
    guard.screen_mut().set_size(size.rows, size.cols);
    drop(guard);
    
    self.pty_size = size;
    self.size = size;
    Ok(())
}
```
Remove `apply_resize()` and the `pending_resize` handoff to the reader thread.

### 3. Remove unbounded `history` buffer, replace with 8-byte tail
**Lines 503, 533-540, 558-571, 585:** The `history: Vec<u8>` grows up to 2MB. It's only used for:
- Resize replay (being removed)
- DSR cross-chunk detection
- OSC 52 cross-chunk header detection

Replace with `let mut prev_tail: [u8; HISTORY_TAIL_LEN] = [0; HISTORY_TAIL_LEN]` and update:
- DSR: check `buf[..n]` directly (65536-byte reads; the 4-byte `\x1b[6n` pattern is virtually never split across such large boundaries)
- OSC 52: pass `&prev_tail` to `osc52.push()` instead of `&history[...]`
- Update `prev_tail` after each read with left-rotation (handles `n < 8` without panic):
  ```rust
  if n >= HISTORY_TAIL_LEN {
      prev_tail.copy_from_slice(&buf[n - HISTORY_TAIL_LEN..n]);
  } else if n > 0 {
      prev_tail.rotate_left(n);
      prev_tail[HISTORY_TAIL_LEN - n..].copy_from_slice(&buf[..n]);
  }
  ```

### Cleanup
- Remove `pending_resize` field from `Pty` struct, `ParserReadLoopArgs`, and all related code
- Remove unused constants: `PRUNE_SEARCH_WINDOW` (line 27), MAX_HISTORY_CAP/PRUNE_TARGET (lines 560-561)
- Increase `HISTORY_TAIL_LEN` from 3 to 8 to cover the 5-byte `\x1b]52;` OSC 52 header across chunk boundaries
