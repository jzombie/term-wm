# Fix: Text selection offset when floating window is dragged off-screen

## Bugs

1. **Bug 1 — Ghost hitboxes** (already fixed by user)
2. **Bug 2 — Cursor selection at wrong column**: When a floating window is partially off-screen left (negative `dest.x`), text selection starts at the beginning of the line instead of where the mouse is.

## Root Cause

`ctx.screen_area()` returns a signed `LayoutRect` (i32 x/y, u16 w/h). Throughout `terminal.rs`, this value is repeatedly converted to unsigned `ratatui::layout::Rect` (u16 x/y) via `layout_rect_to_rect()`. When `x` is negative, `x as u16` wraps around (underflow), producing coordinates like `65496`. This corrupts all subsequent coordinate math — `mouse.column.saturating_sub(area.x)` with `area.x = 0xFFF6` wraps to a huge value instead of a small positive offset.

**Conversion chain:**
1. `handle_events` line 172: `try_handle_link_click(layout_rect_to_rect(area), ...)`
2. `handle_events` line 197: `mouse.column.saturating_sub(area.x as u16)` — `area.x` is negative, `as u16` underflows
3. `SelectionViewport::logical_position_from_point` line 731: `let area = layout_rect_to_rect(area);`
4. `RenderDragHost::logical_position_from_point` line 963: `let area = layout_rect_to_rect(area);`
5. `link_at_position` line 905: takes `Rect` (unsigned), all bounds comparisons fail

## Fix

Keep all coordinate math in signed `LayoutRect` (i32). Remove every `layout_rect_to_rect` call in the selection/link paths.

### Changes in `crates/term-wm-ui-components/src/terminal.rs`

3. **`logical_position_from_point` (~line 780):** Accept `LayoutRect`, use i32 math.

```rust
fn logical_position_from_point(
    &mut self, area: LayoutRect, column: u16, row: u16,
) -> Option<LogicalPosition> {
    if area.width == 0 || area.height == 0 { return None; }
    let max_x = area.x.saturating_add(i32::from(area.width)).saturating_sub(1);
    let max_y = area.y.saturating_add(i32::from(area.height)).saturating_sub(1);
    let clamped_col = i32::from(column).clamp(area.x, max_x);
    let clamped_row = i32::from(row).clamp(area.y, max_y);
    let local_col = clamped_col.saturating_sub(area.x) as usize;
    let local_row = clamped_row.saturating_sub(area.y) as usize;
    let pane = self.pane.get_mut();
    let scrollback_value = pane.scrollback();
    let used = pane.max_scrollback();
    let row_base = used.saturating_sub(scrollback_value);
    Some(LogicalPosition::new(row_base.saturating_add(local_row), local_col))
}
```

4. **`link_at_position` (~line 905) and `try_handle_link_click` (~line 921):** Accept `LayoutRect`, use i32 math.

```rust
fn link_at_position(&self, area: LayoutRect, mouse: &MouseEvent) -> Option<String> {
    if area.width == 0 || area.height == 0 { return None; }
    let mouse_col = i32::from(mouse.column);
    let mouse_row = i32::from(mouse.row);
    if mouse_col < area.x
        || mouse_col >= area.x.saturating_add(i32::from(area.width))
        || mouse_row < area.y
        || mouse_row >= area.y.saturating_add(i32::from(area.height))
    { return None; }
    let local_x = mouse_col.saturating_sub(area.x) as usize;
    let local_y = mouse_row.saturating_sub(area.y) as usize;
    self.link_overlay.borrow().link_at(local_y, local_x)
}

fn try_handle_link_click(&mut self, area: LayoutRect, mouse: &MouseEvent) -> bool {
    if !matches!(mouse.kind, MouseEventKind::Press(MouseButton::Left)) { return false; }
    if let Some(url) = self.link_at_position(area, mouse)
        && self.invoke_link_handler(&url)
    { return true; }
    false
}
```

5. **`RenderDragHost::logical_position_from_point` (~line 957):** Remove `layout_rect_to_rect`, use `LayoutRect` directly (same i32 math as #3).

6. **Revert `dispatch_mouse` changes** in `mod.rs` — the reconstruction was working on the wrong level. The `screen_area` is already correct as a signed `LayoutRect`; the bug was downstream conversion to unsigned.
