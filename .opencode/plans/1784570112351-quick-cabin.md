# Fix: Floating window hitboxes extend off-screen when dragged left

## Bugs

1. **Bug 1 — Ghost hitboxes**: When a floating window is partially off-screen left (negative `dest.x`), chrome/content hitboxes registered at negative screen-space coordinates overlap the visible tiled window area. Since floating windows render after tiled windows (`bifurcate_draw_order`), their hitboxes appear first in the reverse-scan `hit_test`. Clicks at screen columns where the tiled window should be reachable instead activate the floating window.

2. **Bug 2 — Coordinate conversion offset**: After Bug 1 is fixed (hitboxes clipped), the `hit_rect` from the registry is clipped to `x=0` instead of preserving the true signed origin (e.g., `x=-39`). The component's event handler subtracts this clipped origin from the global mouse position, producing incorrect local coordinates for text selection.

## Root Cause

### Bug 1
`register_window_chrome_hitboxes()` at `crates/term-wm-console/src/draw_plan_renderer.rs:43` registers hitboxes via `to_screen(lx, ly, lw, lh)` which computes `x = ox + lx` where `ox = surface.dest.x as i16` (can be negative, e.g., `-40`). No clip rect is pushed on the `HitboxRegistry`'s clip stack (`crates/term-wm-core/src/hitbox_registry.rs`), so `register()` does not clamp coordinates. Hitboxes with width=78 starting at x=-39 extend to column 39, capturing clicks that should fall through to the tiled window layer.

### Bug 2
The `HitboxRegistry::register()` already has clip-stack support (`clip_stack` field, `push_clip`/`pop_clip` methods). If a clip rect is pushed to fix Bug 1, the stored `hit_rect` is clamped to `x=0`. In `dispatch_mouse` (`crates/term-wm-core/src/window/window_manager/mod.rs`), `with_screen_area(hit_rect)` passes this clipped origin to the component, corrupting `screen_col - screen_area.x` calculations.

## Solution

Two coordinated changes:

### Step 1: Thread `screen_bounds` through `ChromeCtx` and push clip rect

**File:** `crates/term-wm-console/src/draw_plan_renderer.rs`

1.1 Add `screen_bounds: LayoutRect` field to `ChromeCtx` (struct at line 22).

1.2 In `render_window_chrome` (line 146), push `ctx.screen_bounds` as a clip rect before calling `register_window_chrome_hitboxes()` and pop it immediately after:

```rust
registry.push_clip(ctx.screen_bounds);
register_window_chrome_hitboxes(registry, key, frame_size, screen_origin, content_hitbox_id);
registry.pop_clip();
```

This causes `HitboxRegistry::register()` to intersect every subsequent hitbox area with the screen bounds. Hitboxes that previously extended to negative x (e.g., x=-4) are trimmed to x=0, so off-screen areas no longer intercept clicks.

### Step 2: Pass `managed_area` at the call site

**File:** `src/lib.rs`

2.1 When constructing `ChromeCtx` at line 130, set `screen_bounds` to the managed area as a `LayoutRect`.

```rust
screen_bounds: LayoutRect {
    x: wm.managed_area().x,
    y: wm.managed_area().y,
    width: wm.managed_area().width,
    height: wm.managed_area().height,
},
```

(Note: `managed_area` returns a `Rect`; convert fields to `LayoutRect` compatible values.)

### Step 3: Reconstruct true screen_area in `dispatch_mouse`

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs`

3.1 In Phase 1 (active capture, `MouseCaptureState::ComponentInteraction` arm, around line 1309): after clipping fixes Bug 1, the `screen_area` stored in the capture state is clipped (e.g., x=0). Reconstruct the true mathematical origin by computing the delta between the floating rect spec and the clipped region. Add this delta to the `screen_area` before passing it to `.with_screen_area()`.

```rust
ComponentOwner::Window(key) => {
    if !self.windows.contains_key(*key) {
        false
    } else {
        let focused = *self.focus.current() == *key;
        let mut true_area = *screen_area;
        if let Some(spec) = self.floating_rect(*key) {
            let dest = spec.resolve_signed(self.managed_area);
            let full = self.full_region_for_key(*key);
            true_area.x += dest.x - full.x as i32;
            true_area.y += dest.y - full.y as i32;
            if true_area.width == full.width { true_area.width = dest.width; }
            if true_area.height == full.height { true_area.height = dest.height; }
        }
        let ctx = self
            .component_context_for(focused, *key)
            .with_screen_area(true_area)
            .with_active_hitbox(*hitbox_id);
        // ... rest unchanged
    }
}
```

3.2 In Phase 4 (press events hit-test the registry, around line 1534): Apply the same reconstruction to `hit_rect` before it is used for `.with_screen_area()` and stored in `MouseCaptureState`:

```rust
ComponentOwner::Window(key) => {
    if matches!(kind, MouseEventKind::Press(_)) && self.is_window_floating(key) {
        self.bring_floating_to_front_key(key);
    }
    let focused = *self.focus.current() == key;
    let mut true_area = hit_rect;
    if let Some(spec) = self.floating_rect(key) {
        let dest = spec.resolve_signed(self.managed_area);
        let full = self.full_region_for_key(key);
        true_area.x += dest.x - full.x as i32;
        true_area.y += dest.y - full.y as i32;
        if true_area.width == full.width { true_area.width = dest.width; }
        if true_area.height == full.height { true_area.height = dest.height; }
    }
    let ctx = self
        .component_context_for(focused, key)
        .with_screen_area(true_area)
        .with_active_hitbox(hitbox_id);
    if let Some(comp) = self.component_for_key_mut(key) {
        let result = comp.handle_events(&core_event, &ctx);
        if !result.is_ignored() && matches!(kind, MouseEventKind::Press(_)) {
            self.mouse_capture = Some(MouseCaptureState::ComponentInteraction {
                owner: ComponentOwner::Window(key),
                screen_area: true_area,
                hitbox_id,
            });
        }
        result.map(|action| (Some(key), action))
    } else {
        EventResult::Ignored
    }
}
```
