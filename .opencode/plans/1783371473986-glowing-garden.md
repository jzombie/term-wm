# Plan: Simplify term-wm API for Third-Party Apps

## Goal

Make `term-bench`, `examples/dual_image.rs`, and third-party apps easier to use with the window manager by eliminating the dual-trait boilerplate (`WindowManagerHost` + `WindowProvider`) and providing a fluent builder API.

## Current Pain Points

1. Every app must implement **both** `WindowManagerHost` and `WindowProvider` traits (coupled via supertrait)
2. `window_component()` requires manual key→component routing (the "manual dispatcher anti-pattern")
3. Components needing their `WindowKey` must use `Arc<Mutex<Option<WindowKey>>>` hacks
4. Terminal setup/teardown (raw mode, alternate screen, mouse capture) is manual boilerplate
5. No `prelude` module — imports are verbose

## Approach: Pragmatic Incremental API

Rather than the full blueprint rewrite (which would break the runner), we add a **`StandaloneApp` builder** that internally implements the existing traits and handles all boilerplate. This is backward-compatible and immediately usable.

---

## Step 1: Add `on_mount` hook to `Component` trait

**File:** `crates/term-wm-core/src/components/mod.rs`

Add a default method to the `Component<Msg>` trait:

```rust
/// Called once after the component is registered with the window manager.
/// Receives the assigned WindowKey and an AppContext for triggering redraws.
fn on_mount(&mut self, _key: WindowKey, _app: &crate::app_context::AppContext) {}
```

Default is no-op — no existing code breaks. Components that need their key (e.g., PTY engines) can override this.

**Also:** Add `spawn_with_context` method to `WindowManager` in `crates/term-wm-core/src/window/window_manager/mod.rs`:

```rust
pub fn spawn(
    &mut self,
    mut component: Box<dyn Component<TermWmAction>>,
    app_ctx: &AppContext,
) -> WindowKey {
    let key = self.create_window(component);
    if let Some(comp) = self.component_for_key_mut(key) {
        comp.on_mount(key, app_ctx);
    }
    key
}
```

## Step 2: Create `StandaloneApp` builder

**New file:** `crates/term-wm-core/src/standalone.rs`

```rust
pub struct StandaloneApp {
    wm: WindowManager,
    app_ctx: Arc<AppContext>,
    windows: Vec<WindowKey>,
    should_quit: bool,
}

impl StandaloneApp {
    /// Create with full standalone chrome (panels, command menu, etc.)
    pub fn new(app_name: &str, app_version: &str) -> Self { ... }

    /// Create with custom WmConfig
    pub fn with_config(config: WmConfig, app_ctx: AppContext) -> Self { ... }

    /// Register a component, returning its WindowKey
    pub fn register<C: Component<TermWmAction> + 'static>(
        &mut self,
        component: C,
    ) -> WindowKey { ... }

    /// Run the event loop (blocks until quit)
    pub fn run<O: RenderTarget, D: EventSource>(
        self,
        output: &mut O,
        driver: &mut D,
    ) -> io::Result<()> { ... }

    /// Mutable access to the underlying WindowManager
    pub fn wm(&mut self) -> &mut WindowManager { ... }
}

impl WindowManagerHost for StandaloneApp { ... }
impl WindowProvider for StandaloneApp { ... }
```

The `run` method calls `run_window_app` internally. The trait impls delegate to the internal `WindowManager` and `Vec<WindowKey>`.

**Register in:** `crates/term-wm-core/src/lib.rs` — add `pub mod standalone;`

## Step 4: Create `WidgetAdapter` for third-party Ratatui widgets

**New file:** `crates/term-wm-ui-components/src/widget_adapter.rs`

```rust
use ratatui::widgets::{Widget, StatefulWidget};

/// Wraps any Ratatui Widget as a term-wm Component.
pub struct WidgetAdapter<W: Widget> {
    widget: W,
}

impl<W: Widget> WidgetAdapter<W> {
    pub fn new(widget: W) -> Self { Self { widget } }
}

impl<W: Widget + 'static> Component<TermWmAction> for WidgetAdapter<W> {
    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, _ctx: &ComponentContext, _reg: &mut HitboxRegistry) {
        frame.render_widget(&self.widget, area);
    }
    // handle_events defaults to Ignored — widgets are display-only by default
}

/// Wraps a Ratatui StatefulWidget + its State as a term-wm Component.
pub struct StatefulWidgetAdapter<W: StatefulWidget> {
    widget: W,
    state: W::State,
}

impl<W: StatefulWidget + 'static> StatefulWidgetAdapter<W> {
    pub fn new(widget: W, state: W::State) -> Self { Self { widget, state } }
    pub fn state(&self) -> &W::State { &self.state }
    pub fn state_mut(&mut self) -> &mut W::State { &mut self.state }
}

impl<W: StatefulWidget + 'static> Component<TermWmAction> for StatefulWidgetAdapter<W> {
    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, _ctx: &ComponentContext, _reg: &mut HitboxRegistry) {
        // Need interior mutability for StatefulWidget::render
        // Use RefCell for the state
    }
}
```

**Note:** `StatefulWidget::render` takes `&mut State`, but `Component::render` takes `&self`. The `StatefulWidgetAdapter` will need a `RefCell<W::State>` to handle this. This is safe because rendering is single-threaded.

**Register in:** `crates/term-wm-ui-components/src/lib.rs` — add `pub mod widget_adapter; pub use widget_adapter::*;`

## Step 5: Rewrite `examples/dual_image.rs`

**File:** `examples/dual_image.rs`

Before (126 lines, dual-trait boilerplate):
```rust
struct App { wm: WindowManager, ... }
impl WindowManagerHost for App { ... }  // 5 methods
impl WindowProvider for App { ... }     // 4 methods
```

After (~40 lines, clean builder):
```rust
use term_wm::prelude::*;

fn main() -> io::Result<()> {
    let mut app = StandaloneApp::new("dual-image", "0.1.0");

    let mut left = SvgImageComponent::new();
    left.set_keep_aspect(true);
    left.set_colorize(true);
    left.load_from_path("assets/zenOSmosis-logo.svg")?;

    let mut right = SvgImageComponent::new();
    right.set_keep_aspect(true);
    right.set_colorize(true);
    right.load_from_path("assets/zenOSmosis-logo.svg")?;

    app.register(left);
    app.register(right);

    let mut output = ConsoleRenderTarget::new()?;
    output.enter()?;
    let mut input = ConsoleEventSource::new();
    let result = app.run(&mut output, &mut input);
    output.exit()?;
    result
}
```

**Reduction: ~126 lines → ~40 lines (68% less code)**

## Step 6: Add optional term-wm integration to `term-bench`

**Files:** `crates/term-bench/Cargo.toml`, `crates/term-bench/src/main.rs`

Add a `wm` feature flag:
```toml
[features]
wm = ["dep:term-wm"]

[dependencies]
term-wm = { workspace = true, optional = true }
```

Add a `--wm` CLI flag:
```rust
/// Run inside a term-wm window manager (for integration testing)
#[arg(long)]
wm: bool,
```

When `--wm` is passed, use `StandaloneApp` to wrap the noise rendering in a managed window. This lets the benchmark test WM overhead without changing the core benchmark logic.
