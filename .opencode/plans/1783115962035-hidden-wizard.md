# Plan: Window Lifecycle State Machine (Requirement 2b)

## Summary

Replace ad-hoc imperative window lifecycle management (`minimized: bool` flag, scattered mutations to z-order/layout/focus collections) with a canonical state machine. Add async teardown via `AppExited(WindowKey)` control message. Establish strict Window Server / Application Runner boundary.

---

## Phase 1: `WindowState` Enum + State Machine

### 1a. Define `WindowState` enum and canonical transitions

**File:** `crates/term-wm-core/src/window/entry.rs`

Add:
```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum WindowState {
    Realized,    // Allocated in SlotMap, invisible, not in layout tree
    Mapped,      // Visible, geometry routed to layout tree
    Unmapped,    // Hidden (withdrawn), in memory
    Iconic,      // Mapped but hidden from workspace (minimized)
    Shaded,      // Chrome-only visible (title bar only)
}
```

Replace `pub minimized: bool` on `Window` with:
```rust
pub state: WindowState,
```

Valid transitions (enforced by `transition_to` method):
```
Realized  â†’ Mapped     (tile/set_focus)
Realized  â†’ Unmapped   (hide without showing)
Mapped    â†’ Iconic     (minimize)
Iconic    â†’ Mapped     (restore)
Mapped    â†’ Unmapped   (withdraw)
Unmapped  â†’ Mapped     (remap)
Mapped    â†’ Shaded     (shade)
Shaded    â†’ Mapped     (unshade)
Any       â†’ REMOVAL    (close_window removes from SlotMap)
```

### 1b. Add state transition method to `WindowManager`

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs`

Add:
```rust
pub fn transition_window(&mut self, key: WindowKey, new_state: WindowState) { ... }
```

This is the **single point of truth** for all lifecycle changes. It:
1. Reads `Window.state`
2. Validates the transition
3. Performs side-effects (update z_order, managed_draw_order, focus ring, regions, layout)
4. Writes `Window.state = new_state`

**File:** `crates/term-wm-core/src/window/window_manager/layout.rs`

### 2f. Handle `AppExited` in runner

**File:** `crates/term-wm-core/src/runner.rs`

In the event handler (alongside the `PtyWakeup` handling), add:
```rust
// Drain AppExited events from the unified event source.
// These are accumulated in a separate list during event processing.
```

The `AppExited` handling should:
1. Call `app.windows().close_window(key)` (which transitions to Unmapped then removes from SlotMap)
2. Call `app.wm_close_window(key)` (which cleans up app state)

**Alternative design:** Process `AppExited` in the existing closed-window drain:
```rust
// In the idle path or event handler:
for key in app.windows().take_exited_windows() {
    app.windows().close_window(key);
    app.wm_close_window(key)?;
}
```
Where `take_exited_windows()` stores keys that received `AppExited`.

### 2g. Deprecate `has_exited()` polling

**File:** `src/main.rs`

Modify `enumerate_windows()` to stop filtering by `has_exited()`. Windows are now closed event-driven via `AppExited`. The exited window is removed from `terminals` map in `wm_close_window()`, so the next `enumerate_windows()` iteration naturally skips it.

## Phase 3: Window Server / Application Runner Boundary

### 3a. Extract `WindowServer` struct

**New file:** `crates/term-wm-core/src/window/server.rs`

```rust
/// Window Server â€” the UI arbiter. Owns the SlotMap and provides the
/// canonical state machine API for window lifecycle transitions.
/// Layout trees and z-order hold only lightweight `WindowKey` handles,
/// never window data directly.
pub struct WindowServer {
    windows: SlotMap<WindowKey, Window>,
    pub(crate) z_order: Vec<WindowKey>,
    next_window_seq: usize,
    closed_windows: Vec<WindowKey>,
    exited_windows: Vec<WindowKey>,  // from AppExited events
}
```

Methods:
- `insert(window) -> WindowKey` â€” allocate in `Realized` state
- `transition(key, new_state)` â€” validate and apply state transition
- `remove(key)` â€” remove from SlotMap (only from `Unmapped` state)
- `get(key) -> Option<&Window>`, `get_mut(key) -> Option<&mut Window>`
- `window_state(key) -> Option<WindowState>`
- `register_exited(key)` â€” record an AppExited notification
- `take_closed_windows() -> Vec<WindowKey>`
- `take_exited_windows() -> Vec<WindowKey>`

### 3b. Integrate with `WindowManager`

**File:** `crates/term-wm-core/src/window/window_manager/mod.rs`

`WindowManager` gains:
```rust
pub struct WindowManager {
    server: WindowServer,  // replaces raw `windows: SlotMap<WindowKey, Window>`
    // ... all existing fields except windows closed_windows
}
```

Most lifecycle methods delegate to `self.server`:
```rust
pub fn create_window(&mut self) -> WindowKey {
    self.server.insert(Window::new(self.server.next_seq()))
}
pub fn transition_window(&mut self, key: WindowKey, new_state: WindowState) {
    self.server.transition(key, new_state);
    // Post-transition: update managed_draw_order, focus, etc.
    self.sync_after_transition(key, new_state);
}
pub fn close_window(&mut self, key: WindowKey) {
    self.server.transition(key, WindowState::Unmapped);
    self.server.remove(key);
}
```

### 3c. Application Runner boundary

The `WindowManagerHost` trait becomes the **Application Runner** interface:
```rust
pub trait WindowManagerHost {
    fn windows(&mut self) -> &mut WindowManager;  // sole access to WindowServer
    fn wm_new_window(&mut self) -> io::Result<()>;
    fn wm_close_window(&mut self, key: WindowKey) -> io::Result<()>;
    // ...
}
```

The boundary rule: Application Runner (PTY I/O, `main.rs`) calls `wm_new_window`/`wm_close_window`. Window Server (UI arbiter, `WindowManager`/`WindowServer`) handles state transitions. Neither side directly mutates the other's data.
