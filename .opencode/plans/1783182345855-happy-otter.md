# Plan: Multi-Phase Component Lifecycle Architecture

## Overview

Switch from the current immediate-mode `Component` trait (where `handle_event` mutates state directly and `render` takes `&mut self`) to a five-phase Elm-inspired architecture that decouples **state mutation** from **rendering**:

1. `init` ‚Äî runs once on mount
2. `handle_events` ‚Äî returns `Option<Action>` without mutating state
3. `update` ‚Äî mutates state exclusively via `Action` messages
4. `render` ‚Äî pure function (`&self`) translates state to drawing instructions
5. `destroy` ‚Äî teardown hook

No reverse-compatibility hacks. Full switch.

### 1a. Define `TermWmAction` ‚Äî the concrete message type

**File:** `crates/term-wm-core/src/actions.rs`

Unify the existing `Action` (keybinding dispatch) and `WmMenuAction` into a single enum. This is the concrete `Msg` type used by the internal term-wm window manager and all built-in components.

#[derive(Debug, Clone)]
pub enum TermWmAction {

    KeyToBytes(Vec<u8>),

    // ... all existing Action variants

    ConfirmAction(ConfirmAction),
    ClipboardPaste(String),

```

All keybinding-matching code is updated to match against `TermWmAction`.

### 1b. `EventResult<Msg>` ‚Äî decouple routing from mutation

`Consumed`/`Ignored` fight the Elm paradigm ‚Äî they are routing status, not state mutation commands. Use a tri-state result that lets the event loop stop propagation immediately without queuing a no-op action:

```rust
#[derive(Debug, Clone)]
pub enum EventResult<Msg> {
    /// Event not handled ‚Äî continue routing to next layer.
    Ignored,
    /// Event handled, no state change needed ‚Äî stop routing.
    Consumed,
    /// Event produced a mutation command ‚Äî queue for update phase.
    Action(Msg),
}

impl<Msg> EventResult<Msg> {
    pub fn is_ignored(&self) -> bool { matches!(self, Self::Ignored) }
    pub fn is_consumed(&self) -> bool { matches!(self, Self::Consumed) }
    pub fn into_action(self) -> Option<Msg> {
        match self { Self::Action(msg) => Some(msg), _ => None }
    }
}
```

The trait signature becomes:

```rust
fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<Msg>;
```

### 1b. The Component trait is generic over `Msg`

The trait takes a type parameter `Msg` ‚Äî the message/action type that the component emits from `handle_events` and accepts in `update`. This keeps the trait **object-safe** because `Msg` is concrete at the `dyn` site:

```rust
pub trait Component<Msg> {
    fn init(&mut self) {}
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> Option<Msg>;
    fn update(&mut self, action: &Msg, ctx: &ComponentContext);
    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, ctx: &ComponentContext);
    fn destroy(&mut self) {}
    fn selection_status(&self) -> SelectionStatus { ... }
    fn selection_text(&self) -> Option<String> { None }
    fn set_selection_enabled(&mut self, _enabled: bool) {}
    fn paste(&mut self, _text: &str) -> bool { false }
}
```

### 1c. Storage in Window/SlotMap

All internal term-wm components use `TermWmAction` as their `Msg`:

```rust
pub struct Window {
    pub component: Option<Box<dyn Component<TermWmAction>>>,
    // ...
}
```

External library users can define their own `enum MyAppAction { ... }` and use `Box<dyn Component<MyAppAction>>`.

## 2. New `Component<Msg>` Trait

**File:** `crates/term-wm-core/src/components/mod.rs`

### Trait definition

```rust
/// Generic over Msg ‚Äî the action/message type emitted by handle_events
/// and consumed by update. Object-safe because Msg is concrete at the dyn site.
pub trait Component<Msg> {
    /// Phase 1: Called once when the component is mounted.
    fn init(&mut self) {}

    /// Phase 2: Evaluate raw events, return an EventResult.
    /// Does NOT mutate state. Consumed = handled no action, Ignored = pass through.
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<Msg> {
        EventResult::Ignored
    }

    /// Phase 3: Mutate state in response to an Action.
    /// Called exactly once per action, in FIFO order.
    /// Can push new actions into the queue via the sink parameter.
    fn update(&mut self, action: &Msg, ctx: &ComponentContext, actions: &mut VecDeque<(WindowKey, Msg)>) {}

    /// Phase 4: Pure render. Takes &self ‚Äî no state mutation allowed.
    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, ctx: &ComponentContext);

    /// Phase 5: Teardown. Called before the component is unmounted.
    fn destroy(&mut self) {}

    // Queries (no mut, return state info)
    fn selection_status(&self) -> SelectionStatus { SelectionStatus::default() }
    fn selection_text(&self) -> Option<String> { None }

    // Mutations needed for clipboard/selection (still &mut self)
    fn set_selection_enabled(&mut self, _enabled: bool) {}
    fn paste(&mut self, _text: &str) -> bool { false }
}
```

### Key changes from current trait

| Method | Old signature | New signature | Notes |
|--------|--------------|---------------|-------|
| Trait | `Component` | `Component<Msg>` | Generic over message type |
| `resize` | `(&mut self, Rect, &ComponentContext)` | **removed** | Handled in `update` via `TermWmAction` variant |
| `render` | `(&mut self, ...)` | `(&self, ...)` | **Critical** ‚Äî enables shared borrows |
| `handle_event` | `(&mut self, &Event, &ComponentContext) -> bool` | `handle_events` returns `EventResult<Msg>` | No state mutation |
| *(new)* `update` | ‚Äî | `(&mut self, &Msg, &ComponentContext, &mut VecDeque<(WindowKey, Msg)>)` | Exclusive mutation point + cascading sink |
| *(new)* `init` | ‚Äî | `(&mut self)` | One-time setup |
| *(new)* `destroy` | ‚Äî | `(&mut self)` | Teardown |
| `selection_status` | `(&self)` | unchanged | |
| `selection_text` | `(&mut self)` | `(&self)` | Now a pure query |
| `set_selection_enabled` | `(&mut self)` | unchanged | |
| `paste` | `(&mut self)` | unchanged | |

## 4. Runner / Event Loop Changes

**File:** `crates/term-wm-core/src/runner.rs`

### 4a. `run_app` event handler

The core event loop in `run_app` changes from:

```rust
// OLD: handle_event mutates state inline
component.handle_event(&event, &ctx);
```

To a **three-phase dispatch** with `EventResult` and a `VecDeque` for cascading actions:

```rust
// NEW: handle_events ‚Üí drain action queue ‚Üí (later) render
let mut action_queue: VecDeque<(WindowKey, TermWmAction)> = VecDeque::new();

// Phase 2: handle_events (collect actions, no mutation)
match component.handle_events(&event, &ctx) {
    EventResult::Ignored => {
        // Event not handled ‚Äî continue routing to WM/chrome
    }
    EventResult::Consumed => {
        // Stop routing, no state mutation needed
    }
    EventResult::Action(action) => {
        // WM-level actions handled immediately
        match &action {
            TermWmAction::CloseWindow => { /* close directly */ },
            TermWmAction::FocusNext => { /* focus directly */ },
            _ => action_queue.push_back((window_key, action)),
        }
    }
}

// Phase 3: drain action queue (allows cascading actions)
while let Some((key, action)) = action_queue.pop_front() {
    if let Some(component) = app.window_component_mut(key) {
        component.update(&action, &ctx, &mut action_queue);
    }
    // WM-wide actions emitted by update itself are also caught here:
    // e.g. closing the last tab spawns CloseWindow, which gets processed
    // before the render phase starts.
}
```

### 4b. Idle path (rendering)

The idle tick changes to:

```rust
// Phase 4: render ‚Äî all components via &self
output.draw(|frame| {
    // WM state preparation (still mutable)
    app.windows().begin_frame();
    app.windows().prepare_draw();

    // Component rendering (shared borrows)
    for (key, surface, focused) in draw_plan {
        let ctx = app.windows().component_context_for(focused, key);
        if let Some(component) = app.window_component(key) {
            component.render(subframe, surface.inner, &ctx);
        }
    }

    // Panels/overlays
    app.windows().render_panel(frame);
    app.windows().render_overlays(frame, ...);
});
```

### 4c. `WindowProvider` trait

Add an immutable accessor for the render phase. All component types are `dyn Component<TermWmAction>`:

```rust
pub trait WindowProvider: WindowManagerHost {
    /// For event/update phase (mutable access)
    fn window_component_mut(&mut self, key: WindowKey) -> Option<&mut dyn Component<TermWmAction>>;

    /// For render phase (shared access)
    fn window_component(&self, key: WindowKey) -> Option<&dyn Component<TermWmAction>>;

    // Keep other methods...
}
```

The `WindowManagerHost::windows()` method remains `&mut WindowManager` for WM state mutations during initialization/setup.

## 6. Component Rewrites (17 implementations)

Each `impl Component for X` must be rewritten to the new lifecycle. Here is the migration guide for each category:

### 6a. TerminalComponent (`terminal.rs`)

| Old method | New behavior |
|-----------|-------------|
| `resize` | Remove. Resize handled via `update` on `Action::Resize(u16, u16)` |
| `handle_event` | Split into `handle_events` (returns Actions) + `update` (processes them) |
| `render` | `&self` ‚Äî use `RefCell` for cursor blink timer, otherwise all fields readable |
| *(new)* `init` | Create PTY, set up link handler |
| *(new)* `destroy` | Clean up PTY process |

**handle_events results:**
- Key ‚Üí `EventResult::Action(TermWmAction::KeyToBytes(bytes))`
- Shift+PageUp/Down ‚Üí `EventResult::Action(TermWmAction::Scroll(delta))`
- Mouse (selection, but no state change needed) ‚Üí `EventResult::Consumed`
- Mouse (selection drag ending, state change) ‚Üí `EventResult::Action(TermWmAction::SelectionDragEnd)`
- Mouse (terminal) ‚Üí `EventResult::Action(TermWmAction::MouseToBytes(bytes))`
- Link click ‚Üí `EventResult::Action(TermWmAction::LinkClicked(idx))`
- Irrelevant event ‚Üí `EventResult::Ignored`

**update processes:**
- `KeyToBytes(bytes)` ‚Üí `pane.write_bytes(&bytes)`, clear selection
- `Scroll(delta)` ‚Üí `pane.set_scrollback(...)`
- `MouseToBytes(bytes)` ‚Üí `pane.write_bytes(&bytes)`
- `LinkClicked(idx)` ‚Üí open link via `open::that()`
- `Resize(w, h)` ‚Üí `pane.resize(PtySize{...})`

**render with `&self`:**
- All screen reading (pane.screen(), pane.scrollback()) is read-only
- Cursor blink: use `RefCell<Instant>` for last cursor move time
- Selection rendering: selection state stored in struct fields, read during render

### 6b. ScrollViewComponent (`scroll_view.rs`)

| Old method | New behavior |
|-----------|-------------|
| `resize` | Remove. Area stored during `render` |
| `handle_event` | Returns `TermWmAction::ScrollView(delta)` or child actions |
| `render` | `&self` ‚Äî inner child rendered via `child.render()` (child is `Box<dyn Component<TermWmAction>>`, called with `&self`) |
| *(new)* `update` | Process scroll actions, forward relevant actions to child |

No `RefCell` needed here. Because `ScrollViewComponent` *owns* the `Box<dyn Component<TermWmAction>>`, mutability flows from `self` at compile time with zero overhead:

```rust
pub struct ScrollViewComponent {
    child: Box<dyn Component<TermWmAction>>,
    scroll_state: ScrollState,
}

impl Component<TermWmAction> for ScrollViewComponent {
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<TermWmAction> {
        // Handle scrollbar/keyboard scrolling here (return EventResult::Action(...))
        // Delegate to child for other events
        self.child.handle_events(event, ctx)
    }

    fn update(&mut self, action: &TermWmAction, ctx: &ComponentContext, actions: &mut VecDeque<_>) {
        match action {
            TermWmAction::ScrollView(delta) => { /* update scroll offset */ }
            _ => self.child.update(action, ctx, actions),
        }
    }

    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, ctx: &ComponentContext) {
        // &self ‚Üí &self.child ‚Äî works naturally because child is owned
        self.child.render(frame, area, ctx);
    }
}
```

The only `RefCell` needed in the entire hierarchy is for *interior mutability during render*: cursor blink timers, animation counters. These are plain fields on the leaf component, not on the wrapper tree.
```

### 6c. MarkdownViewerComponent / TextRendererComponent

Same pattern as ScrollView ‚Äî owned children. `TextRendererComponent` may need a `RefCell<Instant>` for cursor blink timing, but the child component wrapper is a plain owned field.

### 6d. ListComponent / ToggleListComponent

### 6e. Overlay Components (confirm, help, keybindings, menu)

## 9. Borrow Checker Strategy

The critical goal: `render(&self)` enables **multiple simultaneous shared references** during the render phase, while **only one mutable reference** exists during the update phase.

### No RefCell needed for component hierarchy

Wrapper components (`ScrollViewComponent`, `MarkdownViewerComponent`, etc.) *own* their children as plain `Box<dyn Component<TermWmAction>>`. Mutability flows from `self` at compile time:
- `render(&self)` ‚Üí can call `self.child.render(...)` ‚Äî child is `&dyn Component`
- `update(&mut self, ...)` ‚Üí can call `self.child.update(...)` ‚Äî child is `&mut dyn Component`

Zero runtime overhead. The compiler enforces the phase separation.

### RefCell for render-time interior mutability

Any component that needs interior mutability during render (cursor blink timers, animation state) uses `RefCell<T>`:
```rust
pub struct TerminalComponent {
    cursor_last_blink: RefCell<Instant>,
    pane: Box<dyn Pane>,  // Pane operations are all &self (read-only for render)
    // ...
}
```

### WindowManager split borrows

The `WindowManager` must support both `&self` (for render-phase queries like `focused_window()`, `visible_region_for_key()`) and `&mut self` (for setup like `register_managed_layout()`). This is handled by:
1. WM state setup runs first (`&mut self`)
2. Component rendering runs second (via pre-computed draw plan, using `&self` for remaining WM queries)
3. Panel/overlay rendering runs third (`&mut self`)

The draw plan is pre-computed in step 1 so step 2 only needs `&self` access.

## 10. Async I/O ‚Üí Synchronous Queue Bridge

**Question:** How do asynchronous PTY streams (ANSI bytes from child processes) inject into this strict synchronous three-phase queue?

### Current architecture (unchanged for PTY I/O)

The PTY reader thread is already decoupled from the event loop. It writes to a shared buffer that `Pane::screen()` reads from during `render`. This does **not** need actions ‚Äî the terminal's visual state lives in the PTY parser, not in the component's fields:

```
[Child Process] ‚îÄ‚îÄstdout‚îÄ‚îÄ> [PTY Reader Thread] ‚îÄ‚îÄwrites‚îÄ‚îÄ> [vt100::Parser (Arc<Mutex>)]
                                                                     ‚îÇ
                                                          Pane::screen() (read-only)
                                                                     ‚îÇ
                                                              render(&self)
```

The `TerminalComponent::render` reads `pane.screen()` directly ‚Äî no action needed.

### What DOES need an action

External events that require *state mutation* in the component or WM:
- **Process exit** ‚Üí `TermWmAction::ProcessExited(WindowKey)` ‚Üí `update` closes the window
- **PTY resize response** ‚Üí the PTY may send SIGWINCH back; this is already a synchronous response to a `resize` call
- **Clipboard from OSC 52** ‚Üí the PTY reader captures it; `update` stores it in the WM's clipboard

### Injection mechanism

The driver/event source already exposes a way to collect "out-of-band" events before the main event handler runs. In the current code:

```rust
for key in driver.take_exited_windows() {
    app.windows().close_window(key);
}
```

This pattern extends naturally: before each event loop iteration, poll the driver for external signals and push corresponding actions into the `VecDeque`:

```rust
// Phase 0: Collect external events before processing user input
for key in driver.take_exited_windows() {
    action_queue.push_back((key, TermWmAction::ProcessExited));
}
if let Some(profile_change) = driver.take_power_profile_change() {
    // Push WM-level action
    action_queue.push_back((ROOT_KEY, TermWmAction::ProfileChange(profile_change)));
}
// Process external actions before user input
while let Some((key, action)) = action_queue.pop_front() {
    if let Some(component) = app.window_component_mut(key) {
        component.update(&action, &ctx, &mut action_queue);
    }
}
// Then handle user event ‚Üí handle_events ‚Üí update drain ‚Üí render
```

This keeps the three-phase loop intact while letting background threads signal the queue cleanly.

### Timer events (cursor blink, animations)

These are handled naturally by the idle (render) path. On each idle tick, before rendering:
- Cursor blink: compare `cursor_last_blink` (the `RefCell<Instant>` on `TerminalComponent`) against `Instant::now()` ‚Äî if enough time elapsed, toggle blink state.
- This is a *query*, not a mutation of the component tree. The blink state is an interior-mutable field read during `render`.

No timers need to inject actions into the queue ‚Äî the render phase implicitly picks up elapsed time.
