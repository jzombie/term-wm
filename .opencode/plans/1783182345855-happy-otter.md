# Plan: Multi-Phase Component Lifecycle Architecture

## Overview

Switch from the current immediate-mode `Component` trait (where `handle_event` mutates state directly and `render` takes `&mut self`) to a five-phase Elm-inspired architecture that decouples **state mutation** from **rendering**:

1. `init` ŌĆö runs once on mount
2. `handle_events` ŌĆö returns `Option<Action>` without mutating state
3. `update` ŌĆö mutates state exclusively via `Action` messages
4. `render` ŌĆö pure function (`&self`) translates state to drawing instructions
5. `destroy` ŌĆö teardown hook

No reverse-compatibility hacks. Full switch.

### 1a. Define `TermWmAction` ŌĆö the concrete message type

**File:** `crates/term-wm-core/src/actions.rs`

Unify the existing `Action` (keybinding dispatch) and `WmMenuAction` into a single enum. This is the concrete `Msg` type used by the internal term-wm window manager and all built-in components.

```rust
#[derive(Debug, Clone)]
pub enum TermWmAction {
    // Terminal-level actions
    KeyToBytes(Vec<u8>),
    Scroll(isize),
    MouseToBytes(Vec<u8>),
    ClearSelection,
    LinkClicked(usize),

    // ScrollView actions
    ScrollView(isize),
    ScrollToTop,
    ScrollToBottom,

    // WM-level actions (from existing Action + WmMenuAction)
    CloseWindow,
    NewWindow,
    FocusNext,
    FocusPrev,
    Quit,

    OpenHelp,
    OpenKeybindings,
    MenuUp, MenuDown, MenuSelect,
    ToggleMouseCapture,

    CloseOverlay(OverlayId),
    // ... all existing Action variants

    // Clipboard
    ConfirmAction(ConfirmAction),
    ClipboardPaste(String),

    // Marker variants
    Consumed,
    Ignored,
}
```

All keybinding-matching code is updated to match against `TermWmAction`.

### 1b. The Component trait is generic over `Msg`

The trait takes a type parameter `Msg` ŌĆö the message/action type that the component emits from `handle_events` and accepts in `update`. This keeps the trait **object-safe** because `Msg` is concrete at the `dyn` site:

```rust
pub trait Component<Msg> {
    fn init(&mut self) {}
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> Option<Msg>;
    fn update(&mut self, action: &Msg, ctx: &ComponentContext);
    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, ctx: &ComponentContext);
    fn destroy(&mut self) {}
    fn selection_status(&self) -> SelectionStatus { ... }
    fn selection_text(&self) -> Option<String> { None }
    fn set_selection_enabled(&mut self, _enabled: bool) {}
    fn paste(&mut self, _text: &str) -> bool { false }
}
```

### 1c. Storage in Window/SlotMap

All internal term-wm components use `TermWmAction` as their `Msg`:

```rust
pub struct Window {
    pub component: Option<Box<dyn Component<TermWmAction>>>,
    // ...
}
```

External library users can define their own `enum MyAppAction { ... }` and use `Box<dyn Component<MyAppAction>>`.

## 2. New `Component<Msg>` Trait

**File:** `crates/term-wm-core/src/components/mod.rs`

### Trait definition

```rust
/// Generic over Msg ŌĆö the action/message type emitted by handle_events
/// and consumed by update. Object-safe because Msg is concrete at the dyn site.
pub trait Component<Msg> {
    /// Phase 1: Called once when the component is mounted.
    fn init(&mut self) {}

    /// Phase 2: Evaluate raw events, return an Action.
    /// Does NOT mutate state. Returns None if the event is irrelevant.
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> Option<Msg> {
        None
    }

    /// Phase 3: Mutate state in response to an Action.
    /// Called exactly once per action, in FIFO order.
    fn update(&mut self, action: &Msg, ctx: &ComponentContext) {}

    /// Phase 4: Pure render. Takes &self ŌĆö no state mutation allowed.
    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, ctx: &ComponentContext);

    /// Phase 5: Teardown. Called before the component is unmounted.
    fn destroy(&mut self) {}

    // Queries (no mut, return state info)
    fn selection_status(&self) -> SelectionStatus { SelectionStatus::default() }
    fn selection_text(&self) -> Option<String> { None }

    // Mutations needed for clipboard/selection (still &mut self)
    fn set_selection_enabled(&mut self, _enabled: bool) {}
    fn paste(&mut self, _text: &str) -> bool { false }
}
```

### Key changes from current trait

| Method | Old signature | New signature | Notes |
|--------|--------------|---------------|-------|
| Trait | `Component` | `Component<Msg>` | Generic over message type |
| `resize` | `(&mut self, Rect, &ComponentContext)` | **removed** | Handled in `update` via `TermWmAction` variant |
| `render` | `(&mut self, ...)` | `(&self, ...)` | **Critical** ŌĆö enables shared borrows |
| `handle_event` | `(&mut self, &Event, &ComponentContext) -> bool` | `handle_events` returns `Option<Msg>` | No state mutation |
| *(new)* `update` | ŌĆö | `(&mut self, &Msg, &ComponentContext)` | Exclusive mutation point |
| *(new)* `init` | ŌĆö | `(&mut self)` | One-time setup |
| *(new)* `destroy` | ŌĆö | `(&mut self)` | Teardown |
| `selection_status` | `(&self)` | unchanged | |
| `selection_text` | `(&mut self)` | `(&self)` | Now a pure query |
| `set_selection_enabled` | `(&mut self)` | unchanged | |
| `paste` | `(&mut self)` | unchanged | |

## 3. `Overlay` Trait Changes

**File:** `crates/term-wm-core/src/components/mod.rs`

`Overlay` and `MenuOverlay` are also generic over `Msg`:

```rust
pub trait Overlay<Msg>: Component<Msg> + Any {
    fn visible(&self) -> bool { true }
    fn shadow_rect(&self, area: Rect) -> Option<Rect> { None }
    fn handle_confirm_event(&mut self, event: &Event) -> Option<ConfirmAction>;
}
```

The `Overlay` trait no longer duplicates Component methods. `as_any`/`as_any_mut` are removed ŌĆö use `Component::downcast_ref`/`downcast_mut` via the `Any` bound.

`MenuOverlay<Msg, R>` uses the new lifecycle:
- `handle_events` returns menu navigation actions
- `update` processes them to change selection
- `render` draws the menu from frozen state

**Storage types change:**
```rust
// Before:
Box<dyn Overlay>
Box<dyn MenuOverlay<WmMenuAction>>
// After:
Box<dyn Overlay<TermWmAction>>
Box<dyn MenuOverlay<TermWmAction, TermWmAction>>
```

## 4. Runner / Event Loop Changes

**File:** `crates/term-wm-core/src/runner.rs`

### 4a. `run_app` event handler

The core event loop in `run_app` changes from:

```rust
// OLD: handle_event mutates state inline
component.handle_event(&event, &ctx);
```

To a **three-phase dispatch** (using `TermWmAction` as the concrete Msg type):

```rust
// NEW: handle_events ŌåÆ collect ŌåÆ update ŌåÆ (later) render
let mut pending_actions: Vec<(WindowKey, TermWmAction)> = Vec::new();

// Phase 2: handle_events (collect actions, no mutation)
if let Some(action) = component.handle_events(&event, &ctx) {
    // WM-level actions handled here (close, focus, etc.)
    match &action {
        TermWmAction::CloseWindow => { ... },
        TermWmAction::FocusNext => { ... },
        _ => pending_actions.push((window_key, action)),
    }
}

// Phase 3: update (one component at a time)
for (key, action) in &pending_actions {
    if let Some(component) = app.window_component_mut(*key) {
        component.update(action, &ctx);
    }
}
```

### 4b. Idle path (rendering)

The idle tick changes to:

```rust
// Phase 4: render ŌĆö all components via &self
output.draw(|frame| {
    // WM state preparation (still mutable)
    app.windows().begin_frame();
    app.windows().prepare_draw();

    // Component rendering (shared borrows)
    for (key, surface, focused) in draw_plan {
        let ctx = app.windows().component_context_for(focused, key);
        if let Some(component) = app.window_component(key) {
            component.render(subframe, surface.inner, &ctx);
        }
    }

    // Panels/overlays
    app.windows().render_panel(frame);
    app.windows().render_overlays(frame, ...);
});
```

### 4c. `WindowProvider` trait

Add an immutable accessor for the render phase. All component types are `dyn Component<TermWmAction>`:

```rust
pub trait WindowProvider: WindowManagerHost {
    /// For event/update phase (mutable access)
    fn window_component_mut(&mut self, key: WindowKey) -> Option<&mut dyn Component<TermWmAction>>;

    /// For render phase (shared access)
    fn window_component(&self, key: WindowKey) -> Option<&dyn Component<TermWmAction>>;

    // Keep other methods...
}
```

The `WindowManagerHost::windows()` method remains `&mut WindowManager` for WM state mutations during initialization/setup.

Overlay `render` calls change from `&mut self` to `&self`. All types use `dyn Component<TermWmAction>`:


## 6. Component Rewrites (17 implementations)

Each `impl Component for X` must be rewritten to the new lifecycle. Here is the migration guide for each category:

### 6a. TerminalComponent (`terminal.rs`)

| Old method | New behavior |
|-----------|-------------|
| `resize` | Remove. Resize handled via `update` on `Action::Resize(u16, u16)` |
| `handle_event` | Split into `handle_events` (returns Actions) + `update` (processes them) |
| `render` | `&self` ŌĆö use `RefCell` for cursor blink timer, otherwise all fields readable |
| *(new)* `init` | Create PTY, set up link handler |
| *(new)* `destroy` | Clean up PTY process |

**handle_events actions returned:**
- Key ŌåÆ `TermWmAction::KeyToBytes(bytes)` 
- Shift+PageUp/Down ŌåÆ `TermWmAction::Scroll(delta)`
- Mouse (selection) ŌåÆ `TermWmAction::Consumed` (selection state tracked in `update`)
- Mouse (terminal) ŌåÆ `TermWmAction::MouseToBytes(bytes)`
- Link click ŌåÆ `TermWmAction::LinkClicked(idx)`

**update processes:**
- `KeyToBytes(bytes)` ŌåÆ `pane.write_bytes(&bytes)`, clear selection
- `Scroll(delta)` ŌåÆ `pane.set_scrollback(...)`
- `MouseToBytes(bytes)` ŌåÆ `pane.write_bytes(&bytes)`
- `LinkClicked(idx)` ŌåÆ open link via `open::that()`
- `Resize(w, h)` ŌåÆ `pane.resize(PtySize{...})`

**render with `&self`:**
- All screen reading (pane.screen(), pane.scrollback()) is read-only
- Cursor blink: use `RefCell<Instant>` for last cursor move time
- Selection rendering: selection state stored in struct fields, read during render

### 6b. ScrollViewComponent (`scroll_view.rs`)

| Old method | New behavior |
|-----------|-------------|
| `resize` | Remove. Area stored during `render` |
| `handle_event` | Returns `TermWmAction::ScrollView(delta)` or child actions |
| `render` | `&self` ŌĆö inner child rendered via `child.render()` (child is `Box<dyn Component<TermWmAction>>`, called with `&self`) |
| *(new)* `update` | Process scroll actions, forward relevant actions to child |

**Key challenge:** `ScrollViewComponent` wraps a child `Box<dyn Component<TermWmAction>>`. With `render(&self)`, the child also needs `&self`. Use `RefCell<Box<dyn Component<TermWmAction>>>` for the child wrapper, allowing shared access for render and mutable access for update.

**Pattern:**
```rust
pub struct ScrollViewComponent {
    child: RefCell<Box<dyn Component<TermWmAction>>>,
    scroll_state: RefCell<ScrollState>,
    // ... other fields
}

impl Component<TermWmAction> for ScrollViewComponent {
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> Option<TermWmAction> {
        // Handle scrollbar/keyboard scrolling here
        // Delegate to child for other events
        self.child.borrow_mut().handle_events(event, ctx)
    }

    fn update(&mut self, action: &TermWmAction, ctx: &ComponentContext) {
        match action {
            TermWmAction::ScrollView(delta) => { /* update scroll offset */ }
            _ => self.child.borrow_mut().update(action, ctx),
        }
    }

    fn render(&self, frame: &mut UiFrame<'_>, area: Rect, ctx: &ComponentContext) {
        // Read scroll state (no &mut needed)
        // Render scrollbars
        // Render child
        self.child.borrow().render(inner_frame, inner_area, &inner_ctx);
    }
}
```

### 6c. MarkdownViewerComponent / TextRendererComponent

Similar pattern to ScrollView ŌĆö `RefCell` for child components and any mutable rendering state (link hover tracking).

### 6d. ListComponent / ToggleListComponent

Straightforward: selection index, scroll offset stored as fields. `handle_events` returns navigation actions, `update` modifies selection. `render` draws from current state.

### 6e. Overlay Components (confirm, help, keybindings, menu)

All follow the same pattern. The `Overlay` trait's `handle_confirm_event` can remain as a specialized method (returns `Option<ConfirmAction>`). The actual rendering uses `&self`.

### 6f. Image Components (ascii_image, svg_image)

Stateless rendering. `handle_events` returns `None`. `render` draws from stored image data. Easiest migration.

### 6g. WmBottomPanelComponent / WmTopPanelComponent

These are rendered by `WindowManager::render_panel`. With `render(&self)`, the panel components need `RefCell` for any mutable state (hover tracking, animation counters).

### 6h. WmDebugLogComponent

Log buffer is `Arc<Mutex<VecDeque>>` ŌĆö already thread-safe. `render` reads the buffer. `handle_events` returns scroll actions.

## 9. Borrow Checker Strategy

The critical goal: `render(&self)` enables **multiple simultaneous shared references** during the render phase, while **only one mutable reference** exists during the update phase.

### RefCell pattern for wrapper components

`ScrollViewComponent` wraps a child with `RefCell<Box<dyn Component<TermWmAction>>>`:
- `render` calls `child.borrow()` for `&dyn Component<TermWmAction>` access
- `update` calls `child.borrow_mut()` for `&mut dyn Component<TermWmAction>` access
- Borrow checking is *runtime* rather than *compile-time*, but the phase separation guarantees no overlap

### RefCell pattern for render-time mutability

Any component that needs interior mutability during render (cursor blink timers, animation state) uses `RefCell<T>`:
```rust
pub struct TerminalComponent {
    cursor_last_blink: RefCell<Instant>,
    pane: Box<dyn Pane>,  // Pane operations are all &self (read-only for render)
    // ...
}
```

### WindowManager split borrows

The `WindowManager` must support both `&self` (for render-phase queries like `focused_window()`, `visible_region_for_key()`) and `&mut self` (for setup like `register_managed_layout()`). This is handled by:
1. WM state setup runs first (`&mut self`)
2. Component rendering runs second (via pre-computed draw plan, using `&self` for remaining WM queries)
3. Panel/overlay rendering runs third (`&mut self`)

The draw plan is pre-computed in step 1 so step 2 only needs `&self` access.
