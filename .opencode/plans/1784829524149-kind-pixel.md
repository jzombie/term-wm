# Plan: Fix Floating Geometry Loss on Unmap + Z-Order Trapping

## Problem

1. **Floating rect erased on Unmap**: `transition_window(_, Unmapped)` calls `clear_floating_rect`, wiping the geometry. With `ClosePolicy::Unmap`, the window comes back as tiled (no floating rect) and can't be brought to the front.

2. **bring_to_front_key guards non-floating**: `bring_to_front_key` returns early for non-floating windows. A re-opened system panel that lost its floating rect is stuck behind all floating windows forever.

3. **No auto-float on re-map**: `Unmapped → Mapped` always calls `reattach_to_tiling_layout`. If other windows are floating, the window should float too, not tile.

## Changes

### 1. Preserve floating_rect across Unmap

**`crates/term-wm-core/src/window/window_manager/mod.rs`** — Remove `self.clear_floating_rect(key)` from the `(_, WindowState::Unmapped)` match arm (line 501). The z-order removal, region removal, scroll removal, focus fallback, and tiling detach remain.

### 2. Auto-float on Unmapped → Mapped when workspace is floating

**`crates/term-wm-core/src/window/window_manager/mod.rs`** — In the `(WindowState::Unmapped, WindowState::Mapped)` match arm: if any other mapped window is floating, assign a default floating rect instead of tiling. Only fall back to `reattach_to_tiling_layout` when all mapped windows are tiled.

Also call `self.bring_to_front_key(key)` after focus_add so the window elevates to the top of the z-order.

### 3. Promote to floating on bring_to_front_key

**`crates/term-wm-core/src/window/window_manager/layout.rs`** — In `bring_to_front_key`: if the target is non-floating but other mapped windows are floating, detach it from the tiling layout and assign a floating rect from its current region, then push to the front of z_order. This allows any window that receives focus to float up when floating windows exist.
