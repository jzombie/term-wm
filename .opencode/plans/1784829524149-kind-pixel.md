# Plan: Scroll to Bottom on Keypress in Non-Direct-Mode Terminal

## Bug

Terminal window with scrollbars: scroll up, type a key, viewport does NOT scroll to bottom. The pane's internal scrollback resets to 0 but the `ScrollViewComponent`'s `offset_y` stays stale. The render sync logic then interprets the stale offset as user intent and scrolls back up.

## Root Cause

`ScrollViewComponent::update` (scroll_view.rs) does not intercept `KeyToBytes` or `ClipboardPaste` actions. It forwards them to the child without updating its own viewport. The terminal's `render_screen` then sees `scrollback=0` with stale `view_offset > 0` and reverses the scroll.

## Fix

### 1. Add `KeyToBytes` / `ClipboardPaste` handling in `ScrollViewComponent::update`

In `crates/term-wm-ui-components/src/scroll_view.rs`, before the `_ => { ... }` catch-all, intercept `KeyToBytes` and `ClipboardPaste`:

```rust
TermWmAction::KeyToBytes(_) | TermWmAction::ClipboardPaste(_) => {
    let handle = self.scroll_handle();
    let info = handle.info();
    let child_ctx = ctx.with_viewport(info, Some(handle));
    self.content.borrow_mut().update(action, &child_ctx, actions);
    // Terminal resets scrollback to 0 on input — scroll viewport to bottom
    handle.scroll_vertical_to(usize::MAX);
}
```

### 2. Verify `render_screen` sync logic

The existing sync logic in `terminal.rs:render_screen` already handles `scrollback == 0` with `view_offset >= last_max - 1` → `scroll_vertical_to(usize::MAX)`. By setting the viewport offset to bottom AFTER the terminal resets scrollback, the sync logic will confirm we're at the bottom and not reverse.
