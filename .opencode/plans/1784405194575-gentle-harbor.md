# Plan: Replace `HitTarget` with Opaque `HitboxId` — Hierarchical Delegation

## Goal

Remove `HitTarget` enum from `HitboxRegistry`. Make the registry a pure spatial
database storing `(HitboxId, LayoutRect)`. Route events via hierarchical
delegation: the WM dispatches to top-level structures in Z-order; each
component implements "delegate first, self-identify second."

This permanently solves the enum bloat problem without creating a flat routing
table that must know about every component type.

---

## Core Concepts

### 1. Pure Spatial Database

`HitboxRegistry` stores only opaque IDs:

```rust
pub struct HitboxEntry {
    pub id: HitboxId,
    pub area: LayoutRect,
}
```

`hit_test(position)` returns `Option<(HitboxId, LayoutRect)>` — the topmost
pixel under the cursor. No enums, no routing targets, no payloads.

**Removed**: `HitTarget` enum, `build_component_map()`, `component_area()`, `window_area()`.

### 2. `HitboxId` Type

```rust
// hitbox_registry.rs
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct HitboxId(pub u64);

impl HitboxId {
    pub fn new() -> Self {
        static NEXT_ID: AtomicU64 = AtomicU64::new(1);
        Self(NEXT_ID.fetch_add(1, Ordering::Relaxed))
    }
}
```

### 3. `ComponentContext.active_hitbox`

```rust
pub struct ComponentContext {
    // ... existing fields ...
    active_hitbox: Option<HitboxId>,
}

pub fn active_hitbox(&self) -> Option<HitboxId>
pub fn with_active_hitbox(&self, id: HitboxId) -> Self
```

### 4. `Component` Trait — `hitbox_id()` Convenience (Optional)

```rust
trait Component<Msg> {
    /// If the component registers its own hitbox area, return its ID here.
    /// The default handle_events uses this for leaf-component occlusion.
    fn hitbox_id(&self) -> Option<HitboxId> { None }
    
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<Msg> {
        if let Event::Mouse(mouse) = event {
            // Leaf-component convenience: skip if our ID doesn't match
            if let Some(my_id) = self.hitbox_id() {
                if ctx.active_hitbox() != Some(my_id) {
                    return EventResult::Ignored;
                }
            }
            // ... coordinate conversion and semantic dispatch to on_mouse_* ...
        }
        self.on_key(event, ctx)
    }
}
```

This is a **convenience**, not an enforcement mechanism. Container components
override `handle_events` and implement the "delegate first, self-identify
second" pattern explicitly (see below). The primary occlusion mechanism is
the WM's Z-order dispatch, not this check.

## Hierarchical Dispatch

### `dispatch_mouse()` Algorithm

```
1. hit_test(position) → Some((active_id, hit_rect))
2. Set ctx.active_hitbox = Some(active_id), ctx.screen_area = hit_rect

3. === WM Structural Elements (not components) ===
   Scan ResizeHandle[] for matching hitbox_id → execute resize capture
   Scan DragHandle[] for matching hitbox_id → execute drag capture
   Scan SplitHandle[] for matching hitbox_id → layout.handle_event()
   (Header buttons: Close, Max, Min, ToggleDirectMode → execute directly)
   If consumed → return

4. === Command Palette / Overlays (top Z-order) ===
   command_menu_component.handle_events(event, ctx)
   if consumed → return

5. === Singletons in Z-order ===
   fab_component.handle_events(event, ctx)
   if consumed → return

6. === Windows in Z-order ===
   For each window (front-to-back):
     window.component.handle_events(event, ctx)
     if consumed → return
   
7. === Panels (bottom Z-order) ===
   top_component.handle_events(event, ctx)
   bottom_component.handle_events(event, ctx)

8. === Notification (catches remaining) ===
   notification_component.handle_events(event, ctx)

9. Fall through → Ignored
```

The WM does NOT know which component was clicked. It propagates `active_id`
and lets each component decide.

### Component Pattern: "Delegate First, Self-Identify Second"

Every component follows this structure:

```rust
// Container component (has children)
impl Component for ScrollViewComponent {
    fn handle_events(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<Msg> {
        // 1. Delegate to children first
        if self.child_list.handle_events(event, ctx).is_consumed() {
            return EventResult::Consumed;
        }
        // 2. Check own surface (e.g., scrollbar background)
        if let Event::Mouse(_) = event {
            if ctx.active_hitbox() == Some(self.scrollbar_background_id) {
                return EventResult::Consumed;
            }
        }
        EventResult::Ignored
    }
}

// Leaf component (no children)
impl Component for TerminalComponent {
    fn hitbox_id(&self) -> Option<HitboxId> {
        Some(self.surface_id) // default handle_events auto-checks occlusion
    }
}
```

### Why This Works

**Zero Global Enums**: No `HitTarget`, `RouteTarget`, or `WindowAction` with
app widget variants. Any widget anywhere can add itself to the hierarchy.

**Perfect Occlusion, Naturally**:
- A Menu renders above a Window. Spatial DB sets `active_id` to Menu's ID.
- WM asks Menu to handle event. Menu matches its ID, returns Consumed.
- Window never sees the event. Correct occlusion without any enum.

**Composite-Safe**:
- A Button renders inside a Menu. Spatial DB sets `active_id` to Button's ID.
- WM asks Menu. Menu delegates to Button. Button matches, consumes.
- Menu's own background check never runs (already consumed).

**Structural Elements Stay Bounded**: WM scans its own finite set of resize
handles, drag handles, layout handles before the component pass. These are
WM-internal and have constant upper bounds (~200 items).

## Bug Fix: Capture State Missing `hitbox_id` (SEV-1)

**Root cause**: `MouseCaptureState::ComponentInteraction` stores `(key, screen_area)`
but NOT the `hitbox_id` of the component that initiated the capture. During Phase 1
capture continuation, the `ComponentContext` is built without `with_active_hitbox()`,
leaving `active_hitbox = None`. The default `Component::handle_events` occlusion
check (`ctx.active_hitbox() != Some(self.hitbox_id)`) then evaluates to
`None != Some(id) = true` and **silently ignores every captured event**.

**Effect**: Any component that overrides `hitbox_id()` and uses the default
`handle_events` (e.g., TerminalComponent, WmFabComponent, WmCommandPaletteComponent)
will have its drag/release/move events silently dropped during a capture sequence.
Text selection drag, FAB interactions, and command palette menu interactions
are all broken.

**Fix** (3 edits in `crates/term-wm-core/src/window/window_manager/mod.rs`):

1. **Add `hitbox_id` field** to `MouseCaptureState::ComponentInteraction`:
   ```rust
   ComponentInteraction {
       key: WindowKey,
       screen_area: LayoutRect,
       hitbox_id: HitboxId,  // ← ADD
   },
   ```

2. **Set `hitbox_id` at capture creation** (Phase 4, window content routing, ~line 1666):
   `MouseCaptureState::ComponentInteraction { key, screen_area: hit_rect, hitbox_id }`

3. **Inject `active_hitbox` during capture continuation** (Phase 1, ~line 1281):
   `.with_screen_area(*screen_area).with_active_hitbox(hitbox_id)`

## Component-Level Keyboard Focus

### Design

Store `active_keyboard_focus: Option<HitboxId>` on the **Window** struct
(not `WindowManager`). The `WindowManager` routes via `FocusRing` (unchanged);
the focused `Window` injects its local focus into `ComponentContext` before
delegating. Components request focus via `TermWmAction::RequestKeyboardFocus(HitboxId)`.
This guarantees `FocusRing` switches inherently revoke micro-level focus.

### Files to modify

| File | Change |
|------|--------|
| `crates/term-wm-core/src/window/entry.rs` | Add `active_keyboard_focus: Option<HitboxId>` field to `Window`, init to `None` |
| `crates/term-wm-core/src/component_context.rs` | Add `keyboard_focus_id: Option<HitboxId>` field + accessor + builder |
| `crates/term-wm-core/src/actions.rs` | Add `TermWmAction::RequestKeyboardFocus(HitboxId)` variant |
| `crates/term-wm-core/src/components/mod.rs` | Update default `handle_events`: for keyboard events, only call `on_key` when `ctx.keyboard_focus_id() == Some(self.hitbox_id())`. Components **without** a hitbox_id forward to `on_key` unconditionally (backward compat). Components **with** a hitbox_id return `Ignored` on mismatch so parent containers can continue delegation. |

The default `handle_events` keyboard path becomes:
```rust
if let Event::Key(_) = event {
    if let Some(my_id) = self.hitbox_id() {
        // Has hitbox_id = participates in focus system
        if ctx.keyboard_focus_id() == Some(my_id) {
            return self.on_key(event, ctx);
        }
        // Not focused → return Ignored (enables parent delegation)
        return EventResult::Ignored;
    }
    // No hitbox_id = legacy component → forward unconditionally
    return self.on_key(event, ctx);
}
```
Containers that override `handle_events` must delegate `Event::Key` to children
before checking their own focus — standard "delegate first, self-identify second"
applies to keyboard focus just as it does to mouse occlusion.
| `crates/term-wm-core/src/window/window_manager/mod.rs` | In `component_context_for()`, inject the focused window's `active_keyboard_focus` as `ctx.keyboard_focus_id`; In `process_action`, intercept `RequestKeyboardFocus` and set `Window.active_keyboard_focus` |
| `crates/term-wm-core/src/runner.rs` | Route `RequestKeyboardFocus` action to set window-level focus key |

## Bug: Singleton Capture Missing — Scrollbar Drag Activates Window Behind

### Root Cause

In `dispatch_mouse()` Phase 4, only the **window content** routing path creates a
`ComponentInteraction` capture state on consumed Press events. The command palette,
FAB, overlays, top panel, and bottom panel paths return `Consumed` directly without
setting the capture lock.

When a user presses on the command palette scrollbar:
1. Phase 4 routes to command palette → returns `Consumed`
2. No capture state is created
3. On subsequent `Drag`, Phase 1 finds no active capture
4. Falls to Phase 4 again → command palette returns `Ignored` (handles Press, not Drag)
5. Falls through to window content routing → window behind activates

### Fix: Unified `LayerManager`

The structural root cause is state fragmentation — each singleton is stored as a
separate field on `WindowManager`, forcing the dispatch logic to hardcode each
one. The fix is a unified `LayerManager` backed by a generational arena, storing
a Z-ordered list of trait objects.

### 1. `LayerManager` Struct

```rust
// window_manager/layer_manager.rs

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct LayerId(pub u64);

impl LayerId {
    pub fn new() -> Self {
        static NEXT_ID: std::sync::atomic::AtomicU64 = std::sync::atomic::AtomicU64::new(1);
        Self(NEXT_ID.fetch_add(1, std::sync::atomic::Ordering::Relaxed))
    }
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ZPlane {
    Background,  // panels — rendered before windows
    Foreground,  // overlays, FAB, command palette — rendered after windows
}

/// Singular macro-focus authority. Replaces `FocusRing<WindowKey>`.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum MacroFocus {
    FocusRing(slotmap::DefaultKey),  // existing window focus (WindowKey)
    Layer(LayerId),                 // singleton component has focus
}

pub struct LayerManager {
    layers: HashMap<LayerId, Box<dyn WmComponent>>,
    background_order: Vec<LayerId>,
    foreground_order: Vec<LayerId>,
    /// Isolated keyboard focus for foreground layers (separate from window focus).
    pub active_keyboard_focus: Option<HitboxId>,
}

impl LayerManager {
    pub fn insert(&mut self, comp: Box<dyn WmComponent>, plane: ZPlane) -> LayerId {
        let id = LayerId::new();
        self.layers.insert(id, comp);
        match plane {
            ZPlane::Background => self.background_order.push(id),
            ZPlane::Foreground => self.foreground_order.push(id),
        }
        id
    }

    pub fn remove(&mut self, id: LayerId) {
        self.layers.remove(&id);
        self.background_order.retain(|l| *l != id);
        self.foreground_order.retain(|l| *l != id);
    }

    pub fn get_mut(&mut self, id: LayerId) -> Option<&mut dyn WmComponent> {
        self.layers.get_mut(&id).map(|c| c.as_mut())
    }

    /// Dispatch foreground layers (front-to-back = reverse render order).
    /// Called first — overlays have highest Z.
    pub fn dispatch_foreground(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<TermWmAction> {
        let layer_ctx = ctx.clone().with_keyboard_focus_id(self.active_keyboard_focus);
        for id in self.foreground_order.iter().rev() {
            if let Some(comp) = self.layers.get_mut(id) {
                if comp.handle_events(event, &layer_ctx).is_consumed() {
                    return EventResult::Consumed;
                }
            }
        }
        EventResult::Ignored
    }

    /// Dispatch background layers (panels).
    pub fn dispatch_background(&mut self, event: &Event, ctx: &ComponentContext) -> EventResult<TermWmAction> {
        let layer_ctx = ctx.clone().with_keyboard_focus_id(self.active_keyboard_focus);
        for id in self.background_order.iter().rev() {
            if let Some(comp) = self.layers.get_mut(id) {
                if comp.handle_events(event, &layer_ctx).is_consumed() {
                    return EventResult::Consumed;
                }
            }
        }
        EventResult::Ignored
    }

    /// Render background (panels) — back-to-front.
    pub fn render_background(&mut self, backend: &mut dyn RenderBackend, area: LayoutRect,
                             ctx: &ComponentContext, registry: &mut HitboxRegistry) {
        for id in &self.background_order {
            if let Some(comp) = self.layers.get_mut(id) {
                comp.render(backend, area, ctx, registry);
            }
        }
    }

    /// Render foreground (overlays, command palette, FAB) — back-to-front.
    pub fn render_foreground(&mut self, backend: &mut dyn RenderBackend, area: LayoutRect,
                             ctx: &ComponentContext, registry: &mut HitboxRegistry) {
        for id in &self.foreground_order {
            if let Some(comp) = self.layers.get_mut(id) {
                comp.render(backend, area, ctx, registry);
            }
        }
    }
}
```

### 4. Keyboard Dispatch — Singular `MacroFocus`

Replace `FocusRing<WindowKey>` with `MacroFocus`. Before any keyboard dispatch:

```rust
// runner.rs — keyboard event routing
match app.wm().macro_focus {
    MacroFocus::FocusRing(key) => {
        // existing window keyboard routing via FocusRing
    }
    MacroFocus::Layer(id) => {
        let ctx = app.wm().component_context(false)
            .with_keyboard_focus_id(app.wm().layer_manager.active_keyboard_focus);
        if let Some(comp) = app.wm().layer_manager.get_mut(id) {
            if comp.handle_events(event, &ctx).is_consumed() { return; }
        }

}
```

- Click on a Window → `MacroFocus::FocusRing(window_key)`
- Click on a foreground layer → `MacroFocus::Layer(layer_id)`
